## Q1. Як створити, протестувати і задеплоїти Docker-образ? 🟣

Docker пакує застосунок з усіма залежностями в **образ** — незмінний артефакт, що однаково запуститься будь-де. Образ описується в `Dockerfile`, збирається в образ, запускається як **контейнер**.

```dockerfile
# Dockerfile — multi-stage: збірка окремо, легкий фінальний образ
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci                 # ставить і dev-залежності (потрібні для тестів)
COPY . .
RUN npm run build
RUN npm test               # тести як частина збірки: образ не збереться, якщо впали

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
CMD ["node", "dist/main.js"]
```

«Протестувати» — на двох рівнях: **unit-тести під час збірки** (у build-стейджі, де ще є devDeps — вище) і **health-check піднятого контейнера** (переконатись, що образ реально стартує), і лише потім `push`:
```bash
# 1. СТВОРИТИ (тут же в build-стейджі проганяються тести)
docker build -t myapp:1.0 .

# 2. ПРОТЕСТУВАТИ, що образ живий
docker run -d -p 3000:3000 --name myapp myapp:1.0
curl -f http://localhost:3000/health || exit 1   # -f → ненульовий код, якщо не 2xx

# 3. ЗАДЕПЛОЇТИ (лише якщо збірка+тести й health-check пройшли)
docker tag myapp:1.0 user/myapp:1.0
docker push user/myapp:1.0                         # у реєстр (Docker Hub / GHCR)
```

**➕ Пов'язане: що таке multi-stage build і навіщо? 🔵**
Multi-stage — кілька `FROM` в одному Dockerfile: у першому «важкому» етапі збираєш (з dev-залежностями, компілятором), у фінальний копіюєш лише результат. Так фінальний образ маленький і без зайвого — швидший деплой і менша поверхня атаки.

**➕ Пов'язане: чим образ відрізняється від контейнера? 🟢**
**Образ** — незмінний шаблон (як клас). **Контейнер** — запущений екземпляр образу (як об'єкт). З одного образу можна підняти багато контейнерів.

**➕ Пов'язане: навіщо `.dockerignore`? 🟢**
Виключає з контексту збірки зайве (`node_modules`, `.git`, `.env`) — прискорює build і не тягне в образ секрети чи сміття.

## Q2. Як працювати з сервером через SSH? 🟣

SSH — захищений протокол для віддаленого доступу до сервера в терміналі. Автентифікація — найкраще через пару ключів (приватний у тебе, публічний на сервері), а не пароль.

```bash
ssh-keygen -t ed25519 -C "you@example.com"      # згенерувати ключ
ssh-copy-id user@203.0.113.10                    # покласти публічний ключ на сервер
ssh user@203.0.113.10                            # підключитись
scp ./build.tar.gz user@203.0.113.10:/opt/app/   # скопіювати файл на сервер
```

Зручно налаштувати `~/.ssh/config`, щоб не писати повну адресу щоразу:
```
Host gfa
  HostName 203.0.113.10
  User deploy
  IdentityFile ~/.ssh/id_ed25519
# тепер просто: ssh gfa
```

**➕ Пов'язане: чому ключі безпечніші за пароль? 🟢**
Пароль можна підібрати брутфорсом; приватний ключ практично неможливо. Публічний ключ на сервері нічого не розкриває, а приватний ніколи не залишає твою машину. Тому на проді парольний вхід зазвичай узагалі вимикають.

**➕ Пов'язане: що таке SSH-тунель (port forwarding)? 🟢**
Дозволяє «прокинути» порт через SSH — напр. підключитись до БД на сервері, яка назовні закрита, через `ssh -L 5432:localhost:5432 gfa`. Локальний порт стає вікном у закритий сервіс на сервері.

## Q3. Як налаштувати LLM-flow для перевірки простих security breaches перед комітом? 🟣

Мета — автоматично ловити забуті секрети (API-ключі, паролі, токени) **до** того, як вони потраплять у git. Реалізується як **pre-commit hook**: скрипт, що запускається перед кожним комітом і блокує його при знахідці.

```yaml
# .pre-commit-config.yaml — детектор секретів + власна LLM-перевірка
repos:
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks            # ловить типові патерни ключів/токенів
```

Поверх «класичного» детектора можна додати LLM-крок, що аналізує diff на неочевидні витоки:
```bash
#!/usr/bin/env bash
# .git/hooks/pre-commit (спрощено)
DIFF=$(git diff --cached)
echo "$DIFF" | llm -s "Перевір diff на витоки секретів/credentials. \
Відповідай лише 'BLOCK: причина' або 'OK'." | grep -q '^BLOCK' && {
  echo "❌ Можливий витік секрету — коміт заблоковано"; exit 1;
}
```

Ідея: детектор ловить відомі патерни, LLM додає перевірку контексту (напр. приватний ключ, вставлений як багаторядковий текст).

**➕ Пов'язане: чим pre-commit hook відрізняється від перевірки в CI? 🔵**
Pre-commit спрацьовує **локально до коміту** — швидкий зворотний зв'язок, секрет не потрапляє навіть в історію. CI перевіряє **після push** — надійніша «остання лінія», бо hook можна пропустити (`--no-verify`). Тому роблять обидва: hook для зручності, CI як гарантія.

**➕ Пов'язане: що робити, якщо секрет уже потрапив у git? 🔵**
Просто видалити в новому коміті **недостатньо** — він лишається в історії. Треба **відкликати/ротувати** сам секрет (він скомпрометований) і за потреби переписати історію (`git filter-repo`/BFG). Головне — ротація, бо видалення з історії не гарантує, що його ніхто не встиг зчитати.

## Q4. Як налаштувати LLM-flow для перевірки code convention перед комітом? 🟣

Та сама механіка pre-commit hook, але мета — стиль і конвенції коду. Базовий рівень — детерміновані лінтери/форматери; LLM-крок додає перевірки, які важко виразити правилом (зрозумілі назви, відповідність домовленостям команди).

```yaml
# .pre-commit-config.yaml — лінт+формат на закомічених файлах
repos:
  - repo: local
    hooks:
      - id: eslint
        name: eslint
        entry: npx eslint --fix
        language: system
        files: \.(ts|tsx)$
      - id: prettier
        name: prettier
        entry: npx prettier --write
        language: system
        files: \.(ts|tsx|json|md)$
```

Лінтери — лише базовий, детермінований рівень. Власне **LLM-крок** (те, про що питання) додають поверх — він читає diff і перевіряє те, чого правилом не виразиш:
```bash
#!/usr/bin/env bash
# .git/hooks/pre-commit — після eslint/prettier
DIFF=$(git diff --cached)
echo "$DIFF" | llm -s "Перевір diff на наші конвенції: осмислені назви, \
без god-функцій, коментарі до нетривіальної логіки. \
Відповідай лише 'BLOCK: причина' або 'OK'." | grep -q '^BLOCK' && {
  echo "❌ Порушення конвенцій — коміт заблоковано"; exit 1;
}
```
Комбінація: лінтер тримає жорсткі правила (`no-unused-vars`), LLM ловить «дух» — напр. `const d = getData()` для ESLint валідний, а LLM позначить `d` як незрозумілу назву й попросить перейменувати.

**➕ Пов'язане: чим лінтер відрізняється від форматера? 🟢**
**Форматер** (Prettier) відповідає за вигляд (відступи, лапки) — переписує автоматично. **Лінтер** (ESLint) ловить потенційні помилки й погані патерни (невикористані змінні, небезпечні конструкції). Форматер — про стиль, лінтер — про якість; використовують разом.

**➕ Пов'язане: що таке conventional commits? 🟢**
Домовленість про формат повідомлень коміта (`feat:`, `fix:`, `chore:`). Дозволяє автоматично генерувати changelog і визначати версію (semver). Перевіряється hook'ом `commitlint`.

## Q5. Як створити serverless-функцію і чим вона відрізняється від VM? 🟣

Serverless (AWS Lambda й аналоги) — модель, де ти завантажуєш **лише функцію**, а хмара сама запускає її на подію й масштабує. Платиш за фактичні виклики/час, не за час простою. Немає сервера, який ти адмініструєш.

```ts
// AWS Lambda handler — вебхук, що обробляє подію й повертає відповідь
export const handler = async (event: APIGatewayEvent) => {
  const body = JSON.parse(event.body ?? "{}");
  await processWebhook(body);
  return { statusCode: 200, body: JSON.stringify({ ok: true }) };
};
```

**VM vs serverless:**

| | VM / контейнер (сервер) | Serverless (Lambda) |
|---|---|---|
| Хто керує сервером | Ти | Провайдер |
| Оплата | За час роботи (навіть простою) | За виклик/час виконання |
| Масштабування | Налаштовуєш сам | Автоматичне |
| Старт | Завжди «гарячий» | Можливий cold start |
| Коли | Постійне навантаження, стан, довгі процеси | Спорадичні події, вебхуки, обробка черг |

**➕ Пов'язане: що таке cold start? 🔵**
Коли функцію давно не викликали, провайдер «піднімає» середовище з нуля — перший виклик повільніший (cold start). Наступні йдуть по «теплому» середовищу. Критично для латентно-чутливих API; лікують provisioned concurrency або тримають функцію «прогрітою».

**➕ Пов'язане: коли serverless — погана ідея? 🟢**
Довгі процеси (є ліміт часу виконання), постійне високе навантаження (за викликами виходить дорожче за сервер), потреба в стійкому стані/з'єднаннях, чутливість до cold start. Тоді краще контейнер/VM.

## Q6. Як працювати з Linux VPS і дебажити всередині контейнера? 🟣

VPS — віддалений Linux-сервер, яким керуєш через SSH. Треба впевнено володіти базовими командами навігації й уміти зайти всередину контейнера для діагностики.

```bash
# базова навігація
pwd                     # де я
ls -la                  # вміст каталогу (з прихованими)
cd /opt/app && mkdir logs
nano .env               # або vim — редагувати файл

# робота з контейнерами на сервері
docker ps               # які контейнери запущені
docker logs -f app      # дивитись логи контейнера наживо
docker exec -it app sh  # зайти ВСЕРЕДИНУ контейнера (shell) для дебагу
#   всередині: перевірити env, файли, мережу, запустити процес вручну
docker stats            # споживання CPU/пам'яті контейнерами
```

Типовий дебаг: `docker logs` показує помилку → `docker exec -it app sh` заходиш всередину → перевіряєш змінні середовища, доступність БД, конфіги.

**➕ Пов'язане: як подивитися, що «їсть» ресурси на сервері? 🔵**
`top`/`htop` — процеси за CPU/пам'яттю; `df -h` — місце на диску (частий винуватець падінь — забитий диск логами); `free -h` — пам'ять; `journalctl -u service` — логи systemd-сервісу. Це перший набір при «сервер гальмує/впав».

**➕ Пов'язане: навіщо знати права доступу (chmod/chown)? 🟢**
У Linux файли мають власника й права (читання/запис/виконання). Часта проблема деплою — процес не може писати в каталог через неправильного власника. `chown` міняє власника, `chmod` — права; `ls -l` показує поточні.

## Q7. Що таке IaC (Infrastructure as Code) і яку проблему вирішує? 🟢

IaC — описувати інфраструктуру (сервери, мережі, БД, права) **кодом** у файлах, а не клікати в консолі хмари вручну. Інструменти: Terraform, AWS CloudFormation, Pulumi.

```hcl
# Terraform: описуємо ресурс декларативно
resource "aws_s3_bucket" "uploads" {
  bucket = "gfa-course-uploads"
  tags   = { project = "gfa" }
}
```

**Головна проблема, яку вирішує** — «мутація інфраструктури» (configuration drift): коли хтось руками щось підкрутив у консолі, і тепер продакшн відрізняється від того, що всі думають. IaC робить інфраструктуру **відтворюваною** (той самий код → те саме середовище), версійованою (git, рев'ю) і задокументованою самим фактом коду.

**➕ Пов'язане: чим декларативний підхід відрізняється від імперативного? 🟢**
Декларативний (Terraform) — описуєш **бажаний стан** («хай буде 3 сервери»), інструмент сам вирахує, що змінити. Імперативний (bash-скрипт) — описуєш **кроки** («створи сервер, потім…»). Декларативний ідемпотентний: застосуй двічі — результат той самий.

**➕ Пов'язане: що таке Terraform state? 🟢**
Файл стану, де Terraform тримає, які ресурси він уже створив, щоб знати різницю між кодом і реальністю. У команді його зберігають віддалено (S3+lock), щоб не було конфліктів і паралельних змін.

## Q8. Як працюють CI/CD pipelines (на прикладі GitHub Actions)? 🟣

**CI (Continuous Integration)** — на кожен push автоматично збирати, лінтити, тестувати. **CD (Continuous Delivery/Deployment)** — автоматично деплоїти те, що пройшло CI. Мета — швидкий і надійний шлях коду від коміту до продакшну.

У GitHub Actions pipeline описується YAML-файлом у `.github/workflows/`:

```yaml
name: CI
on:
  push:
    branches: [main]
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build
```

Ключові поняття: **workflow** (весь файл), **job** (набір кроків на одному раннері), **step** (окрема дія/команда), **trigger** (`on:` — коли запускати).

**➕ Пов'язане: чим job відрізняється від step, і що виконується паралельно? 🔵**
**Step** — послідовні команди в межах одного **job** (спільне середовище/файли). **Jobs** за замовчуванням виконуються **паралельно** й на різних раннерах (ізольовано); залежності задають через `needs:`. Тому build і lint можна гнати паралельно, а deploy — `needs: [build, test]`.

**➕ Пов'язане: що таке matrix build? 🟢**
`strategy.matrix` запускає той самий job для набору варіантів (кілька версій Node, ОС) паралельно. Зручно перевірити сумісність без дублювання конфігу.

**➕ Пов'язане: що таке кешування залежностей у CI? 🟢**
Кеш (`actions/cache` або вбудований у `setup-node`) зберігає `node_modules`/`~/.npm` між запусками, щоб не качати залежності щоразу — суттєво прискорює pipeline.

## Q9. Що треба знати про синтаксис і структуру YAML-конфігу? 🔵

YAML — формат конфігів, де структура задається **відступами** (пробіли, не таби). У CI/CD ним описують pipeline.

```yaml
name: Deploy
on: { push: { branches: [main] } }

env:
  GLOBAL_REGION: eu-central-1        # глобальна env — доступна всім job

jobs:
  deploy:
    runs-on: ubuntu-latest
    env:
      STAGE: production               # локальна env — лише для цього job
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        if: github.ref == 'refs/heads/main'   # conditional — крок лише для main
        run: ./deploy.sh
        env:
          TOKEN: ${{ secrets.DEPLOY_TOKEN }}   # secret — з захищеного сховища, не в коді
```

Ключові поняття: **env** (глобальні vs локальні для job/step), **secrets** (чутливі значення зі сховища репозиторію), **variables** (нечутливі налаштування), **conditionals** (`if:`). Уніфікують pipeline для різних середовищ через reusable workflows / параметри.

**➕ Пов'язане: чим secrets відрізняються від variables? 🔵**
**Secrets** — чутливі (токени, паролі), зашифровані, маскуються в логах, недоступні для PR із форків. **Variables** — звичайні налаштування (регіон, назва середовища), видимі в логах. Правило: усе, що не має світитися, — тільки в secrets.

**➕ Пов'язане: чому в YAML критичні відступи і як уникати помилок? 🟢**
YAML структурує дані відступами з **пробілів** (таби заборонені); зайвий/пропущений пробіл ламає весь файл. Рятує лінтер (`yamllint`) і схема в IDE, що підсвічує помилки одразу.

**➕ Пов'язане: що таке reusable workflow? 🟢**
Винесений спільний workflow, який викликають кілька пайплайнів (`uses: ./.github/workflows/deploy.yml`). Дозволяє не копіювати однакові кроки між середовищами — DRY для CI.

## Q10. Як запускати code quality, linting і тести в CI/CD (multi-staging)? 🟣

Автоматичні перевірки в pipeline ставлять так, щоб **швидкі й дешеві** йшли першими й відсіювали погане рано, а дорогі — потім. Це і є ідея multi-stage pipeline.

```yaml
jobs:
  lint:                       # 1. найшвидше — стиль/статичний аналіз
    runs-on: ubuntu-latest
    steps: [ { uses: actions/checkout@v4 }, { run: npm ci }, { run: npm run lint } ]

  test:                       # 2. тести — лише якщо lint пройшов
    needs: lint
    runs-on: ubuntu-latest
    steps: [ { uses: actions/checkout@v4 }, { run: npm ci }, { run: npm test } ]

  build-deploy:               # 3. найдорожче — build+deploy лише після тестів
    needs: test
    runs-on: ubuntu-latest
    steps: [ { uses: actions/checkout@v4 }, { run: npm run build }, { run: ./deploy.sh } ]
```

Порядок і оптимізація: спершу lint (секунди) → тести → build/deploy. `needs:` вибудовує ланцюг, тож дорогий деплой не запуститься, якщо впав лінт. Оптимізують кешуванням залежностей, паралельними незалежними job'ами і запуском лише потрібних кроків (напр. деплой тільки на `main`).

**➕ Пов'язане: на якому етапі краще ставити найдорожчі перевірки? 🔵**
Принцип «fail fast»: найдешевші й найчастіші причини падінь (лінт, типи, юніт-тести) — на початку, щоб розробник отримав фідбек за секунди. Повільні (e2e, build образу, integration) — далі, і часто лише для основних гілок/PR, а не на кожен коміт.

**➕ Пов'язане: що таке quality gate? 🟢**
Порогова умова, яку зміна має пройти, щоб рухатись далі (мінімальне покриття тестами, нуль critical-issues у SonarQube, зелений лінт). Не пройшов gate — merge/deploy блокується.
