## Q1. Що таке нормалізація (1NF, 2NF, 3NF)? 🟣

Нормалізація — процес організації таблиць так, щоб прибрати дублювання даних і аномалії оновлення. Кожна нормальна форма додає правило:
- **1NF** — атомарні значення (жодних списків у клітинці), кожен рядок унікальний.
- **2NF** — 1NF + кожен неключовий стовпець залежить від **усього** первинного ключа (актуально для складених ключів).
- **3NF** — 2NF + немає транзитивних залежностей (неключовий стовпець не залежить від іншого неключового).

```sql
-- ❌ порушення 1NF: список у клітинці
-- courses(id, title, tags="ai, ml, nlp")

-- ✅ 1NF: винесли в окрему таблицю
CREATE TABLE course_tags (
  course_id UUID REFERENCES courses(id),
  tag       TEXT,
  PRIMARY KEY (course_id, tag)
);
```

```
-- ❌ порушення 3NF: expert_name транзитивно залежить від expert_id, а не від course_id
-- courses(id, title, expert_id, expert_name)
-- ✅ 3NF: expert_name живе в таблиці experts, у courses лише expert_id
```

Мета — одна одиниця даних зберігається в одному місці; змінюєш її в одному рядку, а не в сотнях.

**➕ Пов'язане: що таке денормалізація і коли вона виправдана? 🔵**
Денормалізація — свідоме додавання дублювання (напр. зберегти `comments_count` у пості замість щоразу рахувати `COUNT`). Виправдана, коли читань набагато більше за записи й агрегації дорогі. Ціна — треба руками тримати дубль у синхроні. Тобто нормалізація — дефолт, денормалізація — цільова оптимізація під конкретне вузьке місце.

**➕ Пов'язане: що таке первинний і зовнішній ключ? 🟢**
**Первинний ключ (PK)** — стовпець(і), що унікально ідентифікує рядок. **Зовнішній ключ (FK)** — стовпець, що посилається на PK іншої таблиці, забезпечуючи цілісність (не можна створити урок для неіснуючого курсу). FK — основа зв'язків між таблицями.

## Q2. Що таке транзакційність relational DB — коли і навіщо? 🔵

Транзакція — група операцій, що виконуються **як одне ціле**: або всі успішні, або жодна (відкат). Класичний приклад — переказ грошей: списати з одного рахунку й зарахувати на інший мають статись разом; якщо друга операція впала, перша теж відкочується.

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 'a';
UPDATE accounts SET balance = balance + 100 WHERE id = 'b';
COMMIT;   -- або ROLLBACK, якщо щось пішло не так — тоді жодна зміна не застосується
```

Коли потрібна: будь-яка операція, що змінює **кілька** пов'язаних рядків/таблиць і має лишити БД у цілісному стані (створити замовлення + списати товар зі складу; зареєструвати користувача + створити його профіль).

**➕ Пов'язане: що таке ACID? 🔵**
Гарантії транзакцій: **Atomicity** (все або нічого), **Consistency** (БД лишається валідною за всіма правилами/обмеженнями), **Isolation** (паралельні транзакції не заважають одна одній), **Durability** (закомічене переживе збій/перезапуск). Це те, що відрізняє «серйозну» реляційну БД.

**➕ Пов'язане: що таке рівні ізоляції транзакцій? 🔵**
Керують, наскільки транзакції «бачать» зміни одна одної: `READ COMMITTED` (дефолт у Postgres — бачиш лише закомічене), `REPEATABLE READ`, `SERIALIZABLE` (найсуворіший, ніби транзакції по черзі). Вищий рівень безпечніший, але дорожчий і частіше дає конфлікти. Пов'язані аномалії: dirty read, non-repeatable read, phantom read.

## Q3. Як аналізувати запити (EXPLAIN, full scan vs indexed scan)? 🟣

Щоб зрозуміти, *чому* запит повільний, дивляться його **план виконання** через `EXPLAIN ANALYZE`. План показує, як БД насправді дістає дані.

```sql
EXPLAIN ANALYZE
SELECT * FROM enrollments WHERE expert_id = 'x';
```
План **до індексу** — БД перебирає всю таблицю:
```text
Seq Scan on enrollments  (cost=0.00..1834.00 rows=12 width=64)
                         (actual time=0.02..48.3 rows=12 loops=1)
  Filter: (expert_id = 'x')
  Rows Removed by Filter: 99988        -- прочитала 100k рядків, щоб знайти 12
Execution Time: 48.4 ms
```

Ключове, що читаєш у плані:
- **Seq Scan (full scan)** — БД читає **всю** таблицю рядок за рядком (тут «Rows Removed by Filter: 99988» — марно перебрані рядки). Прийнятно для малих таблиць, повільно для великих.
- **Index Scan** — БД використовує індекс і дістає лише потрібні рядки. Швидко.
- **cost / actual time / rows** — оцінка планувальника, реальний час і скільки рядків опрацьовано.

Якщо на великій таблиці бачиш Seq Scan там, де фільтруєш по стовпцю — це сигнал додати індекс:

```sql
CREATE INDEX idx_enrollments_expert ON enrollments(expert_id);
```
Той самий `EXPLAIN ANALYZE` тепер дає Index Scan — cost і час падають на порядки:
```text
Index Scan using idx_enrollments_expert on enrollments
    (cost=0.42..8.44 rows=12 width=64) (actual time=0.03..0.06 rows=12 loops=1)
  Index Cond: (expert_id = 'x')        -- одразу до потрібних рядків, без Rows Removed
Execution Time: 0.08 ms
```

**➕ Пов'язане: що таке індекс і як він працює (B-tree)? 🔵**
Індекс — окрема відсортована структура (найчастіше B-tree), що дозволяє знаходити рядки за значенням стовпця за `O(log n)` замість перегляду всієї таблиці. Аналогія — алфавітний покажчик у книзі: не гортаєш усе, а йдеш одразу на потрібну сторінку.

**➕ Пов'язане: чому індекс не завжди використовується? 🟢**
БД може обрати Seq Scan, якщо запит повертає велику частку таблиці (індекс тоді не вигідний), якщо статистика застаріла (`ANALYZE`), або якщо умова не «індексована» (напр. функція над стовпцем — `WHERE lower(email)=...` без відповідного індексу).

**➕ Пов'язане: коли індекс шкодить? 🟢**
Кожен індекс треба **оновлювати** при кожному `INSERT`/`UPDATE`/`DELETE` і він займає місце. Багато зайвих індексів уповільнюють запис. Тому індексують те, за чим реально фільтрують/сортують, а не все підряд.

## Q4. Коли використовувати ORM, а коли raw SQL? 🔵

**ORM** зручний для 80% типових операцій: CRUD, прості зв'язки, безпека (параметризація проти SQL-ін'єкцій), типобезпека. **Raw SQL** беруть, коли ORM заважає: складні аналітичні запити (віконні функції, хитрі CTE), тонка оптимізація, специфічні можливості БД, які ORM не покриває.

```ts
// ORM — читабельно й безпечно для типового
const courses = await courseRepo.find({ where: { expertId }, take: 20 });

// raw SQL — коли треба складна аналітика, якої ORM не виражає зручно
const rows = await dataSource.query(
  `SELECT expert_id, COUNT(*) FILTER (WHERE status='active') AS active
   FROM enrollments GROUP BY expert_id`,
);
```

Правило: почни з ORM; переходь на raw SQL точково там, де ORM робить запит неефективним або нечитабельним.

**➕ Пов'язане: що таке query builder як проміжний варіант? 🟢**
Query builder (TypeORM `createQueryBuilder`, Knex) — між ORM і сирим SQL: будуєш запит програмно (методами), лишаючись близько до SQL, але з типами й захистом від ін'єкцій. Зручний для динамічних запитів (фільтри, що залежать від умов).

## Q5. Як писати складніші SQL-запити (GROUP BY, nested SELECT, CTE, window functions)? 🟣

**GROUP BY + агрегати** — згорнути рядки в групи:
```sql
SELECT expert_id, COUNT(*) AS students
FROM enrollments GROUP BY expert_id;
```

**CTE (`WITH`)** — іменований проміжний результат, робить складний запит читабельним:
```sql
WITH active AS (
  SELECT * FROM enrollments WHERE status = 'active'
)
SELECT expert_id, COUNT(*) FROM active GROUP BY expert_id;
```

**Window functions (`PARTITION BY`, `ROW_NUMBER`)** — рахують по «вікну» рядків **без згортання** таблиці. Напр. пронумерувати записи всередині кожного експерта за датою:
```sql
SELECT
  id, expert_id, created_at,
  ROW_NUMBER() OVER (PARTITION BY expert_id ORDER BY created_at DESC) AS rn
FROM enrollments;
-- rn = 1 → найновіший запис кожного експерта (зручно для "останній на групу")
```

**Nested SELECT (підзапит):**
```sql
SELECT * FROM courses
WHERE expert_id IN (SELECT id FROM experts WHERE plan = 'pro');
```

**➕ Пов'язане: чим window-функції відрізняються від GROUP BY? 🔵**
`GROUP BY` **згортає** групу в один рядок (втрачаєш деталі). Window-функція рахує агрегат/номер **по вікну, але лишає всі рядки** — можна поряд бачити і значення рядка, і, скажімо, його ранг у групі. Тому «топ-N на групу» роблять саме через `ROW_NUMBER() OVER (PARTITION BY ...)`.

**➕ Пов'язане: що таке рекурсивний CTE? 🔵**
`WITH RECURSIVE` дозволяє запиту посилатися сам на себе — для ієрархій (дерево коментарів, категорій, орг-структура). Один запит обходить усі рівні дерева без циклів у коді застосунку.

**➕ Пов'язане: у чому різниця WHERE і HAVING? 🟢**
`WHERE` фільтрує рядки **до** групування, `HAVING` — групи **після** агрегації. Напр. `HAVING COUNT(*) > 10` лишає лише групи з понад 10 записами — у `WHERE` так не можна, бо агрегат ще не порахований.

## Q6. Що таке DB triggers (наприклад, для audit log)? 🟢

Тригер — код (функція), що БД **автоматично виконує** при події над таблицею (`INSERT`/`UPDATE`/`DELETE`). Типове застосування — audit log: щоразу при зміні рядка записати, хто і що змінив, без залучення застосунку.

```sql
CREATE OR REPLACE FUNCTION log_course_change() RETURNS trigger AS $$
BEGIN
  INSERT INTO course_audit(course_id, changed_at, old_title, new_title)
  VALUES (NEW.id, now(), OLD.title, NEW.title);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_course_audit
AFTER UPDATE ON courses
FOR EACH ROW EXECUTE FUNCTION log_course_change();
```

**➕ Пов'язане: тригер vs логіка в застосунку — що обрати? 🟢**
Тригер спрацює **завжди**, навіть якщо дані змінили повз застосунок (міграція, ручний запит) — плюс для аудиту/цілісності. Мінус — «прихована» логіка в БД, яку важче побачити й тестувати. Тому бізнес-логіку зазвичай тримають у застосунку, а тригери лишають для наскрізних гарантій (audit, автозаповнення `updated_at`).

**➕ Пов'язане: що таке stored procedure? 🟢**
Збережена процедура — іменований блок SQL/процедурного коду, що зберігається й виконується на боці БД. Корисна для важких операцій «ближче до даних», але ускладнює версіонування логіки (вона поза кодом застосунку).

## Q7. Як працюють нереляційні (NoSQL) бази — коли обирати й як структурувати? 🟢

NoSQL — сімейство БД без жорсткої табличної схеми. Обирають, коли: дані слабо структуровані або схема часто міняється; потрібне горизонтальне масштабування на величезні обсяги; модель доступу — «дістати весь документ за ключем», а не складні join'и.

**Ключова відмінність у моделюванні:** у реляційних БД нормалізують (розкладають), у NoSQL часто **денормалізують** — зберігають дані так, як їх читатимуть, разом в одному документі:

```js
// MongoDB: курс із вкладеними уроками — читається одним запитом, без join
{
  _id: "course1", title: "Intro",
  lessons: [ { title: "Lesson 1" }, { title: "Lesson 2" } ]  // flattening/вкладення
}
```

Обмеження: дублювання даних (оновлення в кількох місцях), слабші транзакції між документами. Масштабування — **шардинг** (розкидання даних за ключем на багато вузлів).

**➕ Пов'язане: що таке CAP-теорема? 🔵**
У розподіленій БД неможливо мати одночасно всі три: **Consistency** (усі бачать однакові дані), **Availability** (система завжди відповідає), **Partition tolerance** (працює при розриві мережі). Оскільки мережеві розриви неминучі, обирають між C і A. Реляційні зазвичай тяжіють до CP, багато NoSQL — до AP.

**➕ Пов'язане: які є типи NoSQL? 🟢**
**Document** (MongoDB — JSON-документи), **key-value** (Redis, DynamoDB — проста мапа), **column-family** (Cassandra — величезні обсяги записів), **graph** (Neo4j — зв'язки/соцграфи). Тип обирають під модель доступу до даних.

## Q8. Які бувають бекапи (incremental, full) і коли який? 🔵

- **Full backup** — повна копія всієї БД. Простий у відновленні, але великий і повільний; робити часто дорого.
- **Incremental backup** — зберігає лише **зміни** з моменту попереднього бекапу. Швидкий і компактний, але для відновлення потрібен ланцюг (останній full + усі incremental після нього).

Типова стратегія: повний бекап раз на тиждень + інкрементальні щодня. Вибір залежить від розміру БД, вікна відновлення (RTO) і допустимої втрати даних (RPO).

**➕ Пов'язане: що таке point-in-time recovery (PITR)? 🟢**
PITR дозволяє відновити БД на **конкретний момент** (напр. «за секунду до помилкового `DELETE`»), комбінуючи базовий бекап і журнал транзакцій (WAL у Postgres). Рятує від логічних помилок, а не лише від апаратних збоїв.

**➕ Пов'язане: чим бекап відрізняється від реплікації? 🔵**
Реплікація тримає **живу копію** БД на іншому вузлі (для доступності й читань) — але помилковий `DELETE` миттєво реплікується теж. Бекап — це **історична точка** для відновлення. Реплікація захищає від збою заліза, бекап — від логічних помилок і корупції; потрібні обидва.

## Q9. Як працювати з транзакціями в ORM і як код впливає на транзакції? 🟣

В ORM транзакцію обгортають навколо кількох операцій; усередині всі записи або застосуються разом, або відкотяться.

```ts
// TypeORM
await dataSource.transaction(async (manager) => {
  const course = await manager.save(Course, { title, expertId });
  await manager.save(Enrollment, { courseId: course.id, userId }); // разом із курсом
  // якщо тут кинути помилку — обидва save відкотяться
});
```

**Як інший код впливає на транзакції (важливо для L3):**
- Довга робота *всередині* транзакції (виклик зовнішнього API, важке обчислення) тримає **блокування** й з'єднання довше → ризик deadlock і вичерпання пулу. Зовнішні виклики роби **поза** транзакцією.
- Не кожна операція має бути в транзакції: якщо запис незалежний, транзакція лише марно тримає ресурси.
- Помилка, «проковтнута» try/catch усередині, може непомітно зламати атомарність — виняток має долітати до межі транзакції, щоб стався відкат.

**➕ Пов'язане: що таке deadlock і як його уникати? 🔵**
Deadlock — дві транзакції взаємно чекають блокування одна одної й зависають (A тримає рядок 1 і хоче 2, B тримає 2 і хоче 1). БД одну з них «вбиває». Уникають: брати блокування в **однаковому порядку**, тримати транзакції короткими, не робити всередині них зовнішніх викликів.

**➕ Пов'язане: optimistic vs pessimistic locking? 🔵**
**Pessimistic** — блокуєш рядок одразу (`SELECT ... FOR UPDATE`), інші чекають; надійно, але знижує паралельність. **Optimistic** — не блокуєш, а тримаєш стовпець `version`; при записі перевіряєш, чи версія не змінилась, інакше — конфлікт і повтор. Optimistic кращий, коли конфлікти рідкі.

## Q10. Що таке profiling SQL і аналіз slow query log? 🔵

Profiling — пошук повільних запитів, щоб знати, що оптимізувати. Два підходи: разово через `EXPLAIN ANALYZE` (один запит) і системно через **slow query log** — журнал, куди БД пише всі запити, що виконувались довше за поріг.

```sql
-- Postgres: логувати запити, що тривали понад 500 мс
ALTER SYSTEM SET log_min_duration_statement = 500;
SELECT pg_reload_conf();
```

Далі дивишся, які запити найповільніші/найчастіші, і оптимізуєш саме їх (індекс, переписати, кеш). Розширення `pg_stat_statements` показує агреговану статистику по запитах (сумарний час, к-сть викликів).

**➕ Пов'язане: як побачити N+1 на рівні БД? 🔵**
N+1 проявляється як **багато однакових запитів** із різними id підряд у логах/трейсі (`SELECT * FROM lessons WHERE course_id = ?` ×100). Помітивши це, замінюють на один запит із JOIN або `IN (...)`. APM-інструменти (Datadog, Sentry) підсвічують такі патерни автоматично.

**➕ Пов'язане: що таке connection pool і навіщо? 🔵**
Відкривати нове з'єднання з БД на кожен запит дорого. Пул тримає набір відкритих з'єднань і видає їх запитам, повертаючи назад після використання. Розмір пулу треба узгоджувати з лімітом з'єднань БД — інакше під навантаженням запити стають у чергу або падають.

## Q11. Навіщо робити shared queries для команди? 🟣

Shared queries — збережені й доступні всій команді запити (у DataGrip, у репозиторії, у BI-інструменті чи внутрішній вікі). Замість того щоб кожен щоразу писав з нуля запит на «активні студенти по експерту», його пишуть один раз, вивіряють і шарять. Плюси: не дублюється робота, менше помилок у критичних аналітичних запитах, новачок одразу має готовий набір.

```sql
-- приклад "командного" запиту, який зберігають і перевикористовують:
-- daily_active_by_expert.sql
SELECT expert_id, COUNT(DISTINCT user_id) AS dau
FROM events
WHERE created_at >= current_date
GROUP BY expert_id;
```

Зберігати можна: у спільному репозиторії (`/queries/*.sql` під версійним контролем), у DataGrip (shared scratch), або як saved queries у BI/адмін-панелі.

**➕ Пов'язане: чому shared queries краще тримати під git? 🟢**
У git запит має історію змін, рев'ю через PR і єдине джерело правди — на відміну від «запиту, що гуляє в чаті». Це той самий підхід «усе як код», що й до інфраструктури: аналітику теж версіонують.
