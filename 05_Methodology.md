## Q1. Що таке GitFlow і які просунуті операції Git треба знати? 🔵

**GitFlow** — модель гілкування з довгоживучими гілками: `main` (реліз), `develop` (інтеграція), + тимчасові `feature/*`, `release/*`, `hotfix/*`. Фіча відгалужується від `develop`, вливається назад; реліз готується в `release/*`, потім у `main`.

Просунуті операції Git для L3:
```bash
git rebase -i HEAD~3        # інтерактивно переписати/склеїти останні коміти
git cherry-pick <hash>      # перенести окремий коміт в іншу гілку
git revert <hash>           # безпечно "відкотити" коміт новим комітом
git stash                   # тимчасово сховати незакомічені зміни
git reflog                  # історія переміщень HEAD — врятує "втрачені" коміти
git bisect                  # бінарний пошук коміта, що вніс баг
```

**➕ Пов'язане: чим `merge` відрізняється від `rebase`? 🔵**
`merge` зберігає історію як є й додає merge-коміт (правдива, але «кущиста» історія). `rebase` переписує коміти поверх іншої гілки — лінійна чиста історія, але змінює хеші. Правило: не робити rebase уже опублікованих спільних гілок.

**➕ Пов'язане: що таке fast-forward merge? 🟢**
Якщо цільова гілка не мала нових комітів, git просто «пересуває» вказівник уперед без merge-коміту. Іноді його вимикають (`--no-ff`), щоб зберегти видимість того, що була окрема фіча-гілка.

## Q2. Чим GitFlow відрізняється від Trunk-based development? 🔵

**Trunk-based** — усі працюють в одній головній гілці (`main`/trunk), вливаючи маленькі зміни часто (кілька разів на день), за короткоживучими гілками (години). Незавершене ховають за feature-флагами.

| | GitFlow | Trunk-based |
|---|---|---|
| Гілки | Багато довгоживучих | Одна головна + короткі |
| Частота інтеграції | Рідше (релізами) | Часто (щодня) |
| Merge-конфлікти | Більші, рідше | Дрібні, часто |
| Підходить | Релізи за розкладом, кілька версій | CI/CD, часті деплої |

Trunk-based — стандарт для команд із сильним CI/CD; GitFlow — коли треба підтримувати кілька версій і релізи за розкладом. Головний недолік GitFlow — довгі гілки → великі болючі merge; недолік trunk-based — вимагає дисципліни, тестів і feature-флагів.

**➕ Пов'язане: що таке feature flag і навіщо в trunk-based? 🔵**
Feature flag — умовний перемикач, що приховує незавершену фічу в проді (`if (flags.newChat) {...}`). Дозволяє вливати код у trunk, не показуючи його користувачам, і вмикати поступово. Основа частих безпечних деплоїв.

**➕ Пов'язане: що таке merge hell і як його уникати? 🟢**
Merge hell — болючі конфлікти, коли довга гілка сильно розійшлась із основною. Уникають частою інтеграцією (trunk-based), маленькими PR і регулярним підтягуванням `main` у свою гілку.

## Q3. Чим Vibe coding відрізняється від Agentic development (AI code engineering)? 🔵

- **Vibe coding** — швидко генеруєш код природною мовою «на відчуття», приймаєш запропоноване, мало вникаючи в деталі. Добре для прототипів, демо, одноразових скриптів, де важлива швидкість, а не надійність.
- **Agentic development** — інженерний підхід: AI-агент працює в керованому процесі зі специфікацією, доступом до інструментів (тести, лінт, репозиторій), ітераціями й перевіркою. Ти керуєш контекстом і критеріями якості, а не просто «приймаєш».

Ключова різниця — **контроль і відповідальність за якість**. Vibe coding пускає деталі; agentic development вбудовує перевірки (тести, ревʼю, spec) у процес, тож результат придатний для продакшну.

**➕ Пов'язане: коли vibe coding доречний, а коли небезпечний? 🟢**
Доречний: прототип, PoC, разовий скрипт, дослідження ідеї. Небезпечний: продакшн-код, безпека, гроші, складна бізнес-логіка — там потрібні spec, тести й ревʼю. Vibe-код часто виглядає робочим, але має приховані діри.

**➕ Пов'язане: що таке "human in the loop"? 🟢**
Принцип, коли людина лишається контрольною точкою — переглядає й затверджує рішення агента перед застосуванням (merge, деплой, зміна даних). Баланс між автоматизацією і відповідальністю.

## Q4. Що таке spec-driven development (AI-assisted) — елементи й послідовність? 🔵

Spec-driven development — підхід, де перед кодуванням складають **специфікацію** (що і як має працювати), і вже за нею AI-агент генерує код. Специфікація — джерело правди й для людини, і для агента.

Типова послідовність:
1. **Requirements / spec** — чіткий опис задачі, вхід/вихід, критерії прийняття.
2. **Plan / design** — розбиття на кроки, контракти, структура.
3. **Generate** — агент реалізує за планом.
4. **Verify** — тести й перевірка відповідності специфікації.
5. **Iterate** — уточнюєш spec → перегенеровуєш, а не «латаєш» наосліп.

Перевага перед «просто попросити код»: чіткий контракт зменшує двозначність, полегшує рев'ю (звіряєш код зі spec) і робить результат відтворюваним.

**➕ Пов'язане: чим spec-driven відрізняється від TDD? 🔵**
TDD — окремий випадок: спершу пишеш **тести** (виконувану специфікацію), потім код, що їх задовольняє. Spec-driven ширший — специфікація може бути документом, контрактом API чи тестами. Обидва ставлять «що має бути» перед «як зробити».

**➕ Пов'язане: чому чіткий spec важливий саме для AI-агентів? 🟢**
Агент заповнює прогалини здогадками; розмитий запит → непередбачуваний результат. Точна специфікація звужує простір інтерпретацій, тож вихід ближчий до потрібного з першого разу й легше перевіряється.

## Q5. Як робити reuse AI-промптів для розробки (skills, workflows, commands)? 🔵

Замість того щоб щоразу писати той самий промпт, повторювані інструкції оформлюють як **перевикористовувані артефакти**:
- **Skills** — багаторазові набори інструкцій/знань під конкретний тип задачі (напр. «як ми пишемо NestJS-контролери»).
- **Workflows** — послідовності кроків для типового процесу (створити фічу: spec → код → тести → PR).
- **Commands** — іменовані шаблони-команди для частих дій (`/review`, `/write-tests`).

```md
<!-- приклад reusable "команди": .agent/commands/write-tests.md -->
# Ціль: покрити вказаний модуль юніт-тестами
- Використовуй Vitest та AAA-структуру
- Мокай зовнішні залежності, не мокай чисту логіку
- Мінімум: happy path + межові випадки + помилки
```

Ідея та сама, що з кодом: винести повторюване в одне місце, версіонувати, ділитися командою — щоб якість і стиль були однакові незалежно від того, хто ставить задачу агенту.

**➕ Пов'язане: що таке контекст-інженерія? 🔵**
Свідоме керування тим, що агент «бачить»: релевантні файли, правила проєкту, приклади — і **прибирання зайвого**, щоб не розмивати фокус. Часто важливіше за формулювання самого промпту: правильний контекст дає правильну відповідь.

**➕ Пов'язане: навіщо файли правил проєкту (напр. AGENTS.md/rules)? 🟢**
Це постійні інструкції, що автоматично додаються в контекст агента (стек, конвенції, чого не робити). Гарантують, що агент дотримується домовленостей команди без повторення їх у кожному промпті.

## Q6. Що таке DRY, KISS, YAGNI і як змусити AI-асистента їх дотримуватись? 🔵

Три принципи простоти:
- **DRY** (Don't Repeat Yourself) — не дублюй знання/логіку; повторюване виноси в одне місце.
- **KISS** (Keep It Simple, Stupid) — обирай найпростіше робоче рішення, не ускладнюй.
- **YAGNI** (You Aren't Gonna Need It) — не роби «про запас» те, що не потрібне зараз.

Як застосувати з AI-асистентом: явно закладати ці принципи в правила/промпт і **розпізнавати порушення** в згенерованому коді — зайві абстракції «на майбутнє» (порушення YAGNI), скопійовані блоки (DRY), надмірно хитрі конструкції (KISS).

```ts
// ❌ порушення YAGNI+KISS: універсальна "фабрика" там, де треба одна функція
class NotificationFactoryBuilderStrategy { /* ... 100 рядків "гнучкості" ... */ }

// ✅ KISS: рівно те, що потрібно зараз
function sendEmail(to: string, body: string) { /* ... */ }
```

**Важливий нюанс:** AI-асистенти схильні **переускладнювати** (додавати абстракції, обробку неможливих випадків). Тому «simplicity is the key» — часто ти маєш активно *прибирати* зайве з їхнього виводу.

**➕ Пов'язане: чи можна перестаратися з DRY? 🔵**
Так. Передчасне усунення «дублювання», яке насправді випадкове (два шматки схожі зараз, але еволюціонують по-різному), створює зв'язність і крихкі абстракції. Правило «rule of three»: виносити, коли повторення реальне й з'явилось тричі, а не за першою схожістю.

**➕ Пов'язане: що таке over-engineering? 🟢**
Рішення, складніше за задачу: зайві шари, конфігурованість, патерни «на виріст». Коштує часу зараз і підтримки потім, а користі не дає. Протиотрута — KISS/YAGNI і чесне питання «це потрібно сьогодні?».

## Q7. Що таке SDLC і які його фази (включно з support & maintenance)? 🔵

SDLC (Software Development Life Cycle) — повний цикл життя ПЗ від ідеї до виводу з експлуатації. Класичні фази:
1. **Requirements** — збір і аналіз вимог.
2. **Design** — архітектура, технічний дизайн.
3. **Implementation** — розробка.
4. **Testing** — перевірка якості.
5. **Deployment** — вихід у продакшн.
6. **Support & Maintenance** — виправлення багів, оновлення, адаптація до нових вимог — **найдовша** фаза за часом і вартістю.

Ключове розуміння для L3: розробка — лише частина; більшість вартості ПЗ припадає на підтримку. Тому код пишуть так, щоб його було легко супроводжувати (читабельність, тести, документація).

**➕ Пов'язане: чим Waterfall відрізняється від Agile SDLC? 🔵**
**Waterfall** проходить фази послідовно один раз (усі вимоги → весь дизайн → …); зміни дорогі. **Agile** проходить ці ж фази **ітеративно** маленькими циклами, даючи робочий продукт часто й приймаючи зміни. Agile — не заміна фаз SDLC, а інший спосіб їх проходити.

**➕ Пов'язане: що таке technical debt у контексті maintenance? 🟢**
Технічний борг — накопичені компроміси (швидкі рішення, пропущені тести), що уповільнюють подальшу розробку, як «відсотки». Керування ним — частина maintenance: свідомо виділяти час на рефакторинг, інакше борг «з'їдає» швидкість команди.

## Q8. Як якісно покривати функціонал юніт-тестами за допомогою AI-асистентів? 🟣

AI пришвидшує написання тестів, але якість лишається на тобі: асистент легко генерує «тести заради покриття», що нічого не перевіряють. Твоя задача — направити його на **осмислені** випадки.

Що вимагати від AI-згенерованих тестів:
- **Happy path + межові випадки + помилки** (не лише «все добре»).
- **AAA-структура** (Arrange-Act-Assert), зрозумілі назви.
- Мокати зовнішні залежності (мережа, БД), **не** мокати чисту логіку.
- Перевіряти поведінку/результат, а не деталі реалізації.

```ts
// осмислений набір під бізнес-функцію — те, що просиш в асистента
describe("remainingBudget", () => {
  it("рахує залишок", () => expect(remainingBudget(10000, 3000)).toBe(7000));
  it("не йде в мінус при перевитраті", () => expect(remainingBudget(10000, 12000)).toBe(0));
  it("нульовий бюджет", () => expect(remainingBudget(0, 0)).toBe(0));
});
```

Червоний прапорець — тест, що просто повторює реалізацію (`expect(mock).toHaveBeenCalled()` без перевірки результату) або підганяється під поточний вивід. Такі тести дають хибне відчуття безпеки.

**➕ Пов'язане: чому не варто гнатися за 100% покриттям? 🔵**
Покриття показує, який код *виконали* тести, а не чи він *правильно перевірений*. Можна мати 100% без жодного осмисленого `expect`. Важливіше вкрити критичну (грошову, безпекову) логіку якісно, ніж досягти цифри тривіальними тестами.

**➕ Пов'язане: що таке тест-піраміда? 🟢**
Багато швидких **unit**-тестів в основі, менше **integration**, зовсім мало повільних **e2e** зверху. Пропорція дає надійність без крихкості й повільності. Антипатерн — «морозиво навпаки»: купа e2e і мало unit.

## Q9. Що таке Agile (Scrum, Kanban) і що таке не-Agile методологія? 🔵

**Agile** — підхід із короткими ітераціями, частою поставкою робочого продукту, гнучкістю до змін і тісною співпрацею.
- **Scrum** — робота **спринтами** (1–2 тижні) з фіксованим набором задач, ролями (Product Owner, Scrum Master, команда) і церемоніями.
- **Kanban** — безперервний потік задач на дошці з колонками (To Do → In Progress → Done) і **лімітами WIP** (скільки задач одночасно в роботі); без фіксованих спринтів.

**Не-Agile приклад — Waterfall:** послідовні фази один раз, повний план наперед, зміни дорогі й небажані. Підходить, коли вимоги стабільні й відомі заздалегідь (регульовані/апаратні проєкти).

**➕ Пов'язане: коли Kanban кращий за Scrum? 🔵**
Kanban — коли робота непередбачувана й потокова (підтримка, багфікс, потік дрібних задач): нема сенсу в спринтах, важливий безперервний потік і WIP-ліміти. Scrum — коли можна планувати ітерацію наперед і корисний ритм спринтів.

**➕ Пов'язане: що таке velocity і чи це метрика продуктивності? 🟢**
Velocity — скільки story points команда закриває за спринт; використовується для **прогнозування** обсягу наступних спринтів. Це не показник «продуктивності» для порівняння команд і не ціль сама по собі — інакше стає предметом маніпуляцій.

## Q10. Які є Scrum-церемонії, що вони вирішують і скільки тривають? 🔵

Чотири основні події спринту (тривалість орієнтовна для 2-тижневого спринту):
- **Sprint Planning** (до ~2–4 год) — команда обирає, що зробить у спринті, і як.
- **Daily Standup** (15 хв щодня) — синхронізація: що зробив, що робитиму, які блокери. Не звіт менеджеру, а координація команди.
- **Sprint Review** (~1–2 год) — демонстрація зробленого стейкхолдерам, збір фідбеку.
- **Sprint Retrospective** (~1–1.5 год) — команда обговорює процес: що працювало, що покращити.

Ключове: standup — про блокери й синхронізацію (коротко!), retro — про **процес** (не про людей), review — про **продукт**.

**➕ Пов'язане: що таке backlog refinement (grooming)? 🔵**
Регулярна сесія, де команда уточнює й оцінює майбутні задачі беклогу (деталізує, розбиває великі, ставить оцінки), щоб на Planning брати вже готові. Тримає беклог «здоровим» і планування швидким.

**➕ Пов'язане: що таке Definition of Done? 🟢**
Спільний чекліст, коли задача вважається **завершеною** (код + тести + рев'ю + задеплоєно + документація). Прибирає суперечки «майже готово» і тримає єдиний стандарт якості.

## Q11. Що таке event-driven architecture (переваги, недоліки, компоненти)? 🟢

Event-driven — компоненти спілкуються через **події** (повідомлення), а не прямими викликами. Одна частина публікує подію («замовлення створене»), інші реагують асинхронно, не знаючи одна про одну.

**Компоненти:** черги/брокери (queues — RabbitMQ, Kafka, SQS), функції-обробники (consumers), тригери (що породжують події).

**Переваги:** слабка зв'язність (легко додати нового споживача), масштабованість, стійкість (черга згладжує сплески). **Недоліки:** складніше відлагоджувати (потік асинхронний), потрібні logging/tracing через усі кроки, і стратегія retry для збоїв.

**Time-driven vs user-driven:** time-driven — за розкладом (нічний перерахунок звітів); user-driven — у відповідь на дію користувача. Trade-off: миттєве (user-driven) дорожче й складніше, ніж відкладене пакетне (time-driven), тож частину роботи виносять у фон, де затримка прийнятна.

**➕ Пов'язане: що таке черга повідомлень (message queue) і навіщо? 🟢**
Черга приймає повідомлення й тримає їх, поки споживач не обробить. Розв'язує виробника й споживача в часі: сплеск подій не «завалить» обробника — він розгрібає чергу у своєму темпі. Дає буферизацію, ретраї, рівномірне навантаження.

**➕ Пов'язане: що таке ідемпотентність споживача і чому вона критична? 🟢**
Повідомлення можуть доставитись **двічі** (at-least-once). Тому обробник має бути ідемпотентним — повторна обробка тієї самої події не подвоює ефект (напр. не списує гроші двічі). Реалізують через перевірку id уже опрацьованих подій.

## Q12. Чим відрізняються prototype, PoC і MVP? 🟢

Три різні речі з різними цілями:
- **PoC (Proof of Concept)** — перевірити *технічну* здійсненність ідеї («чи взагалі можливо це зробити?»). Часто викидається після відповіді.
- **Prototype** — показати, *як це виглядатиме/працюватиме* (макет, кликабельний UI), зібрати фідбек про UX. Не робочий продукт.
- **MVP (Minimum Viable Product)** — найпростіша **робоча** версія з реальною цінністю, яку віддають користувачам, щоб перевірити гіпотезу на ринку.

Логіка: PoC відповідає «чи можливо технічно», prototype — «чи так має виглядати», MVP — «чи це потрібно людям».

**➕ Пов'язане: чому MVP часто розуміють неправильно? 🟢**
MVP — не «недороблений продукт», а **мінімальний, але цілісний** зріз цінності для перевірки гіпотези. Помилка — робити «половину великого плану»; правильно — вузьке, але робоче рішення однієї реальної проблеми.

## Q13. Що таке SDD, BDD, DDD? 🟢

Три різні підходи (плутають через схожі абревіатури):
- **SDD (Spec-Driven Development)** — спочатку специфікація, потім код за нею.
- **BDD (Behaviour-Driven Development)** — описують поведінку зрозумілою мовою (Given-When-Then), що є і специфікацією, і тестами; зближує розробників і бізнес.
- **DDD (Domain-Driven Development/Design)** — моделюють код навколо **бізнес-домену** та його мови (ubiquitous language), з чіткими межами (bounded contexts).

```gherkin
# BDD: сценарій зрозумілий і бізнесу, і як тест
Scenario: Студент записується на курс
  Given студент авторизований
  When він натискає "Записатись" на курсі
  Then він бачить курс у "Мої курси"
```

**➕ Пов'язане: що таке ubiquitous language у DDD? 🟢**
Єдина термінологія, якою користуються і розробники, і бізнес, і код (класи/методи названі доменними словами). Прибирає «переклад» між мовою бізнесу й кодом — менше непорозумінь.

**➕ Пов'язане: що таке bounded context? 🟢**
Чітка межа, всередині якої термін має один конкретний сенс. «Користувач» у контексті білінгу й у контексті навчання — різні моделі; DDD розділяє їх, замість «одного класу User на все».

## Q14. Що таке Big O і як оцінювати/оптимізувати продуктивність коду? 🔵

Big O описує, як зростає час (або пам'ять) алгоритму зі зростанням розміру вводу `n` — незалежно від заліза.

| Позначення | Назва | Приклад |
|---|---|---|
| `O(1)` | константа | доступ до елемента масиву за індексом |
| `O(log n)` | логарифм | бінарний пошук, індекс у БД |
| `O(n)` | лінійно | один прохід масивом |
| `O(n log n)` | | ефективне сортування |
| `O(n²)` | квадрат | вкладені цикли по одному масиву |

```ts
// ❌ O(n²): для кожного елемента шукаємо у масиві заново
const dup = arr.filter((x, i) => arr.indexOf(x) !== i);

// ✅ O(n): один прохід із Set
const seen = new Set<number>(); const dups = new Set<number>();
for (const x of arr) { if (seen.has(x)) dups.add(x); else seen.add(x); }
```

Практично: помічати випадковий `O(n²)` (вкладені цикли, `find`/`includes` у циклі), заміняти на `Map`/`Set` (`O(1)` пошук). Але не оптимізувати передчасно — спершу виміряй, чи це вузьке місце.

**➕ Пов'язане: чому іноді O(n²) прийнятне? 🔵**
Для малих `n` простий `O(n²)` може бути швидшим і читабельнішим за складний `O(n log n)` (менші константи, менше накладних витрат). Big O — про **зростання**, тож він важить для великих даних; для десятка елементів різниця неістотна.

**➕ Пов'язане: у чому різниця часової та просторової складності? 🟢**
Часова — скільки **часу**, просторова — скільки **пам'яті** росте з `n`. Часто це trade-off: кеш/`Set` пришвидшує пошук (менше часу) ціною пам'яті. Обирають, що дефіцитніше в конкретній задачі.

## Q15. Що таке PRD і TDD (technical design document)? 🟢

Два документи на різних етапах:
- **PRD (Product Requirements Document)** — *що* і *навіщо* будуємо: проблема, користувачі, функціональні вимоги, критерії успіху. Пише зазвичай продукт-менеджер; мова — бізнесу й користувача.
- **TDD (Technical Design Document)** — *як* реалізуємо: архітектура, моделі даних, API, компроміси, ризики. Пише інженер на основі PRD; мова — технічна.

Логіка: PRD → TDD → код. PRD фіксує «навіщо», TDD — «як», і разом вони дають спільне розуміння до початку розробки.

**➕ Пов'язане: навіщо писати design doc перед кодом? 🟢**
Design doc змушує продумати рішення й отримати рев'ю **до** написання коду, коли зміни дешеві. Ловить проблеми архітектури рано, вирівнює команду й лишає документацію рішень (чому саме так). Особливо цінно для великих/ризикових змін.

## Q16. Що таке Creational patterns (породжувальні патерни)? 🔵

Патерни, що керують **створенням об'єктів**, приховуючи деталі інстанціювання. Основні: **Singleton** (один екземпляр на систему), **Factory** (створення через метод, а не `new`, — обираємо клас у рантаймі), **Builder** (покрокове створення складного об'єкта), **Prototype** (клонування).

```ts
// Factory: клієнт не знає конкретного класу, лише інтерфейс
interface Notifier { send(msg: string): void; }
function createNotifier(kind: "email" | "sms"): Notifier {
  return kind === "email" ? new EmailNotifier() : new SmsNotifier();
}
```

**➕ Пов'язане: чому Singleton часто вважають антипатерном? 🔵**
Singleton вводить прихований глобальний стан і жорстку залежність — ускладнює тестування (не підставиш mock) і паралельність. У сучасних застосунках його роль зазвичай виконує **DI-контейнер** (один екземпляр як провайдер), даючи ту саму «єдиність» без глобальної змінної.

**➕ Пов'язане: коли Builder кращий за конструктор? 🟢**
Коли об'єкт має багато опційних параметрів — замість «телескопічного» конструктора з десятком аргументів, Builder дає читабельне покрокове створення (`.setA().setB().build()`).

## Q17. Що таке Behavioural patterns (поведінкові патерни)? 🔵

Патерни, що керують **взаємодією й розподілом обов'язків** між об'єктами. Основні: **Strategy** (взаємозамінні алгоритми), **Observer** (підписка на події), **Command** (дія як об'єкт), **State** (поведінка залежить від стану), **Iterator**.

```ts
// Strategy: алгоритм передається ззовні, легко замінити
interface PricingStrategy { price(base: number): number; }
const regular: PricingStrategy = { price: (b) => b };
const blackFriday: PricingStrategy = { price: (b) => b * 0.7 };

function checkout(base: number, strategy: PricingStrategy) {
  return strategy.price(base); // поведінку обираємо в рантаймі
}
```

**➕ Пов'язане: чим Strategy відрізняється від простого if/else? 🔵**
`if/else` вшиває варіанти в одну функцію — щоб додати новий, треба її міняти (порушує Open-Closed). Strategy виносить кожен варіант в окрему одиницю; додати новий = додати стратегію, не чіпаючи наявний код. Виправдано, коли варіантів багато й вони ростуть.

**➕ Пов'язане: де Observer у знайомих інструментах? 🟢**
Observer — основа реактивності: підписка на події DOM, RxJS Observables, слухачі стору (Redux subscribe), event emitters у Node. Один об'єкт сповіщає багатьох підписників про зміну.

## Q18. Що таке Structural patterns (структурні патерни)? 🔵

Патерни, що описують, як **композувати об'єкти/класи** в більші структури. Основні: **Adapter** (сумісність несумісних інтерфейсів), **Decorator** (додати поведінку без зміни класу), **Facade** (простий інтерфейс до складної системи), **Proxy** (замісник, що контролює доступ).

```ts
// Adapter: чужий сервіс під наш інтерфейс
interface Storage { save(key: string, data: string): Promise<void>; }

class S3Adapter implements Storage {          // приводимо S3 SDK до нашого Storage
  constructor(private s3: S3Client) {}
  async save(key: string, data: string) { await this.s3.putObject(key, data); }
}
```

**➕ Пов'язане: чим Decorator відрізняється від успадкування? 🔵**
Успадкування додає поведінку статично на етапі компіляції й веде до вибуху підкласів. Decorator «загортає» об'єкт і додає поведінку **в рантаймі**, композуючи кілька обгорток (логування + кеш + ретрай). Гнучкіше й без роздування ієрархії.

**➕ Пов'язане: що таке Facade і навіщо? 🟢**
Facade дає один простий вхід до складної підсистеми (напр. `PaymentService.charge()` ховає роботу з кількома API). Клієнт не знає внутрішньої складності — менше зв'язності й легше міняти нутрощі.

## Q19. Які є способи оцінки User Story (time, Fibonacci, T-shirt)? 🔵

Оцінка потрібна для планування й прогнозу. Три поширені підходи:
- **Time-based** — оцінка в годинах/днях. Інтуїтивно, але оманливо точно і легко промахнутись.
- **Story points (Fibonacci: 1, 2, 3, 5, 8, 13…)** — оцінюють **відносну складність/зусилля**, а не час. Зростаючі проміжки відображають, що чим більша задача, тим більша невизначеність.
- **T-shirt sizes (XS, S, M, L, XL)** — грубе відносне оцінювання для раннього етапу/великих задач, коли деталей ще мало.

Ключова ідея story points: оцінюють **складність відносно інших задач**, а не абсолютний час — це стабільніше, бо різні люди працюють з різною швидкістю, а відносна складність спільна.

**➕ Пов'язане: що таке Planning Poker? 🔵**
Техніка спільної оцінки: кожен одночасно показує свою оцінку (картку), розбіжності обговорюють. Прибирає ефект «якоря» (коли перший озвучений варіант впливає на решту) і виявляє різне розуміння задачі.

**➕ Пов'язане: чому story points не переводять напряму в години? 🟢**
Points вимірюють складність/невизначеність, а не час; зв'язок із годинами непрямий і різний для кожного. Переклад «1 point = X годин» вбиває сенс — краще прогнозувати через velocity (points за спринт історично).
