Конфігурація: репо **KE-training**, **публічний**, під особистим акаунтом **Sensation02**; Claude Actions — за підпискою (OAuth-токен). Прогрес: кроки **1, 2, 5, 8 виконано** агентним циклом (git + репозиторій; структура `levels/` + збірка; синк `/api/state` + клієнт; `make check` + Claude Actions зі SHA-піном action-а і гейтом акторів); кроки **3, 4 — ручні**, за чеклистом `SETUP.md`; крок **6** — наскрізна перевірка, після 3–4; крок **7** (PWA) — після кроку 6.

**Мета:** тренажер доступний за постійним URL з будь-якого пристрою і встановлюється на телефон як застосунок (офлайн включно); у кожного користувача свій прогрес, синхронізований між його пристроями; матеріали редагуються через git — руками або чатом із Claude; додавання нового рівня (L2/L4) — це «покласти папку з .md», а не переробка.

## Архітектура

- **Хостинг + збірка:** публічний GitHub-репо `Sensation02/KE-training` → Cloudflare Pages. На кожен push CI збирає одну сторінку на рівень (`/l3/`, згодом `/l2/`…), корінь — індекс рівнів.
- **Автентифікація:** Cloudflare Access перед усім сайтом — вхід за кодом на email. Оскільки репо публічний, роль Access — **ідентичність для прогресу і захист `/api`**, а не приховування контенту (матеріали в репо видимі всім — свідоме рішення). Сайт лишаємо цілком під Access: сторінка й API мають бути в одному auth-контурі, інакше fetch упирається в логін-редирект.
- **Прогрес:** Pages Function `/api/state` (GET/PUT) + Cloudflare KV. Ідентичність — із заголовка Access (`Cf-Access-Authenticated-User-Email`). Ключ: `state:<email>:<рівень>`. Конфлікти між пристроями одного користувача: last-write-wins (`updatedAt`). localStorage — офлайн-кеш.
- **PWA:** manifest + іконки (встановлюваність) і service worker зі стратегією **network-first** для сторінок; `/api/*` не кешується ніколи.
- **Редагування контенту:** правка `.md` → push → сайт перезбирається за ~1–2 хв. Канали: руками (VS Code, веб-редактор GitHub), чатом через Claude Code (локально або claude.ai/code) та `@claude` в Issues/PR (крок 8).

## Розширення на L2/L4 (потім, коли з'явиться контент)

Додати папку `levels/L2/` з розділами й чеклистом → push. Збірка сама підхопить рівень, він з'явиться в індексі, прогрес у KV уже партиціонований за рівнем. Нуль змін у функції, auth, клієнті та PWA. **Зараз контент L2/L4 не пишемо** — лише структура.

- **Борг:** `study_template.html` захардкожує `<title>`/`<h1>` («L3 Full-stack») і кольори категорій `CAT_COLOR` під L3 — при додаванні L2 це треба параметризувати за рівнем/метаданими розділу, інакше сторінка L2 покаже L3-заголовок і категорії без кольору.

## Зафіксовані рішення

- Репо **публічний** — свідоме рішення: матеріали відкриті, Access захищає лише прогрес/API і дає ідентичність. (Якщо колись передумаєте — перемикання на private не ламає нічого.)
- Cloudflare Access замість секрет-токенів; сайт цілком під Access (один auth-контур зі сторінкою та API).
- Ключ KV одразу містить рівень — без міграцій при розширенні.
- Корінь сайту — індекс рівнів; L3 живе на `/l3/` (URL стабільні назавжди).
- Last-write-wins для конфліктів між пристроями одного користувача.
- PWA: network-first, `/api/*` без кешу; PWA — строго після синку (iOS може стирати localStorage — сервер уже страхує).
- Правки через `@claude` — тільки через PR з рев'ю (не прямі коміти в main).
- Claude Actions — за підпискою (OAuth-токен), без окремого API-білінгу.
- Без власної бази даних і реєстрації (YAGNI).
