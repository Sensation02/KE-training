## Q1. Що таке GitFlow і які просунуті операції Git треба знати? 🔵

**🎯 Відповідь на іспиті:** GitFlow — це модель гілкування з двома довгоживучими гілками — `main`, де кожен коміт відповідає релізу в проді, і `develop`, куди інтегрується все готове, — плюс тимчасові гілки `feature/*`, `release/*` і `hotfix/*`. Вона розводить «готове до релізу» і «в роботі» по різних гілках, тому зручна для релізів за розкладом. З просунутих операцій на практиці найчастіше потрібні `rebase -i`, щоб причесати історію перед PR, `cherry-pick` — перенести окремий коміт в іншу гілку, `bisect` — бінарним пошуком знайти коміт із багом, `stash` — сховати незакомічене заради термінового фікса, і `reflog`, який пам'ятає кожне переміщення HEAD і рятує «втрачені» коміти після `reset --hard`. Важливий нюанс: операції, що переписують історію (`rebase`, `reset`), безпечні лише на своїх ще не опублікованих гілках.

Уяви команду з п'яти розробників, яка релізить раз на два тижні. Всі комітять в одну гілку: хтось доробив пошук курсів, хтось на півдорозі переписує білінг — і настав день релізу. Що деплоїти? У гілці перемішані готове й недороблене. **GitFlow** — це домовленість, яка розводить «готове до релізу» і «в роботі» по різних гілках, щоб таке питання взагалі не виникало.

Спершу нагадаємо чотири слова, на яких тримається все нижче. **Коміт** — збережений знімок проєкту, що посилається на попередній коміт (тому історія — ланцюжок знімків). **Хеш** — унікальний id коміта (`a1b2c3…`), за яким на нього можна послатись. **Гілка** — не «копія коду», а легкий **рухомий вказівник** на коміт: новий коміт просто пересуває вказівник уперед. **HEAD** — вказівник «де я зараз». Майже вся «магія» просунутих операцій — це пересування цих вказівників, і саме тому вона безпечніша, ніж здається.

Дві довгоживучі гілки: `main` (кожен коміт = версія, що була в проді) і `develop` (інтеграція всього готового). Тимчасові: `feature/*` (одна фіча), `release/*` (заморозка перед релізом: тільки фікси, без нових фіч), `hotfix/*` (терміновий фікс проду — від `main`, вливається і в `main`, і в `develop`, щоб фікс не загубився при наступному релізі).

```text
main     ──●──────────────────────●────────●──→   кожен коміт тут = реліз у проді
            \                    /↑        ↑
             \        release/1.2 │  hotfix/critical-bug (від main,
              \          ●───●───●│         вливається в main І в develop)
               \        /         ↓
develop  ──●────●──●───●───●──────●───→           сюди інтегрується все готове
                 \        /
feature/          ●──●───●                        feature/course-search:
                                                  гілка від develop → назад у develop
```

Просунуті операції найкраще вчити не списком команд, а **сценаріями**, в яких вони рятують:

**Сценарій 1: випадково закомітив у `main` замість фіча-гілки.** Коміт не втрачаємо — переносимо:
```bash
git branch feature/oops        # ставимо на цей коміт нову гілку (вказівник)
git reset --hard origin/main   # origin/main = стан main на сервері (remote);
                               # повертаємо локальний main туди — коміт "зникає" з main
git switch feature/oops        # ...але живе у новій гілці, працюємо далі там
```

**Сценарій 2: перед PR у гілці 8 комітів виду "fix", "fix2", "typo".** Рев'юеру потрібна історія, яку можна читати. `git rebase -i HEAD~8` відкриває список комітів: `squash` склеює коміт із попереднім, `reword` міняє повідомлення, `drop` викидає. На виході — 2–3 осмислені коміти. Це переписує хеші, тому робити тільки на **своїй ще не влитій** гілці.

**Сценарій 3: тести падають, баг з'явився десь за останні 50 комітів.** Замість перевіряти всі 50 — бінарний пошук через `bisect`:
```bash
git bisect start
git bisect bad                 # поточний коміт — зламаний
git bisect good HEAD~50        # 50 комітів тому — точно працювало
# git сам чекаутить середину діапазону → запускаєш тест → відповідаєш:
git bisect good                # (або bad) — діапазон звужується вдвічі
# ~6 кроків (log₂ 50) — і git називає точний коміт-винуватець
git bisect reset               # повернутися, де був
```

**Сценарій 4: зробив `reset --hard` і "втратив" коміти.** Насправді коміти ще живі: git видаляє недосяжні об'єкти не одразу, а `reflog` пам'ятає **кожне** переміщення HEAD:
```bash
git reflog                     # a1b2c3 HEAD@{2}: commit: fix enrollment  ← ось він
git reset --hard a1b2c3        # повертаємось на "втрачений" коміт
```

**Сценарій 5: у проді потрібен один конкретний фікс із `develop`, а релізити весь `develop` не можна.** `git cherry-pick <hash>` переносить окремий коміт у поточну гілку (створює новий коміт з тими самими змінами).

**Сценарій 6: посеред роботи прилетів терміновий баг, а в тебе незакомічене напівготове.** `git stash` ховає незакомічені зміни у "кишеню", даючи чисте робоче дерево; після фіксу — `git stash pop` повертає все як було.

**Підсумок-якір:** GitFlow — модель гілкування з довгоживучими `main`/`develop` і тимчасовими feature/release/hotfix гілками, зручна для релізів за розкладом. З просунутих операцій: `rebase -i` — причесати історію перед PR, `cherry-pick` — перенести окремий коміт, `revert` — безпечно скасувати опублікований, `bisect` — бінарним пошуком знайти коміт із багом, `reflog` — знайти "втрачені" коміти.

**➕ Пов'язане: чим `merge` відрізняється від `rebase`? 🔵**
У сценаріях вище ми двічі переписували історію (`rebase -i`, `reset`) — merge і rebase різняться саме ставленням до історії. `merge` зберігає її як є і додає merge-коміт із двома батьками: правдиво, але «кущисто». `rebase` бере твої коміти і **перезаписує** їх поверх іншої гілки — історія лінійна й чиста, але всі хеші нові. Звідси головне правило: rebase — тільки для своїх неопублікованих гілок; якщо гілку вже хтось стягнув, переписані хеші зламають йому історію. Типовий робочий цикл: `git fetch` → `git rebase origin/main` у своїй фіча-гілці (тримаєш її свіжою і лінійною) → merge у main через PR.

**➕ Пов'язане: що таке fast-forward merge? 🟢**
Окремий випадок merge: якщо в цільовій гілці не з'явилось нових комітів після відгалуження, git-у нема що «зливати» — він просто пересуває вказівник гілки вперед, без merge-коміту. Історія виглядає так, ніби гілки й не було. Саме тому його іноді вимикають (`git merge --no-ff`): merge-коміт лишає слід «тут була окрема фіча», що зручно для навігації історією і відкату фічі цілком.

**➕ Пов'язане: чим `revert` відрізняється від `reset`? 🔵**
Обидва «скасовують» коміт, але по-різному. `reset` пересуває вказівник гілки назад — коміт зникає з історії; безпечно лише для неопублікованого (сценарій 1). `revert` створює **новий** коміт із протилежними змінами — історія тільки росте, нічого не переписується, тому це єдиний безпечний спосіб відкотити те, що вже в `main` і в усіх колег. Правило: своє локальне — reset, спільне опубліковане — revert.

## Q2. Чим GitFlow відрізняється від Trunk-based development? 🔵

**🎯 Відповідь на іспиті:** GitFlow — модель із багатьма довгоживучими гілками і рідкою інтеграцією релізами, а trunk-based development — протилежна ставка: всі вливають маленькі зміни в одну головну гілку (trunk) по кілька разів на день, і гілки живуть години, а не тижні. Головна різниця — у ціні інтеграції: у GitFlow довгі гілки розходяться і дають великі болючі merge-конфлікти, у trunk-based той самий біль розмазується на щоденні дрібні дози, але потрібні сильний CI, швидкі тести і feature flags, щоб недороблене їхало в прод вимкненим. Обираю за способом релізу: SaaS із безперервним деплоєм — trunk-based або його м'яка форма з короткими PR-гілками на день-два; кілька підтримуваних версій одночасно (мобільний застосунок, on-premise) — GitFlow чи спрощений варіант із release-гілками.

GitFlow (Q1) вирішує проблему «що готове до релізу», але створює нову: гілки живуть **тижнями**. Поки твоя `feature/billing` зріє три тижні, `develop` втік уперед на сотню комітів — і в день merge на тебе падає конфлікт на 40 файлів, який уже ніхто до кінця не розуміє. Що довше гілки не інтегруються, то більше вони розходяться — біль merge росте нелінійно.

**Trunk-based development** — протилежна ставка: всі працюють в одній головній гілці (trunk = `main`), вливаючи **маленькі** зміни **часто** (в ідеалі кілька разів на день). Гілки або відсутні, або живуть години. Інтеграційний біль нікуди не зникає — він розмазується на щоденні дрібні дози: конфлікт на 3 рядки, який ти щойно писав і пам'ятаєш, замість конфлікту на 40 файлів тритижневої давнини.

Виникає очевидне питання: а як влити в trunk **недороблену** фічу, не зламавши прод? Відповідь — **feature flags**: код їде в прод вимкненим, за умовним перемикачем (детальніше в похідному питанні нижче).

| | GitFlow | Trunk-based |
|---|---|---|
| Гілки | Багато довгоживучих | Одна головна + короткі (години) |
| Частота інтеграції | Рідко (релізами) | Кілька разів на день |
| Merge-конфлікти | Великі, рідкісні, болючі | Дрібні, часті, дешеві |
| Незавершена робота | Живе у гілці | У trunk за feature-флагом |
| Вимагає | Дисципліни гілкування | Сильного CI, тестів, флагів |
| Підходить | Релізи за розкладом, кілька підтримуваних версій | CI/CD, часті деплої |

**Як обирати.** Якщо продукт деплоїться безперервно (типовий SaaS — вебпродукт, який усі користувачі отримують з одного сервера, тому «версія» одна й завжди остання; плюс CI/CD — автоматичним конвеєром, що на кожен пуш збирає код, ганяє тести і вміє задеплоїти без ручних кроків) — trunk-based або його м'яка форма (короткі PR-гілки на 1–2 дні + обов'язковий CI). Якщо треба підтримувати кілька версій одночасно (мобільний застосунок, on-premise у клієнтів, реліз «версії 2.3» за контрактом) — GitFlow або спрощений варіант із release-гілками. Головний недолік GitFlow — merge hell на довгих гілках; головна ціна trunk-based — без швидких надійних тестів кожен пуш у trunk може зламати всіх.

**Підсумок-якір:** GitFlow — багато довгоживучих гілок, рідка інтеграція, merge hell; trunk-based — маленькі зміни в одну гілку щодня, потребує сильного CI і feature-флагів. Що рідша інтеграція, то дорожчий merge.

**➕ Пов'язане: що таке feature flag і навіщо в trunk-based? 🔵**
Це відповідь на питання «як жити з недоробленим кодом у trunk», яке trunk-based створює сам собі. Feature flag — умовний перемикач у коді (`if (flags.newChat) { ... }`), значення якого живе у конфігу чи сервісі флагів, а не в коді. Незавершена фіча їде в прод **вимкненою**: код інтегрується щодня (немає довгої гілки), але користувачі його не бачать. Бонуси: вмикання поступово (5% користувачів → 50% → всі), миттєвий «вимикач» без деплою, якщо фіча зламалась, A/B-тести. Ціна: флаги треба **прибирати** після повного розкату, інакше код заростає мертвими `if`-ами.

**➕ Пов'язане: що таке merge hell і як його уникати? 🟢**
Це той самий «конфлікт на 40 файлів» з початку відповіді: гілка так довго жила окремо, що злити її назад дорожче, ніж переписати. Профілактика випливає з причини (розбіжність росте з часом): інтегруватися часто (trunk-based), різати роботу на маленькі PR, і якщо довга гілка неминуча — регулярно підтягувати в неї `main` (rebase або merge), щоб розбіжність не накопичувалась.

**➕ Пов'язане: як у trunk-based встигати робити код-рев'ю, якщо вливатися треба щодня? 🔵**
Часта інтеграція не скасовує рев'ю — вона змінює його форму. Маленький PR (до ~200 рядків) рев'юється за 10 хвилин, а не за півдня, тож короткі PR-гілки «до кінця дня» цілком сумісні з trunk-based. Команди з найвищим темпом ідуть далі: pair programming як безперервне рев'ю або пост-коміт-рев'ю для низькоризикових змін. Правило незмінне: що менша зміна, то швидше і якісніше її рев'ю.

## Q3. Чим Vibe coding відрізняється від Agentic development (AI code engineering)? 🔵

**🎯 Відповідь на іспиті:** Vibe coding — це генерація коду природною мовою «на відчуття»: описав, прийняв, запустилось — їдемо далі, деталі реалізації свідомо не вивчаються. Agentic development (AI code engineering) — інженерний процес, де та сама модель працює за специфікацією, має доступ до інструментів (запуск тестів, лінтер, читання репозиторію), ітерує до проходження перевірок, а людина затверджує план і рев'юїть результат проти spec. Тобто різниця не в моделі, а в процесі навколо неї і в тому, на чому тримається відповідальність за якість. Vibe coding легітимний для прототипів, демо і разових скриптів, але небезпечний у продакшні: я не можу відповідати за код, який не розумію, — а за продакшн-код відповідаю я, не модель.

Ситуація, знайома кожному, хто працює з AI-асистентами: просиш «зроби форму логіну з JWT» (JWT — токен, яким фронтенд підтверджує бекенду, хто користувач), отримуєш код, який виглядає робочим, натискаєш «прийняти», запускаєш — працює. Через два тижні з'ясовується: токен лежить у localStorage без терміну життя, refresh-логіки немає, помилки мережі мовчки ковтаються. Код був згенерований і прийнятий, але ніким по-справжньому **не спроєктований і не перевірений**. Це і є пастка, навколо якої розрізняють два підходи.

- **Vibe coding** — генеруєш код природною мовою «на відчуття»: описав → прийняв → якщо запустилось, їдемо далі. Деталі реалізації свідомо не вивчаєш. Це **легітимний режим** для прототипів, демо, разових скриптів — там швидкість важливіша за надійність, а код житиме день.
- **Agentic development (AI code engineering)** — інженерний процес, де AI-агент є виконавцем, а ти — інженером, що керує. Агент працює за **специфікацією** (Q4), має доступ до інструментів (запуск тестів, лінтер, читання репозиторію), ітерує до проходження перевірок, а ти рев'юїш результат проти spec. Різниця не в моделі — та сама LLM (велика мовна модель), — а в **процесі навколо неї**.

Як виглядає agentic-цикл на практиці, на прикладі фічі «запис студента на курс»:
1. Ти пишеш spec: API-контракт, правила (не можна записатись двічі; місця обмежені), критерії прийняття.
2. Агент читає spec + релевантний код репозиторію, пропонує план — ти його **затверджуєш або правиш** (найдешевший момент зловити хибний напрямок).
3. Агент пише код і тести, ганяє їх сам, ітерує до зеленого.
4. Ти рев'юїш diff проти spec — не «чи виглядає красиво», а «чи виконані критерії, чи нема зайвого».

Ключова різниця — **хто відповідає за якість і на чому ця відповідальність тримається**. У vibe coding перевірка — «начебто працює». В agentic development перевірки вбудовані в процес (spec, тести, план, рев'ю), тому результат придатний для продакшну. Типова помилка L3 — vibe coding у продакшн-коді: прийняти згенероване без розуміння, що воно робить. Правило просте: **ти не можеш відповідати за код, який не розумієш** — а за продакшн-код відповідаєш ти, не модель.

**Підсумок-якір:** Vibe coding — «описав → прийняв → начебто працює», ок для прототипів і разових скриптів; agentic development — spec, затверджений план, тести, рев'ю проти spec: та сама модель, інший процес. Не можна відповідати за код, якого не розумієш.

**➕ Пов'язане: коли vibe coding доречний, а коли небезпечний? 🟢**
Критерій випливає з головної різниці — ціни помилки. Доречний: прототип для демо, PoC («чи витягне ця бібліотека наш кейс?»), разовий скрипт міграції даних, який перевіриш очима, дослідження незнайомого API. Небезпечний: усе, що піде в прод і торкається грошей, безпеки, персональних даних чи складної бізнес-логіки. Підступність у тому, що vibe-код **виглядає** завершеним — компілюється, happy path працює — а діри сидять у краєвих випадках і обробці помилок, куди ніхто не дивився.

**➕ Пов'язане: що таке "human in the loop"? 🟢**
Це формалізація «ти затверджуєш план» і «ти рев'юїш diff» з agentic-циклу вище: людина — обов'язкова контрольна точка перед незворотними діями (merge, деплой, зміна даних у проді, розсилка користувачам). Агент може автономно ітерувати всередині пісочниці (писати код, ганяти тести), але перетин межі «це вплине на реальних людей/дані» вимагає людського «так». Це і про якість, і про відповідальність: юридично й етично рішення ухвалює людина.

## Q4. Що таке spec-driven development (AI-assisted) — елементи й послідовність? 🔵

**🎯 Відповідь на іспиті:** Spec-driven development — це підхід, коли перед кодуванням пишеться специфікація — щільний документ з API-контрактом, бізнес-правилами і критеріями прийняття, — і саме вона стає джерелом правди і для AI-агента, і для рев'ю. Потрібен він, бо короткий промпт лишає більшість рішень недоговореними, а агент заповнює пропуски правдоподібними здогадками замість твоїх правил. Послідовність: spec → план, який рев'юїться до коду (найдешевша точка виправити напрямок) → генерація за планом → verify проти критеріїв прийняття → ітерація, причому правиться spec, а не код вручну повз нього. Типова помилка — spec, який не оновлюється і розповзається з кодом: тому специфікація живе в репозиторії і змінюється тим самим PR, що й поведінка.

Чому «просто попросити AI зробити фічу» дає посередній результат? Бо в короткому запиті на кшталт «додай запис на курс» **не сказано** 90% важливого: що робити, коли місць нема? чи можна записатися двічі? який формат помилок? Агент не лишає пропуски порожніми — він **заповнює їх здогадками**, правдоподібними, але не твоїми. Spec-driven development прибирає здогадки: перед кодуванням пишеться **специфікація** — документ, що фіксує «що і як має працювати», — і саме вона стає джерелом правди і для агента, і для рев'ю.

Як виглядає spec на реальну фічу (не «ТЗ на 20 сторінок», а щільна сторінка):

```md
# Фіча: запис студента на курс
## Навіщо: студент записується сам, без адміністратора; місця обмежені
## API: POST /courses/:id/enroll (auth required)
   201 { enrollmentId } | 409 "course_full" | 409 "already_enrolled" | 404
## Правила
- Не можна записатися двічі на той самий курс
- Якщо місця скінчились — enrollment зі статусом "waitlisted", не помилка
- Лічильник місць не має "розїжджатися" при одночасних запитах (транзакція/лок)
## Критерії прийняття
- [ ] Повторний enroll того ж студента → 409, у БД один запис
- [ ] 2 паралельні запити на останнє місце → рівно один enrolled, другий waitlisted
- [ ] Юніт-тести на всі правила, e2e на happy path
```

Зверни увагу, що робить кожен блок: «Навіщо» дає агенту контекст для мікрорішень; API-контракт прибирає свободу вигадувати формат; «Правила» — це якраз ті краєві випадки, які агент інакше вирішив би сам (і не так); «Критерії прийняття» перетворюють рев'ю з «подобається/ні» на чекліст.

Послідовність spec-driven процесу:
1. **Spec** — пишеш документ як вище. Найцінніший крок: половину багів ти ловиш ще тут, бо формулювання правил змушує їх продумати.
2. **Plan / design** — агент (або ти) розбиває на кроки: міграція БД → сервіс → контролер → тести. План рев'юїться до коду — це найдешевша точка виправлення напрямку.
3. **Generate** — агент реалізує за планом, маючи spec у контексті.
4. **Verify** — тести проходять + ручна звірка коду з критеріями прийняття.
5. **Iterate** — якщо результат не той, **правиш spec, а не код**: «а, я не вказав, що waitlist має чергу» → доповнюєш правило → агент допрацьовує. Латання коду вручну повз spec повертає тебе до vibe coding — spec перестає бути правдою.

Найчастіша помилка — spec пишеться, але не оновлюється: після третьої ітерації код і документ розповзаються, і spec стає шумом. Правило: spec живе поруч із кодом (у репозиторії) і змінюється тим самим PR, що й поведінка.

**Підсумок-якір:** Spec → план → генерація → verify → ітерація правкою spec, не коду. Специфікація — джерело правди: агент не заповнює краєві випадки здогадками, а рев'ю йде проти критеріїв прийняття; spec живе в репозиторії і міняється тим самим PR.

**➕ Пов'язане: чим spec-driven відрізняється від TDD? 🔵**
У нашому spec критерії прийняття — це фактично список майбутніх тестів, і це не збіг. TDD (Test-Driven Development) — окремий випадок spec-driven, де специфікація одразу **виконувана**: спершу пишеш тест (він падає), потім мінімальний код, щоб пройшов, потім рефакториш. Spec-driven ширший: специфікація може бути markdown-документом, OpenAPI-контрактом (машиночитаним описом REST API: шляхи, параметри, формати відповідей), Gherkin-сценаріями (Q13) — а вже з неї народжуються тести. Спільна суть обох: зафіксувати «що має бути» до того, як писати «як».

**➕ Пов'язане: чому чіткий spec важливий саме для AI-агентів? 🟢**
Повертаємось до «агент заповнює пропуски здогадками». Людина-колега при неясності перепитає («а що при повному курсі?»); агент за замовчуванням — ні, він обере правдоподібний варіант і поїде далі, а ти дізнаєшся про це на рев'ю або в проді. Чіткий spec звужує простір інтерпретацій до одного варіанта. Бонус: перевіряти код проти явних критеріїв незрівнянно легше, ніж проти «того, що я мав на увазі».

**➕ Пов'язане: що таке acceptance criteria і як їх формулювати? 🔵**
Це той блок «Критерії прийняття» зі spec вище — умови, за яких задача вважається виконаною. Добрий критерій **перевірюваний**: не «запис працює коректно» (як перевірити?), а «повторний enroll → 409, у БД один запис» (перевіряється тестом за хвилину). Формула: конкретна дія + конкретний очікуваний результат. Лакмус: якщо з критерію не можна написати тест — це не критерій, а побажання.

## Q5. Як робити reuse AI-промптів для розробки (skills, workflows, commands)? 🔵

**🎯 Відповідь на іспиті:** Reuse AI-промптів — це винесення інструкцій, які команда повторює агенту знову й знову, у версіоновані артефакти в репозиторії: skills — пакети інструкцій і знань під тип задачі («як у нас пишуться контролери»), commands — іменовані шаблони частих дій на кшталт `/review` чи `/write-tests`, workflows — зашиті послідовності кроків цілого процесу (spec → план → код → тести → PR). Сенс той самий, що й DRY у коді: кожен рядок такого файлу закриває один ступінь свободи, який агент інакше заповнив би здогадкою, і якість виводу перестає залежати від того, хто сьогодні пише промпт. Файл рев'юїться і версіонується як код, тож нова типова помилка агента виправляється одним рядком для всіх, а не поясненнями в кожному чаті. Межа — YAGNI: кодифікувати варто лише промпт, який реально повторився і стабілізувався, а не «про запас».

Проблема виникає тижні через два роботи команди з AI-асистентом: кожен розробник щоразу заново пояснює агенту одне й те саме — «пиши тести на Vitest», «у нас NestJS, DTO (об'єкти-перенощики даних запиту) валідуємо через class-validator», «не чіпай міграції». Той самий текст набирається десятки разів, кожен формулює по-своєму → агент видає код різного стилю і якості залежно від того, **хто** сьогодні поставив задачу. Це те саме дублювання знань, що й у коді, — і лікується так само: **винести повторюване в один версіонований артефакт**.

Три форми таких артефактів (назви різняться між інструментами, суть спільна):
- **Skills** — пакет інструкцій і знань під тип задачі: «як у нас пишуться NestJS-контролери», «як ми оформлюємо міграції». Агент підтягує skill, коли задача відповідного типу.
- **Commands** — іменовані шаблони частих дій: `/review`, `/write-tests`, `/new-endpoint`. Викликаються явно.
- **Workflows** — послідовності кроків для процесу цілком: «нова фіча = spec → план → код → тести → PR» (по суті, зашитий у файл процес із Q4).

```md
<!-- .agent/commands/write-tests.md — розберімо, ЧОМУ кожен рядок тут -->
# Ціль: покрити вказаний модуль юніт-тестами
- Використовуй Vitest та AAA-структуру      ← інакше агент обере фреймворк/стиль сам
- Мокай зовнішні залежності, не мокай чисту логіку   ← найчастіша помилка AI-тестів
- Мінімум: happy path + межові випадки + помилки     ← інакше буде тільки happy path
```

Чому цей маленький файл працює: кожен рядок **закриває один ступінь свободи**, який агент інакше заповнив би здогадкою (Q4 — та сама логіка, що зі spec). Без першого рядка агент може згенерувати Jest-синтаксис у Vitest-проєкті; без другого — замокати саму функцію, яку тестує; без третього — видати три happy-path тести і зупинитись. Файл лежить у репозиторії → він **рев'юїться, версіонується і спільний для всіх**: якість тестів більше не залежить від того, хто сьогодні пише промпт. З'явилась нова типова помилка агента — додаєш рядок у файл один раз, а не пояснюєш кожному в кожному чаті.

Межа підходу: не намагайся закодифікувати все й одразу. Skill, написаний «про запас» до того, як промпт справді повторювався, — це YAGNI (Q6) у світі промптів: він застаріє швидше, ніж знадобиться.

**Підсумок-якір:** Повторюваний промпт — у версіонований файл у репозиторії: skill — знання під тип задачі, command — іменований шаблон дії, workflow — процес цілком. Кодифікуй після реального повторення, не «про запас».

**➕ Пов'язане: що таке контекст-інженерія? 🔵**
Команди й skills — це заготовлені шматки контексту, а контекст-інженерія — дисципліна керування контекстом загалом: **що агент бачить у момент задачі**. Це і додавання потрібного (релевантні файли, правила проєкту, приклади «як у нас прийнято»), і — не менш важливо — **прибирання зайвого**: нерелевантні файли розмивають фокус, і агент починає «мавпувати» патерни не з того місця кодбази. Практичне правило: якість відповіді частіше визначається контекстом, ніж формулюванням промпту. «Зроби як у `CoursesService`» з відкритим файлом працює краще за абзац абстрактних вимог.

**➕ Пов'язане: навіщо файли правил проєкту (напр. AGENTS.md/rules)? 🟢**
Це «нульовий рівень» reuse — контекст, який агент отримує **завжди автоматично**, без виклику: стек проєкту, конвенції, заборони («не редагуй згенеровані файли», «commit-повідомлення англійською»). Різниця зі skills — у тригері: skill підтягується під тип задачі, файл правил присутній у кожній розмові. Тому тримай його коротким: тільки те, що справді стосується кожної задачі, — роздутий файл правил з'їдає контекст і увагу моделі.

**➕ Пов'язане: коли НЕ варто виносити промпт у skill/command? 🟢**
За тим самим «rule of three», що й для коду (Q6): виноси, коли інструкція реально повторилася кілька разів і стабілізувалася. Разовий промпт, оформлений у skill, — мертвий вантаж; інструкція, яку ти ще щоразу переформульовуєш, у файлі передчасно «застигне» в невдалій формі. Спершу промпт доводиться до робочого стану в живих сесіях — потім кодифікується.

## Q6. Що таке DRY, KISS, YAGNI і як змусити AI-асистента їх дотримуватись? 🔵

**🎯 Відповідь на іспиті:** Це три принципи проти невиправданої складності: DRY (Don't Repeat Yourself) — кожне знання живе в одному місці, щоб зміна правила не губилася в дублях; KISS (Keep It Simple, Stupid) — з робочих рішень обирається найпростіше; YAGNI (You Aren't Gonna Need It) — не будувати абстракції «на майбутнє», якого ще нема. AI code assistants системно переускладнюють — додають шари, конфігурованість і патерни там, де вистачає функції, — тому «simplicity is the key»: я закладаю ці принципи у правила проєкту, а на рев'ю виводу розпізнаю порушення — інтерфейс з єдиною реалізацією, скопійований блок замість виклику, три рівні непрямості для простої дії. І активно прошу спрощення («зроби те саме без класу») — асистенти добре спрощують, але зазвичай лише на явний запит.

Три принципи, які захищають від одного й того самого ворога — **невиправданої складності**. Складність — головна причина, чому проєкти сповільнюються: кожен зайвий шар, дубль чи «гнучкість про запас» — це те, що наступний розробник (або ти через пів року) муситиме прочитати, зрозуміти і не зламати.

- **DRY** (Don't Repeat Yourself) — кожне **знання** живе в одному місці. Якщо правило «студент не може записатися двічі» продубльоване в трьох сервісах, то при зміні правила хтось забуде третє місце — і поведінка розійдеться. Важливо: DRY — про знання/логіку, а не про текстову схожість коду (див. похідне про перестарання).
- **KISS** (Keep It Simple, Stupid) — з робочих рішень обирай найпростіше. Простота — не «примітивність», а мінімум речей, які треба тримати в голові, щоб зрозуміти код.
- **YAGNI** (You Aren't Gonna Need It) — не буду́й «на майбутнє», якого ще нема. Абстракція (узагальнений прошарок — інтерфейс, базовий клас, конфігурованість, — що ховає конкретику заради «майбутніх варіантів») під «а раптом додамо SMS» коштує зараз (складніший код) і, найімовірніше, не окупиться: майбутнє прийде іншим, ніж ти вгадав, і абстракцію доведеться ламати.

```ts
// ❌ порушення YAGNI+KISS: "гнучка" ієрархія там, де потрібна одна функція
interface NotificationChannelStrategyFactory { create(cfg: ChannelConfig): ChannelStrategy }
// ...100 рядків реєстрації стратегій — а канал у проєкті ОДИН і другого в планах нема

// ✅ KISS: рівно те, що потрібно сьогодні
async function sendEnrollmentEmail(to: string, courseTitle: string) { /* ... */ }
// коли (якщо!) з'явиться другий канал — ОТОДІ і з'явиться абстракція,
// причому по формі реальної потреби, а не вгаданої
```

Тепер про AI-асистентів, і чому це питання стоїть саме так. Моделі навчалися на морі enterprise-коду і «зразкових» туторіалів, тому мають системний нахил **переускладнювати**: додавати шари абстракцій, конфігурованість, обробку неможливих випадків, патерни там, де вистачає функції. Що з цим робити практично:
1. **Закласти принципи в правила проєкту** (Q5): «обирай найпростіше рішення; не додавай абстракцій без другого використання; не обробляй випадків, яких нема в задачі».
2. **Розпізнавати порушення на рев'ю виводу**: інтерфейс з єдиною реалізацією (YAGNI), скопійований блок замість виклику (DRY), три рівні непрямості для простої дії (KISS).
3. **Активно просити спрощення**: «зроби те саме без класу», «прибери конфігурованість» — асистенти добре спрощують, але зазвичай лише на явний запит. Спрощення згенерованого коду — нормальна, очікувана частина роботи з AI, а не ознака поганої моделі.

**Підсумок-якір:** DRY — одне знання в одному місці; KISS — найпростіше з робочого; YAGNI — нічого «про запас». AI-асистенти системно переускладнюють, тому принципи закладаються в правила проєкту, а зайві абстракції прибираються на рев'ю — «simplicity is the key».

**➕ Пов'язане: чи можна перестаратися з DRY? 🔵**
Так, і це прямо випливає з формулювання «DRY — про знання, а не про схожий текст». Два шматки коду можуть **виглядати** однаково, але представляти різні знання: валідація email при реєстрації і при запрошенні викладача сьогодні збігаються, а завтра еволюціонують по-різному — і передчасна спільна функція обросте прапорцями `if (isInvite)`. Це «випадкове дублювання», і його усунення створює зв'язність (coupling — ступінь, до якого зміна в одному місці коду змушує міняти інше) гіршу за дубль: тепер реєстрацію і запрошення не можна міняти окремо. Практичне правило — **rule of three**: терпи повторення до третього разу; на третьому вже видно, чи це справді одне знання і якою має бути абстракція.

**➕ Пов'язане: що таке over-engineering? 🟢**
Збірна назва для порушень KISS/YAGNI: рішення, складніше за задачу, — зайві шари, патерни «на виріст», конфігурованість, якою ніхто не скористається. Ціна подвійна: час на побудову зараз і податок на розуміння при кожній наступній зміні. Класична причина — проєктування під уявне майбутнє. Протиотрута — чесне питання «це потрібно **сьогодні**?» і звичка дивитись на AI-вивід саме через цю призму, бо асистент за замовчуванням тягне саме в бік over-engineering.

## Q7. Що таке SDLC і які його фази (включно з support & maintenance)? 🔵

**🎯 Відповідь на іспиті:** SDLC (Software Development Life Cycle) — це повний цикл життя програмного продукту від ідеї до виводу з експлуатації: requirements → design → implementation → testing → deployment → support & maintenance. Суть у тому, що кожна фаза відповідає на питання, без якого наступна робиться наосліп: без вимог команда ідеально будує не те, без дизайну архітектура «виростає сама» з випадкових рішень, без тестування тестувальниками стають користувачі в проді. Support & maintenance — найдовша і найдорожча фаза: продукт живе роками після написання, тому оптимізувати треба вартість супроводу — читабельність, тести, документацію рішень. І це логічні етапи, а не обов'язково календарні: Waterfall проходить їх один раз для всього продукту, Agile — багато разів маленькими циклами.

SDLC (Software Development Life Cycle) — повний цикл життя програмного продукту від ідеї до виводу з експлуатації. Сенс моделі не в тому, щоб вивчити список фаз, а в розумінні: **кожна фаза відповідає на питання, без якого наступна робиться наосліп**. Найкраще це видно через «що станеться, якщо фазу пропустити»:

1. **Requirements** — збір і аналіз вимог: яку проблему розв'язуємо, для кого, що є успіхом. *Пропустиш* — команда ідеально збудує не те: класика — пів року розробки «особистого кабінету», яким користувачі не користуються, бо їм була потрібна одна кнопка експорту.
2. **Design** — архітектура і технічний дизайн: структура системи, моделі даних, контракти між частинами, ключові компроміси. *Пропустиш* — архітектура «виросте сама» з випадкових рішень; перші три фічі підуть швидко, а на десятій виявиться, що дані лежать так, що нову фічу дешевше переписати з нуля, ніж вбудувати.
3. **Implementation** — власне розробка. Зверни увагу: у здоровому процесі це вже **третя** фаза, а не перша.
4. **Testing** — перевірка, що збудоване відповідає вимогам. *Пропустиш* — тестувальниками стають користувачі в проді, а вартість багу, знайденого в проді, на порядок вища за знайдений до релізу (додай сюди пошкоджені дані і втрачену довіру).
5. **Deployment** — вихід у продакшн: як код потрапляє до користувачів, як відкотитися, якщо щось пішло не так. *Пропустиш планування цієї фази* — реліз перетворюється на ручний героїзм о другій ночі без плану відкату.
6. **Support & Maintenance** — виправлення багів, оновлення залежностей, адаптація до нових вимог, моніторинг. Це **найдовша і найдорожча** фаза: типовий продукт живе роками після написання, і більша частина сукупної вартості припадає саме сюди.

Ключовий висновок для L3 випливає з пункту 6: код пишеться один раз, а читається і змінюється роками, тому оптимізувати треба **вартість супроводу** — читабельність, тести, документація рішень. «Швидко написав, але ніхто не може змінити» — це дорогий код, хоч би як швидко він з'явився.

Фази — це логічні етапи, а не обов'язково календарні: у Waterfall вони проходяться один раз послідовно, в Agile — багато разів маленькими циклами (див. похідне).

**Підсумок-якір:** SDLC: requirements → design → implementation → testing → deployment → support & maintenance; кожна фаза відповідає на питання, без якого наступна робиться наосліп. Maintenance — найдовша і найдорожча фаза, тому оптимізуй вартість супроводу.

**➕ Пов'язане: чим Waterfall відрізняється від Agile SDLC? 🔵**
Обидва проходять ті самі шість фаз — різниця в **масштабі циклу**. Waterfall проходить їх один раз для всього продукту: всі вимоги → весь дизайн → вся розробка → все тестування; помилка у вимогах виявляється через місяці, коли її виправлення найдорожче. Agile проходить ті самі фази за один-два тижні для маленького шматка продукту: помилка у вимогах спливає на першому ж демо, поки її дешево виправити. Тобто Agile — не «скасування фаз SDLC», а зменшення кроку ітерації, щоб петля зворотного зв'язку була короткою.

**➕ Пов'язане: що таке technical debt у контексті maintenance? 🟢**
Оскільки maintenance — найдовша фаза, саме в ній «спливають відсотки» за швидкі рішення: пропущені тести, «тимчасовий» костиль, який став постійним, застарілі залежності. Це і є технічний борг — метафора точна: взяв у борг час на старті (зрізав кут — поставив фічу швидше), платиш відсотки щоразу, коли зміна в цьому місці йде повільніше і ризикованіше. Керування боргом — свідома частина maintenance: виділяти регулярний час на рефакторинг і оновлення, інакше відсотки поступово з'їдають усю швидкість команди.

## Q8. Як якісно покривати функціонал юніт-тестами за допомогою AI-асистентів? 🟣

**🎯 Відповідь на іспиті:** Головний принцип: AI чудово генерує форму тестів, але за зміст — які кейси важливі і що саме перевіряти — відповідаю я. Тому список кейсів (happy path, межові випадки, помилки) складаю сам або беру з критеріїв прийняття і прошу тести саме на них, а не абстрактне «напиши тести». Далі даю правильний контекст: функцію з типами, наявний тест-файл як зразок стилю і правила — AAA-структура, мокати тільки зовнішні залежності (мережа, БД, час), перевіряти поведінку, а не деталі реалізації. Кожен згенерований тест рев'ю питанням «що він ловить»: червоні прапорці — `expect(mock).toHaveBeenCalled()` як єдина перевірка, тест-дубль реалізації, очікування, підігнане під фактичний вивід замість правильного. Наприкінці перевіряю тести на «мертвість»: навмисно ламаю код і дивлюсь, чи впали — тест, що лишився зеленим на зламаному коді, нічого не охороняє. За 100% покриттям не женуся: coverage міряє виконані рядки, а не перевірену поведінку.

Почнімо з пастки, у яку тут потрапляють найчастіше. Просиш асистента «напиши тести для `EnrollmentService`», отримуєш файл на 200 рядків, покриття 95%, усе зелене — виглядає чудово. Але придивись (мок — підставна заміна справжньої залежності на кшталт БД чи HTTP-клієнта: повертає заздалегідь задані відповіді й записує, як її викликали): половина тестів перевіряє лише, що «мок був викликаний», не перевіряючи результат; краєвих випадків нема; а якщо навмисно зламати бізнес-логіку — половина тестів **лишиться зеленою**. AI геніально генерує *форму* тестів, але за *зміст* — які випадки важливі і що саме перевіряти — відповідаєш ти. Уся методика зводиться до цього розподілу ролей.

Робочий процес із чотирьох кроків:

**Крок 1: список кейсів складаєш ти, а не AI.** Асистент бачить код, але не бачить бізнесу — він не знає, що «запис на заповнений курс має класти в waitlist», якщо цього нема в коді чи spec. Тому спершу сам (30 секунд подумати або взяти з критеріїв прийняття, Q4): happy path, межові випадки, помилки. Тоді запит виглядає не «напиши тести», а «напиши тести на ці шість кейсів».

**Крок 2: дай правильний контекст** (Q5): сама функція + її типи + один наявний тест-файл як зразок стилю + команда на кшталт `write-tests` із правилами (Vitest, AAA, політика моків).

**Крок 3: рев'юй кожен тест питанням «що він ловить?»** Добрий тест впаде, якщо зламати конкретне правило. Вимоги до згенерованого:
- **AAA-структура** (Arrange-Act-Assert: підготував → викликав → перевірив результат) і назви, що читаються як вимога («не йде в мінус при перевитраті»).
- Мокати **зовнішні** залежності (мережа, БД, час) — вони повільні й недетерміновані (дають різний результат від запуску до запуску, і тест «блимає»); **не** мокати чисту логіку — інакше тестуєш мок, а не код.
- Перевіряти **поведінку** (результат, зміну стану), а не деталі реалізації (які внутрішні методи викликались). Тест, прив'язаний до реалізації, ламається від будь-якого рефакторингу — і його починають правити «щоб пройшов», що вбиває його цінність.

**Крок 4: перевір тести на «мертвість»** — навмисно зламай код (інвертуй умову, поверни константу) і подивись, чи впали тести. Тест, що лишився зеленим на зламаному коді, нічого не охороняє — викидай або переписуй. Це ручна версія mutation testing (див. похідне).

```ts
// осмислений набір: КОЖЕН тест ловить конкретне правило бізнес-логіки
describe("remainingBudget", () => {
  it("рахує залишок", () =>
    expect(remainingBudget(10_000, 3_000)).toBe(7_000));   // happy path
  it("не йде в мінус при перевитраті", () =>
    expect(remainingBudget(10_000, 12_000)).toBe(0));      // межовий: бізнес-правило "не < 0"
  it("нульовий бюджет", () =>
    expect(remainingBudget(0, 0)).toBe(0));                // межовий: вироджений вхід
  it("відхиляє від'ємні витрати", () =>
    expect(() => remainingBudget(10_000, -5)).toThrow());  // помилковий вхід — теж контракт
});
```

Червоні прапорці у AI-згенерованих тестах (шукай їх на рев'ю свідомо):
- `expect(mock).toHaveBeenCalled()` як **єдина** перевірка — факт виклику без перевірки результату;
- тест, що дублює реалізацію («порахували так само і порівняли») — впаде разом із багом у логіці, тобто ніколи;
- очікування, «підігнане» під поточний фактичний вивід, а не під правильний (AI охоче так робить, коли тест падає: міняє тест, а не сигналізує про баг!);
- тести на тривіальності (геттери, виклик бібліотечної функції) заради цифри покриття.

Останній пункт про підгонку — критичний: якщо агент сам «чинить» тести до зеленого, він може непомітно закріпити баг. Правило: тест падає → спершу питання «код неправильний чи тест?», і відповідає на нього людина.

**Підсумок-якір:** AI пише тести швидко, але осмисленість — моя робота: я задаю список кейсів (happy path, межі, помилки), даю контекст і правила (AAA, мокати тільки зовнішнє, перевіряти поведінку), рев'юю кожен тест питанням «що він ловить» і перевіряю, що тести падають на зламаному коді.

**➕ Пов'язане: чому не варто гнатися за 100% покриттям? 🔵**
Той файл із пастки на початку — 95% покриття і половина мертвих тестів — і є відповідь. Coverage вимірює, які рядки були **виконані** під час тестів, а не чи були вони **перевірені**: можна виконати весь код без жодного змістовного `expect` і мати 100%. Коли покриття стає ціллю (а з AI нагенерувати його тривіально), з'являються тести-філери, які створюють хибне відчуття захищеності — це гірше, ніж чесна діра. Здоровий підхід: критична логіка (гроші, доступи, дані) покрита якісно, а метрика покриття — індикатор для пошуку білих плям, не KPI.

**➕ Пов'язане: що таке тест-піраміда? 🟢**
Це відповідь на питання «яких тестів скільки писати». Основа — багато швидких **unit**-тестів (мілісекунди, точна локалізація поломки); середина — менше **integration** (перевіряють стики: сервіс + реальна БД); вершина — мало **e2e** (весь застосунок очима користувача: повільні, крихкі, зате перевіряють усе разом). Пропорція дає швидкий зворотний зв'язок без крихкості. Антипатерн «морозиво» (навпаки: купа e2e, мало unit) означає: прогін годинами, падіння «десь у системі», і команда перестає довіряти тестам.

**➕ Пов'язане: що таке mutation testing? 🔵**
Автоматизація кроку 4 («зламай код — подивись, чи впали тести»). Інструмент (для JS/TS — Stryker) сам вносить у код дрібні мутації — `>` на `>=`, `+` на `-`, видалення виклику — і проганяє тести на кожній. Мутант «вижив» (тести зелені на зламаному коді) — значить, ця ділянка реально не перевіряється, хоч coverage і показує 100%. Це найчесніша метрика якості тестів, але дорога за часом прогону, тому зазвичай її ганяють не на кожен коміт, а періодично на критичних модулях.

## Q9. Що таке Agile (Scrum, Kanban) і що таке не-Agile методологія? 🔵

**🎯 Відповідь на іспиті:** Agile — це ставка на те, що вимоги неминуче зміняться, тому процес будується навколо короткої петлі зворотного зв'язку: працюючий продукт показується часто, і курс коригується, поки це дешево. Scrum реалізує це фіксованими спринтами на 1–2 тижні з незмінною ціллю всередині спринту, церемоніями і ролями Product Owner, Scrum Master і команди. Kanban — без спринтів: безперервний потік задач через дошку з WIP-лімітами, які змушують дороблювати почате перед новим («stop starting, start finishing»). Приклад не-Agile методології — Waterfall: повний план наперед і фази один раз; це не «застарілий Agile», а інструмент для стабільних відомих вимог — сертифіковане медичне ПЗ, контракт із зафіксованим обсягом, залізо. Типова помилка — карго-культ: стендапи і спринти є, а короткої петлі фідбеку немає — це Waterfall у костюмі Scrum.

Історична проблема, з якої все виросло: команда вісім місяців сумлінно будує систему за детальним ТЗ, показує замовнику — а той каже «це не те, що нам треба». Не тому, що хтось поганий: за вісім місяців змінився ринок, замовник побачив живий продукт і зрозумів, чого хотів насправді, а частина вимог від початку була здогадками. Вартість цього відкриття — вісім місяців роботи. **Agile** — це ставка на те, що вимоги **неминуче** зміняться, тому процес будується навколо короткої петлі зворотного зв'язку: працюючий продукт показується часто, курс коригується дешево.

Два найпоширеніші фреймворки реалізують цю ідею по-різному:

- **Scrum** — робота **спринтами** (фіксовані ітерації 1–2 тижні). На вході спринту команда бере зі списку задач (беклогу) стільки, скільки реально зробить, і ціль не змінюється до кінця спринту — це захищає команду від смикання «терміново вкиньте ще це». В кінці — демонстрація працюючого результату і ретроспектива процесу (церемонії — Q10). Ролі: Product Owner (відповідає за «що робити» і пріоритети беклогу), Scrum Master (відповідає за процес і усунення перешкод), команда розробки.
- **Kanban** — без спринтів: **безперервний потік** задач через дошку-конвеєр (To Do → In Progress → Review → Done) з головним механізмом — **WIP-лімітами** (Work In Progress: скільки задач може висіти в колонці одночасно). Ліміт — це не бюрократія, а фізика потоку: 10 одночасно «в роботі» задач означає, що жодна не рухається; ліміт 3 змушує **дороблювати** почате перед початком нового. Девіз: «stop starting, start finishing».

**Не-Agile — це не «погано», це інша ставка.** Waterfall (Q7): повний план наперед, фази один раз, зміни дорогі й контрольовані формальним процесом. Він адекватний там, де вимоги справді стабільні й відомі, а ітеративність неможлива чи безглузда: сертифіковане медичне/авіаційне ПЗ, контракт із зафіксованим обсягом, залізо (не «випустиш патч» для чипа). Помилка — вважати Waterfall застарілим Agile'ом: це інструмент під інший клас задач.

Типова помилка на практиці — «карго-культ Agile»: стендапи і спринти є, а працюючий продукт показується раз на пів року і фідбек нікуди не вбудовується. Церемонії без короткої петлі зворотного зв'язку — це Waterfall у костюмі Scrum.

**Підсумок-якір:** Agile — коротка петля зворотного зв'язку, бо вимоги зміняться: Scrum — спринти з незмінною ціллю, Kanban — безперервний потік із WIP-лімітами. Не-Agile — Waterfall: один прохід за повним планом, доречний при стабільних вимогах.

**➕ Пов'язане: коли Kanban кращий за Scrum? 🔵**
Дивись на природу потоку задач. Спринт має сенс, коли роботу можна спланувати на два тижні наперед — продуктова розробка фічами. Якщо ж робота **непередбачувана і вхідна** — підтримка, багфікси, DevOps, потік дрібних запитів — планування спринту стає фікцією: половину плану зносить терміновими вхідними. Тут Kanban чесніший: пріоритезуєш чергу безперервно, WIP-ліміти тримають потік, метрика — час проходження задачі (lead time), а не velocity. Поширений гібрид: продуктова команда — Scrum, чергова/платформна — Kanban.

**➕ Пов'язане: що таке velocity і чи це метрика продуктивності? 🟢**
Velocity — скільки story points (Q19) команда фактично закриває за спринт, усереднено за кілька останніх. Призначення одне — **прогноз**: якщо velocity ≈ 30, брати в спринт 50 — самообман. Використовувати її як метрику продуктивності («команда А краща за Б», «підніміть velocity на 20%») не можна з простої причини: поінти — внутрішня відносна шкала кожної команди, і щойно цифра стає ціллю, вона накручується інфляцією оцінок (та сама задача «дорожчає» з 3 до 5 поінтів) — прогноз ламається, а продуктивність не змінюється. Це закон Гудгарта: метрика, що стала ціллю, перестає вимірювати те, для чого існувала.

## Q10. Які є Scrum-церемонії, що вони вирішують і скільки тривають? 🔵

**🎯 Відповідь на іспиті:** У Scrum чотири основні церемонії, і кожна закриває конкретний збій координації (тривалості — орієнтир для двотижневого спринту). Sprint Planning (~2–4 години на старті) — команда бере з беклогу задачі під ціль спринту; без нього спринт — випадковий набір «хто що вхопив». Daily Standup (15 хвилин щодня) — синхронізація «що зробив / що планую / що блокує»: це координація рівних, а не звіт менеджеру. Sprint Review (~1–2 години наприкінці) — демонстрація працюючого продукту стейкхолдерам і збір фідбеку, а Sprint Retrospective (~1–1.5 години після review) — розмова команди про процес, а не продукт: що заважало і що змінимо. Щоб не плутати: standup — синхронізація, review — про продукт із зовнішніми, retro — про процес тільки командою.

Церемонії Scrum — це не «мітинги, бо так заведено»: кожна закриває конкретний збій координації, який інакше стається сам собою. Найкращий спосіб їх запам'ятати — розуміти, **що ламається без кожної** (тривалості — орієнтир для 2-тижневого спринту):

- **Sprint Planning** (~2–4 год, на старті спринту) — команда бере з беклогу задачі під ціль спринту і промовляє, як їх робитиме. Без нього спринт — випадковий набір «хто що вхопив»: пріоритети в головах різні, залежні задачі беруться у неправильному порядку, наприкінці нічого не складається в ціле.
- **Daily Standup** (15 хв щодня) — синхронізація: що зробив / що планую / що блокує. Це **не звіт менеджеру**, а координація рівних: цінність — у фразах «я чекаю на твій ендпоінт» і «не чіпай enrollment, я його переписую». Без нього два розробники тиждень мовчки блокують один одного або паралельно правлять одне місце. Найчастіша деградація — перетворення на статус-звіт на 40 хвилин; лікується правилом «деталі — після стендапу, окремо з причетними».
- **Sprint Review** (~1–2 год, наприкінці) — демонстрація **працюючого продукту** стейкхолдерам (усім, кого стосується результат: замовник, менеджмент, суміжні команди) і збір фідбеку. Це та сама коротка петля зворотного зв'язку, заради якої існує Agile (Q9): без review фідбек приходить раз на квартал — і місяці роботи можуть їхати не туди.
- **Sprint Retrospective** (~1–1.5 год, після review) — команда обговорює не продукт, а **процес**: що працювало, що заважало, що змінимо (1–2 конкретні дії на наступний спринт). Без ретро ті самі проблеми — повільний CI, нечіткі задачі, вузьке місце на рев'ю — повторюються місяцями, бо їх нема де назвати вголос. Головне правило: розбираємо процес, не людей — інакше всі замовкають, і ретро вмирає.

Швидкий спосіб не плутати на співбесіді: **standup — синхронізація, review — про продукт (із зовнішніми), retro — про процес (тільки команда)**.

**Підсумок-якір:** Planning (2–4 год) — що беремо у спринт; standup (15 хв) — синхронізація, не звіт; review (1–2 год) — про продукт зі стейкхолдерами; retro (~1 год) — про процес тільки командою.

**➕ Пов'язане: що таке backlog refinement (grooming)? 🔵**
П'ята активність, яку часто забувають, хоч без неї розвалюється Planning: регулярна сесія (~1 год на тиждень), де команда «доводить до готовності» майбутні задачі беклогу — уточнює вимоги, розбиває завеликі, ставить оцінки (Q19), з'ясовує питання до Product Owner заздалегідь. Якщо refinement нема, все це відбувається прямо на Planning — і планування розпухає в півдня болючих з'ясувань «а що тут взагалі мається на увазі». Практичний критерій готовності задачі до спринту: зрозуміла, оцінена, має критерії прийняття (Q4).

**➕ Пов'язане: що таке Definition of Done? 🟢**
Спільний чекліст, що означає «задача завершена»: наприклад, код + тести + пройдене рев'ю + задеплоєно на staging (передпродове середовище — копія проду для перевірок) + оновлена документація. Без нього «готово» у кожного своє — розробник каже «готово» (код написаний), а тестів нема і в проді воно не було; на review це щоразу вибухає суперечкою «майже готово». DoD прибирає розмову «а я думав…» і тримає єдину планку якості: задача або відповідає чеклісту, або не Done — без «на 90% готово».

## Q11. Що таке event-driven architecture (переваги, недоліки, компоненти)? 🟢

**🎯 Відповідь на іспиті:** Event-driven architecture — це підхід, коли компоненти спілкуються через події, а не прямі виклики: producer публікує факт («документ завантажено») у чергу (queue/broker — RabbitMQ, Kafka, BullMQ), а consumer — функції-обробники — розгрібають її у своєму темпі; тригерами подій бувають user-driven (дія користувача), time-driven (розклад, cron) і зовнішні системи (webhook), причому все, де затримка прийнятна, вигідніше виносити з миттєвої реакції у пакетну обробку за розкладом. Переваги — слабка зв'язність, незалежне масштабування воркерів і згладжування сплесків; ціна — асинхронність в UX, важчий дебаг (потрібні наскрізні logging і tracing за спільним id, бо стектрейс через чергу не перестрибує) і retry-стратегія з ідемпотентними обробниками. Тому туди, де досить синхронного виклику на 50 мс, event-driven тягнути не варто — це складність без виграшу.

Конкретна проблема з нашого стека: студент завантажує PDF для RAG-бази знань (RAG — підхід, коли AI-модель відповідає, спираючись на знайдені в базі шматки твоїх документів). Обробка — розбити текст на чанки (невеликі фрагменти), порахувати для кожного embeddings (числові вектори, що кодують зміст фрагмента — за ними шукають схожі), зберегти — займає 30–60 секунд. Тримати HTTP-запит відкритим весь цей час не можна: користувач дивиться на спінер, з'єднання відвалюється по таймауту, а при сплеску завантажень сервер захлинається. Рішення — **розчепити** «прийняти запит» і «зробити роботу»: це і є event-driven підхід. Компоненти спілкуються через **події**, а не прямі виклики.

Три компоненти, кожен зі своєю роллю:
- **Producer (видавець)** — фіксує, що щось сталося, і публікує **подію**. Подія — це запис про факт минулого («DocumentUploaded», «замовлення створене»), а не команда «зроби X»: producer не знає і не мусить знати, хто і як відреагує.
- **Broker / черга (queue)** — посередник, який приймає події і зберігає, поки їх не оброблять (RabbitMQ, Kafka, SQS; у Node-стеку — BullMQ поверх Redis). Ключова властивість: producer і consumer розчеплені **в часі** — consumer може бути зайнятий чи навіть лежати, події дочекаються.
- **Consumer (обробник)** — підписаний на події і виконує роботу у своєму темпі, незалежно від темпу надходження.

Як це виглядає на BullMQ у NestJS для нашого інжесту:

```ts
// producer — HTTP-контролер: відповідає МИТТЄВО, роботу кладе в чергу
@Post("documents")
async upload(@Body() dto: UploadDto) {
  const doc = await this.docs.create(dto);
  await this.ingestQueue.add("ingest", { documentId: doc.id }); // ← подія в чергу
  return { id: doc.id, status: "processing" };                  // ← користувач не чекає
}

// consumer — воркер: розгрібає чергу у своєму темпі, окремо від HTTP
@Processor("ingest")
class IngestWorker extends WorkerHost {
  async process(job: Job<{ documentId: string }>) {
    // чанкінг → embeddings → збереження; впало — BullMQ зробить retry
  }
}
```

Звідки беруться події (тригери) — три джерела, і всі лягають в ту саму архітектуру: **user-driven** — дія користувача (завантажив документ, записався на курс); **time-driven** — розклад (нічний перерахунок статистики, щотижневий дайджест: cron — системний планувальник, що запускає задачі за розкладом, — просто виступає producer'ом); **system-driven** — інша система (webhook — HTTP-запит, яким чужий сервіс сам сповіщає твій: платіжка стукає «оплата пройшла»). Практичний trade-off між першими двома: миттєва реакція на кожну дію користувача — дорога; усе, де затримка прийнятна, виносять у відкладену пакетну обробку за розкладом.

**Переваги:** слабка зв'язність (додати нового споживача події = нуль змін у producer), масштабованість (воркерів можна множити незалежно), стійкість (черга згладжує сплески: сто завантажень за хвилину просто чекають у черзі). **Ціна:** система стає асинхронною — «завантажив, а результату ще нема» треба відображати в UX; дебаг важчий (потрібен tracing — наскрізне відстеження шляху події через усі кроки за спільним id, бо стектрейс через чергу не «перестрибує»); потрібна стратегія повторів і обробки «отруйних» повідомлень. Тому **не** тягни event-driven туди, де досить синхронного виклику: для операції на 50 мс черга — це складність без виграшу.

**Підсумок-якір:** Producer → черга → consumer: «прийняти запит» розчеплено з «зробити роботу». Плюси — слабка зв'язність і згладжування сплесків; ціна — асинхронний UX, наскрізний tracing і retry з ідемпотентними обробниками.

**➕ Пов'язане: що таке черга повідомлень (message queue) і навіщо? 🟢**
Це той broker із прикладу вище, найважливіший компонент зв'язки. Черга приймає повідомлення і **тримає** їх, поки consumer не підтвердить обробку. Що це дає: буферизацію (сплеск зі 100 подій не завалює воркера — він бере по одній у своєму темпі), надійність (воркер упав посеред обробки — повідомлення не підтверджене і буде видане знову), retry з відкладенням для помилкових задач і рівномірне розподілення роботи між кількома воркерами. По суті, черга — це «список справ» системи, який не губиться.

**➕ Пов'язане: що таке ідемпотентність споживача і чому вона критична? 🟢**
Випливає прямо з механіки retry вище: якщо воркер обробив подію, але впав **до** підтвердження, черга видасть її вдруге — гарантія доставки зазвичай at-least-once («принаймні один раз», а не «рівно один раз»). Тому consumer має бути ідемпотентним: повторна обробка тієї самої події не подвоює ефект. Для нашого інжесту: перед обробкою перевірити, чи документ уже має статус processed (або тримати таблицю оброблених id подій) — інакше подвоєна подія створить дублікати чанків у базі, а в платіжному сценарії — спише гроші двічі.

## Q12. Чим відрізняються prototype, PoC і MVP? 🟢

**🎯 Відповідь на іспиті:** Це три різні експерименти, що відповідають на три різні питання: PoC (Proof of Concept) — «чи це технічно можливо?» — одноразовий код без UI, який після відповіді викидається; prototype — «чи так це має виглядати і працювати для користувача?» — клікабельний макет без справжнього бекенду для збору фідбеку; MVP (Minimum Viable Product) — «чи це комусь потрібно?» — мінімальна, але справжня робоча версія в руках реальних користувачів. Послідовність знімає ризики по черзі — технічний → UX → ринковий, — але проходити всі три не обов'язково: роби той експеримент, чиє питання зараз найризикованіше. Типова помилка — підміна питань: показати prototype як «майже готовий продукт» або зробити з PoC висновок про попит.

Три терміни плутають, бо всі вони — «щось зробити швидко до повноцінного продукту». Але це три різні експерименти, що відповідають на **три різні питання**, і плутанина коштує грошей: команда будує не той артефакт і отримує відповідь не на своє питання.

- **PoC (Proof of Concept)** відповідає на питання «**чи це технічно можливо?**». Перевіряється найризикованіше технічне припущення — потворним одноразовим кодом, без UI. Наш приклад: «чи зможе RAG-пайплайн точно відповідати на питання по PDF-конспектах українською?» — скрипт на день-два, який жене реальні документи через пайплайн. Відповів «так/ні» — своє відпрацював, код викидається.
- **Prototype** відповідає на «**чи так це має виглядати і працювати для користувача?**». Це макет UX — клікабельний Figma-мокап чи фронт на хардкоджених даних, — який показують живим користувачам до написання бекенду. Не робочий продукт: цінність у фідбеку про сценарії і зрозумілість, поки зміна «пересунути весь флоу» коштує години, а не тижні.
- **MVP (Minimum Viable Product)** відповідає на «**чи це комусь потрібно?**». Мінімальна, але **справжня робоча** версія в руках реальних користувачів: вузький зріз цінності, за яким видно реальну поведінку (користуються? повертаються? платять?). MVP нашої платформи — один курс, один AI-тьютор, оплата хоч вручну через інвойс — але реальні студенти реально вчаться.

Логічна послідовність ризиків: PoC знімає технічний ризик → prototype знімає ризик UX → MVP знімає ринковий ризик. Не обов'язково проходити всі три — роби той експеримент, чиє питання зараз найризикованіше: якщо технологія відома, PoC не потрібен; якщо попит очевидний, можна одразу цілитись в MVP.

Типова помилка — підміна питань: показати інвесторам prototype як «майже готовий продукт» (він не робочий) або зробити висновок «людям треба» з PoC (він нічого не каже про попит).

**Підсумок-якір:** PoC — «чи можливо технічно?» (одноразовий код, викидається); prototype — «чи так має виглядати?» (макет без бекенду); MVP — «чи потрібно комусь?» (вузький, але справжній продукт). Роби експеримент під найризикованіше питання.

**➕ Пов'язане: чому MVP часто розуміють неправильно? 🟢**
Два протилежні спотворення слова «minimum». Перше: MVP = «сирий недороблений продукт» — ні, він мінімальний за **обсягом**, не за якістю: вузький зріз, але цілісний і робочий (одна фіча, що працює добре, а не десять напівзламаних). Друге: MVP = «перша половина великого плану» — ні, це інструмент **перевірки гіпотези**: якщо після запуску ти не дивишся на поведінку користувачів і нічого не коригуєш, це був не MVP, а просто малий реліз. Тест на розуміння: назви гіпотезу, яку твій MVP перевіряє, і що ти зробиш при кожному з результатів.

**➕ Пов'язане: чи можна код PoC пускати в продакшн? 🟢**
Класична пастка: PoC «запрацював», дедлайн тисне — і одноразовий скрипт обростає фічами прямо в проді. Проблема в тому, що PoC писався з правильним для нього ігноруванням помилок, безпеки, тестів і структури — це був його свідомий компроміс заради швидкості відповіді. У продакшні цей компроміс перетворюється на технічний борг (Q7) із першого дня. Здорова практика: PoC доводить здійсненність → викидається → робоча версія пишеться заново вже з інженерними вимогами. Якщо команда «не може дозволити собі переписати» — це варто чесно назвати взятим боргом, а не безкоштовною перемогою.

## Q13. Що таке SDD, BDD, DDD? 🟢

**🎯 Відповідь на іспиті:** Це три підходи зі спільною ідеєю «зафіксувати, що будуємо, до написання коду», але різною «D» посередині: SDD (spec-driven development) — спершу специфікація з контрактами, правилами і критеріями, потім код за нею; BDD (behaviour-driven development) — поведінка описується мовою бізнесу у форматі Given-When-Then, і цей самий текст виконується як тест (Cucumber і подібні); DDD (domain-driven design) — найширший: проєктування системи навколо бізнес-домену зі спільною мовою у розмовах і в коді (ubiquitous language) та явними межами контекстів (bounded contexts). Швидке розведення: SDD — «спершу зафіксуй що», BDD — «опиши поведінку мовою бізнесу, і хай це буде тестом», DDD — «структуруй саму систему за устроєм бізнесу».

Три абревіатури плутають через схожість, але «D» посередині у них різна: **Spec**-driven, **Behaviour**-driven, **Domain**-driven. Спільна ідея одна — зафіксувати «що будуємо» до написання коду; різниця в тому, **хто** є носієм цього знання і **якою мовою** воно записане.

- **SDD (Spec-Driven Development)** — спершу специфікація (документ з контрактами, правилами, критеріями), потім код за нею. Детально розібрано в Q4 — це той самий підхід, що став особливо практичним з AI-агентами. Мова — інженерна, аудиторія — розробники й агенти.
- **BDD (Behaviour-Driven Development)** — специфікація поведінки **мовою бізнесу**, у форматі Given-When-Then, причому цей текст одночасно виконується як тест. Біль, який лікує BDD: продакт формулює вимогу словами, розробник розуміє по-своєму, тестувальник перевіряє третє — і розбіжність спливає в проді. BDD дає всім трьом один артефакт:

```gherkin
# Це читає продакт-менеджер БЕЗ перекладу — і це ж запускається як тест
Scenario: Студент записується на курс із вільними місцями
  Given студент авторизований            # передумова (Arrange)
  And на курсі "TypeScript" є вільні місця
  When він натискає "Записатись"          # дія (Act)
  Then він бачить курс у "Мої курси"      # очікуваний результат (Assert)
```

  Given-When-Then — це та сама AAA-структура тестів (Q8), тільки словами домену. Інструменти (Cucumber та подібні) прив'язують кожен рядок до коду-виконавця. Ціна BDD — підтримка цієї прив'язки; якщо бізнес насправді не читає сценаріїв, виходить «дорогий синтаксис заради нікого» — тоді чесніше писати звичайні тести.
- **DDD (Domain-Driven Design)** — найширший із трьох: підхід до **проєктування** системи навколо бізнес-домену. Біль: код говорить технічною мовою (`DataManager`, `ItemProcessor`, `flag3`), бізнес — доменною («запис», «потік», «атестація»), і кожна розмова вимагає перекладу, в якому губляться нюанси. DDD вимагає спільної мови в розмовах І в коді (**ubiquitous language**) та явних меж між частинами домену (**bounded contexts**) — обидва терміни розгорнуто нижче.

Швидке розведення: SDD — «спершу зафіксуй що», BDD — «опиши поведінку мовою бізнесу, і хай це буде тестом», DDD — «структуруй саму систему за устроєм бізнесу».

**Підсумок-якір:** SDD — спершу зафіксуй «що» (специфікація); BDD — поведінка мовою бізнесу (Given-When-Then) як виконуваний тест; DDD — система структурується за доменом: ubiquitous language + bounded contexts.

**➕ Пов'язане: що таке ubiquitous language у DDD? 🟢**
«Спільна мова» з визначення DDD, доведена до дисципліни: одні й ті самі доменні слова у розмовах, документах **і в коді** — класи `Enrollment`, `Cohort`, метод `enrollStudent()`, а не `DataRecord` і `processItem()`. Механізм користі простий: щоразу, коли між мовою бізнесу і кодом стоїть переклад, у ньому губляться правила («запис» і «бронювання місця» — одне й те саме чи ні?). Спільний словник робить розбіжності видимими одразу — а код стає читабельним для нового розробника буквально на рівні назв.

**➕ Пов'язане: що таке bounded context? 🟢**
Друга опора DDD, відповідь на класичну пастку «один клас User на всю систему». Слово «користувач» у білінгу означає платника (картки, інвойси), у навчанні — студента (прогрес, записи на курси), у підтримці — автора тикетів. Спроба вмістити все в одну модель дає бога-об'єкт із 40 полів, де кожен модуль використовує свої п'ять. Bounded context — явна межа, всередині якої термін має рівно один сенс: контекст «Білінг» має свого `Customer`, контекст «Навчання» — свого `Student`, зв'язані спільним id. Межі контекстів — природні шви для розрізання системи на модулі (або мікросервіси).

## Q14. Що таке Big O і як оцінювати/оптимізувати продуктивність коду? 🔵

**🎯 Відповідь на іспиті:** Big O — це мова для опису того, як зростає час чи пам'ять алгоритму зі зростанням розміру входу n: не «скільки мілісекунд», а характер зростання — O(1), O(log n), O(n), O(n log n), O(n²). Практична навичка — читати складність прямо з коду: вкладені цикли множаться, послідовні кроки додаються і більший з'їдає менший, а найпідступніше — прихована вкладеність, коли `includes` чи `indexOf` у циклі дають O(n²) при одному видимому циклі. Оптимізую за драбиною: спершу виміряти профайлером і знайти реальне вузьке місце (інтуїція тут регулярно бреше), потім виправити асимптотику — типово замінити багаторазовий пошук у масиві на `Map`/`Set` з O(1), — і лише коли асимптотика оптимальна, братися за мікрооптимізації констант. Найчастіша помилка — почати з мікрооптимізацій, минаючи вимірювання.

Практична ситуація: функція дедуплікації списку студентів працює миттєво на тестових 100 записах — і вішає сторінку на реальних 50 000. Код «правильний», але його час зростає **квадратично**: на 500× більше даних — у 250 000× повільніше. Big O — це мова, якою описують саме це: **як зростає** час (чи пам'ять) алгоритму зі зростанням розміру входу `n`. Не «скільки мілісекунд» (це залежить від заліза), а характер зростання.

| Позначення | Назва | Інтуїція: n зріс у 1000× → час... | Приклад |
|---|---|---|---|
| `O(1)` | константа | не змінився | доступ за індексом, `Map.get` |
| `O(log n)` | логарифм | +10 кроків (log₂1000≈10) | бінарний пошук, індекс БД |
| `O(n)` | лінійний | ×1000 | один прохід масивом |
| `O(n log n)` | | ×10 000 | ефективні сортування (`Array.sort`) |
| `O(n²)` | квадратичний | ×1 000 000 | вкладені цикли по тих самих даних |

Головна практична навичка — **читати Big O прямо з коду**. Чотири правила покривають майже все:
1. **Вкладеність множиться:** цикл по n усередині циклу по n → `O(n²)`. Три рівні → `O(n³)`.
2. **Послідовність додається, більший з'їдає менший:** прохід `O(n)`, потім сортування `O(n log n)` → разом `O(n log n)` (при великих n менший доданок неістотний; константи-множники теж відкидаються — `O(2n)` це `O(n)`).
3. **«Ділимо навпіл щокроку» → `O(log n)`:** бінарний пошук, збалансовані дерева. Тому індекс БД знаходить запис серед мільйона за ~20 кроків.
4. **Найпідступніше — прихована вкладеність:** `includes`, `indexOf`, `find` — це **лінійний пошук** усередині; викликаний у циклі, він дає `O(n²)`, хоч у коді видно лише один цикл.

```ts
// ❌ O(n²) — правило 4 у дії: один видимий цикл (filter), але indexOf
//    усередині щоразу сканує масив з початку → n × n
const dups = arr.filter((x, i) => arr.indexOf(x) !== i);

// ✅ O(n): та сама задача одним проходом; Set дає перевірку "чи бачили" за O(1)
const seen = new Set<number>(), dups2 = new Set<number>();
for (const x of arr) (seen.has(x) ? dups2 : seen).add(x);
// ціна — правило trade-off: O(n) додаткової пам'яті під Set (див. похідне)
```

Драбина дій, коли «щось повільно»: (1) спершу **виміряй** і знайди реальне вузьке місце (профайлер — інструмент, що показує, скільки часу програма проводить у кожній функції; грубіше — `console.time` навколо підозрілих ділянок) — інтуїція тут регулярно бреше, а оптимізація не того місця — витрачений час і ускладнений код; (2) у знайденому місці подивись на **асимптотику** (характер зростання — той самий Big O): випадковий `O(n²)` (пошук у циклі) → `Map`/`Set`; робота, що повторюється, → зробити один раз до циклу; (3) і лише коли асимптотика вже оптимальна — мікрооптимізації констант, і то за виміром. Найчастіша помилка L3 — почати з кроку 3, минаючи 1 і 2.

**Підсумок-якір:** Big O — характер зростання, не мілісекунди: вкладеність множиться, прихований лінійний пошук (`includes` у циклі) дає O(n²), `Map`/`Set` повертають O(1). Порядок оптимізації: виміряй → виправ асимптотику → лише потім константи.

**➕ Пов'язане: чому іноді O(n²) прийнятне? 🔵**
Той самий приклад із `filter+indexOf`: на масиві з 20 елементів він виконає ~400 дешевих операцій — це мікросекунди, а читається він у пів рядка. Big O описує **зростання** і каже правду для великих n, але мовчить про константи: у «повільного» алгоритму вони бувають менші (не створюється Set, дані масиву лежать у пам'яті поспіль і читаються швидше), тож на малих даних він реально швидший. Правило: якщо n гарантовано мале й обмежене (дні тижня, колонки таблиці) — пиши найпростіше; якщо n росте з даними користувачів — думай про асимптотику одразу, бо «на моїх тестових даних швидко» і був той баг із початку відповіді.

**➕ Пов'язане: у чому різниця часової та просторової складності? 🟢**
У ✅-прикладі ми виграли час, заплативши пам'яттю під Set — це і є класичний trade-off. Часова складність — як росте **час**, просторова — як росте **додаткова пам'ять** з n. Часто одну купують за іншу: кеш, індекс, hash-таблиця — усе це «пам'ять в обмін на час». Обирати треба за тим, що дефіцитніше: на бекенді зазвичай жертвують пам'яттю заради часу відповіді; в обмежених середовищах (edge, мобільний) — навпаки.

**➕ Пов'язане: яка складність в основних операцій JS-колекцій? 🔵**
Практичний словничок, з якого випливає половина оптимізацій вище: масив — доступ за індексом `O(1)`, `push/pop` `O(1)`, а `includes/indexOf/find` — `O(n)` (лінійний скан!), `shift/unshift` — `O(n)` (зсув усіх елементів), `sort` — `O(n log n)`. `Map`/`Set` — `get/set/has/delete` у середньому `O(1)` (всередині хеш-таблиця: з ключа обчислюється номер комірки, тому пошук не залежить від кількості елементів). Звідси головний рефлекс: **багаторазовий пошук у масиві → перекласти в Map/Set один раз** (`O(n)` підготовки) і далі шукати за `O(1)`.

## Q15. Що таке PRD і TDD (technical design document)? 🟢

**🎯 Відповідь на іспиті:** PRD (Product Requirements Document) фіксує, *що* і *навіщо* будуємо: проблема користувача, функціональні вимоги, критерії успіху — його пише продакт-менеджер мовою бізнесу. TDD (Technical Design Document) фіксує, *як* реалізуємо: архітектура, моделі даних, API-контракти, розглянуті альтернативи і ризики — його пише інженер на основі PRD. Логіка зв'язки PRD → TDD → код: кожен крок звужує невизначеність, поки помилка дешева, — без PRD команда будує не те, без TDD будує не так; але на дрібну фічу окремі документи — оверхед, вони мають масштабуватися під ризик рішення.

Два документи, що знімають два різні ризики до початку розробки. Без першого команда будує **не те**, без другого — будує **не так**.

- **PRD (Product Requirements Document)** — фіксує *що* і *навіщо*: яку проблему користувача розв'язуємо, для кого, які функціональні вимоги, як виміряємо успіх. Пише зазвичай продакт-менеджер, мовою бізнесу й користувача — жодних таблиць БД. Що станеться без нього: кожен тримає в голові власну версію фічі, обсяг повзе («а давайте ще…»), а після релізу неможливо сказати, вдалася фіча чи ні — бо ніхто не записав, що вважається успіхом.
- **TDD (Technical Design Document)** — фіксує *як* реалізуємо: архітектура рішення, моделі даних, API-контракти, розглянуті альтернативи і чому обрано цю, ризики. Пише інженер **на основі PRD**. Що станеться без нього для нетривіальної фічі: ключові рішення ухвалюються нашвидку в процесі кодування, стики між частинами (фронт чекає одне API, бек робить інше) з'ясовуються при інтеграції, а через рік ніхто не пам'ятає, **чому** зроблено саме так — і «дивне» рішення, що захищало від реальної проблеми, зносять разом із захистом.

Приклад зв'язки на фічі «AI-тьютор у курсі»: PRD каже — студенти кидають курс, коли застрягають без допомоги; тьютор відповідає на питання по матеріалах курсу; успіх = зростання завершень курсу. TDD каже — RAG по матеріалах (Q11: інжест через чергу), модель X, історія діалогу в PostgreSQL, ліміт токенів на студента; альтернатива «дообучення моделі» відхилена через вартість.

Логіка проста: PRD → TDD → код, і кожен крок звужує невизначеність до того, як помилка стане дорогою. Межі здорового глузду: на дрібну фічу («додати фільтр у список») окремі документи — оверхед, досить пари абзаців у задачі; документи мають масштабуватися під ризик рішення, а не існувати заради процесу.

**Підсумок-якір:** PRD — «що і навіщо» від продакта мовою бізнесу; TDD (design doc) — «як» від інженера на основі PRD. Без першого будуємо не те, без другого — не так; масштаб документів — під ризик рішення.

**➕ Пов'язане: навіщо писати design doc перед кодом? 🟢**
Розгорнемо головний аргумент на користь TDD-документа: він переносить критику рішення в точку, де зміни **найдешевші**. Помилка архітектури, помічена на рев'ю документа, — це виправлений абзац за десять хвилин; та сама помилка, помічена після реалізації, — тижні переписування. Плюс сам акт письма структурує мислення: половина дір у дизайні виявляється, коли намагаєшся зв'язно його викласти. І довгостроковий бонус: документ лишається як пам'ять рішень — новачок читає «чому так», замість реконструювати логіку з коду.

**➕ Пов'язане: TDD-документ і TDD-практика (test-driven development) — це одне й те саме? 🟢**
Ні, лише колізія абревіатур, і на співбесіді її варто вміти розвести одним реченням: Technical Design Document — це **документ** технічного дизайну перед розробкою (це наше поточне питання); Test-Driven Development — це **практика** написання тестів до коду (згадувалась у Q4). Спільного між ними тільки ідея «спершу зафіксуй намір, потім реалізуй». Який TDD мається на увазі — визначай із контексту: поруч із PRD це майже завжди документ.

## Q16. Що таке Creational patterns (породжувальні патерни)? 🔵

**🎯 Відповідь на іспиті:** Creational (породжувальні) патерни — це іменовані типові рішення проблем створення об'єктів: що робити, коли просте `new` починає боліти. Основні: Singleton — на всю систему потрібен рівно один екземпляр (з'єднання з БД, конфіг), причому в NestJS цю гарантію дає DI-контейнер за замовчуванням, і руками його майже не пишуть; Factory — знання «який конкретний клас створити» збирається в одному місці замість розповзатися if-ами по коду, а клієнти знають лише спільний інтерфейс; Builder — покрокова збірка об'єкта з багатьма опційними параметрами замість конструктора з чергою null-ів (впізнаваний приклад — TypeORM QueryBuilder); Prototype — створення копіюванням наявного об'єкта, що в JS практично вбудовано (`structuredClone`). Головне правило: патерн вводиться у відповідь на реальний біль, а не про запас — «патерноманія» сама по собі антипатерн.

Спершу — навіщо взагалі патерни. Це не бібліотека і не синтаксис, а **іменовані типові рішення** повторюваних проблем проєктування. Половина їхньої цінності — спільна мова: сказати «тут проситься Factory» швидше, ніж описувати рішення на три абзаци. Породжувальна група відповідає за одну ділянку — **створення об'єктів**: що робити, коли просте `new` починає боліти.

Ще два нагадування, потрібні всім трьом питанням про патерни. **Інтерфейс** — контракт «які методи в об'єкта є», без деталей, як вони зроблені; код, що залежить лише від інтерфейсу, байдужий до того, який конкретний клас (реалізація) за ним стоїть — саме на цьому тримається взаємозамінність у більшості патернів. **DI (dependency injection)** — клас не створює свої залежності сам через `new`, а оголошує їх у конструкторі, і DI-контейнер (у нас — NestJS) створює їх і передає; **провайдер** — будь-який клас, яким контейнер керує.

**Singleton — «на всю систему потрібен рівно один екземпляр».** Біль: кожен сервіс створює власне з'єднання до PostgreSQL — пул з'єднань (обмежений набір відкритих з'єднань до БД, які перевикористовуються) вичерпано, конфіг читається десять разів. Класичний Singleton — клас, що сам стежить за єдиністю екземпляра (статичний `getInstance()`). Але в нашому стеку його руками майже не пишуть: **NestJS DI-контейнер за замовчуванням тримає кожен провайдер у єдиному екземплярі** — твій `DatabaseService` уже Singleton, без глобальних змінних і з можливістю підмінити його моком у тестах. Це важлива еволюція патерна: ідея жива, ручна реалізація — здебільшого ні (чому — у похідному питанні).

**Factory — «вибір конкретного класу розповзся по коду».** Біль і рішення в контрасті:

```ts
// ❌ біль: рішення "який клас створити" продубльоване скрізь, де шлемо сповіщення
if (user.channel === "sms") await new SmsNotifier(twilioCfg).send(text);
else await new EmailNotifier(smtpCfg).send(text);
// ...і так у 5 місцях. Додати Telegram = знайти і поправити всі 5 (одне забудеш)

// ✅ Factory: знання "як створюється Notifier" живе в ОДНОМУ місці
function createNotifier(channel: Channel): Notifier {   // всі повертають спільний інтерфейс
  switch (channel) {
    case "sms":   return new SmsNotifier(twilioCfg);
    case "email": return new EmailNotifier(smtpCfg);
  }
}
// клієнтський код знає лише інтерфейс Notifier; новий канал = одна правка тут
```

По суті це DRY (Q6), застосований до інстанціювання. У NestJS та сама ідея — custom provider із `useFactory`, коли створення залежності вимагає логіки чи конфігурації.

**Builder — «конструктор із десятьма параметрами».** Біль: `new Course(title, desc, null, true, undefined, 30, null)` — що означає `true`? який із null-ів обов'язковий? Builder збирає складний об'єкт покроково іменованими викликами: `.title("TS").maxStudents(30).withCertificate().build()` — читається як речення, опційне просто пропускається. Ти вже користуєшся ним щодня: **TypeORM QueryBuilder** (`createQueryBuilder().where().orderBy().getMany()`) — саме Builder. Нюанс TS: для простих випадків досить об'єкта-параметра з опційними полями (`new Course({ title, maxStudents? })`) — це «Builder для бідних», і його часто вистачає (KISS).

**Prototype — «створити об'єкт копіюванням наявного»**, коли створення з нуля дороге або зразок налаштований у рантаймі. У JS клонування вбудоване (`structuredClone`, spread), тож як окремий патерн зустрічається рідко — досить упізнавати назву.

**Коли патерни НЕ треба:** якщо болю нема — нема й патерна. Один канал сповіщень → Factory не потрібна, достатньо `new`. Два обов'язкові параметри → Builder — це шум. «Патерноманія» (вивчив — застосовую скрізь) — сама по собі антипатерн і головний спосіб завалити це питання на практиці: патерн вводиться у відповідь на реальний, наявний біль, а не про запас (YAGNI, Q6).

**Підсумок-якір:** Породжувальні патерни — про створення об'єктів: Singleton — один екземпляр (у NestJS це вже робить DI), Factory — вибір класу в одному місці, Builder — покрокова збірка (QueryBuilder), Prototype — копіювання наявного. Нема болю — нема патерна.

**➕ Пов'язане: чому Singleton часто вважають антипатерном? 🔵**
Вище ми сказали «руками майже не пишуть» — ось чому. Класичний Singleton зі статичним `getInstance()` — це прихований **глобальний стан**: будь-який код може дістати його нізвідки, залежність не видно в сигнатурах, тести не можуть підмінити його моком (він жорстко зашитий), а стан протікає між тестами. DI-контейнер дає ту саму гарантію єдиності **без цих проблем**: залежність оголошена в конструкторі (видима), контейнер створює один екземпляр, а в тестах підставляється мок через той самий механізм. Тобто антипатерн — не ідея «один екземпляр», а спосіб її досягнення через глобальну точку доступу.

**➕ Пов'язане: коли Builder кращий за конструктор? 🟢**
Критерій із прикладу вище: коли параметрів багато і більшість опційні — замість «телескопічних» конструкторів (кілька перевантажень під різні комбінації аргументів) або виклику з чергою `null`-ів. Додатковий виграш: Builder може **валідувати цілісність** у `.build()` (наприклад, «сертифікатний курс мусить мати екзамен») — неможливо отримати недозібраний об'єкт. У TypeScript поріг входу вищий: спершу спробуй об'єкт-параметр — якщо і його недостатньо (потрібні кроки, валідація між полями), тоді Builder.

## Q17. Що таке Behavioural patterns (поведінкові патерни)? 🔵

**🎯 Відповідь на іспиті:** Behavioural (поведінкові) патерни описують взаємодію об'єктів і розподіл обов'язків: хто що знає, хто кого викликає, як додати нову поведінку, не переписуючи стару. Основні: Strategy — взаємозамінні алгоритми замість розростання if/else, новий варіант додається без зміни старих; Observer — джерело лише публікує подію, а споживачі самі підписуються (`EventEmitter2` у NestJS, `addEventListener`, RxJS); Command — дія упакована в об'єкт, який можна покласти в чергу, повторити чи скасувати (job у BullMQ — це він); State — поведінка кожного статусу в окремому класі замість if-ів за статусом по всьому коду; Chain of Responsibility — запит іде ланцюжком обробників, кожен обробляє або передає далі (middleware в Express, Guards → Pipes → Interceptors у NestJS). Правило групи: патерн вводиться, коли біль уже є — третій `else if` за алгоритмом, третій статус, — а Strategy на два стабільні варіанти це over-engineering.

Поведінкові патерни відповідають за **взаємодію об'єктів і розподіл обов'язків**: хто що знає, хто кого викликає, як додати нову поведінку, не переписуючи стару. Розберімо основні через біль, який кожен лікує, і місця в нашому стеку, де ти їх уже використовуєш (часто не знаючи назви).

**Strategy — «розгалуження за алгоритмом росте і множиться».** Біль:

```ts
// ❌ вся логіка ціноутворення вшита в одну функцію
function coursePrice(base: number, user: User) {
  if (user.isStudent) return base * 0.5;
  else if (user.company) return base * 0.8;
  else if (isBlackFriday()) return base * 0.7;   // новий тариф = правка цієї функції,
  return base;                                    // ризик зламати наявні гілки, тести розпухають
}

// ✅ Strategy: кожен алгоритм — окрема взаємозамінна одиниця
type PricingStrategy = (base: number) => number;
const strategies: Record<Plan, PricingStrategy> = {
  student:  (b) => b * 0.5,
  corporate:(b) => b * 0.8,     // новий тариф = новий рядок, старі НЕ ЧІПАЮТЬСЯ
  regular:  (b) => b,           // і кожна стратегія тестується окремо
};
const price = strategies[user.plan](base);
```

Зверни увагу: у TS стратегія — це часто просто функція, клас не обов'язковий. Виправдано, коли варіантів ≥3 і вони додаються; для двох стабільних гілок `if` простіший і чесніший.

**Observer — «джерело події знає всіх зацікавлених».** Біль: `EnrollmentService` після запису сам шле email, оновлює статистику, інвалідує кеш (позначає закешовані дані застарілими, щоб наступний запит перечитав свіжі) — він **знає про всіх** і змінюється при появі кожного нового споживача. Observer розвертає залежність: сервіс лише публікує «студент записався», а споживачі **самі підписуються**. У стеку це всюди: `EventEmitter2` у NestJS (`emit("enrollment.created")` + `@OnEvent(...)`-обробники), RxJS Observables, DOM `addEventListener`, підписки стору. Це також синхронний «молодший брат» event-driven архітектури з Q11 — та сама ідея в межах одного процесу. Ціна: потік виконання стає неявним — «хто реагує на цю подію?» вже не видно в місці виклику.

**Command — «дія як об'єкт».** Біль: дію треба не виконати негайно, а покласти в чергу, повторити при збої, скасувати, залогувати. Рішення — упакувати дію разом з її параметрами в об'єкт, який можна передавати і зберігати. Впізнаваний приклад зі стека: **job у BullMQ** (Q11) — це Command у чистому вигляді (ім'я + payload, виконається пізніше, можливий retry). Другий класичний кейс — undo/redo в редакторах: стек виконаних команд.

**State — «поведінка залежить від статусу, і if-и за статусом розповзлися».** Біль: курс має статуси draft/published/archived, і кожен метод починається з `if (status === ...)` — правила одного статусу розмазані по всьому класу. State збирає поведінку кожного стану в окремий клас/об'єкт: `DraftState.publish()` — валідно, `ArchivedState.publish()` — помилка. Виправдано від ~3+ станів зі складними переходами — по суті виходить state machine (модель «скінченний набір станів + явно дозволені переходи між ними»); для двох статусів досить поля і перевірок.

**Chain of Responsibility — «запит має пройти низку обробників, кожен може обробити або передати далі».** Ти всередині нього щодня: **middleware в Express і Guards → Pipes → Interceptors у NestJS** — запит іде ланцюжком, будь-яка ланка може зупинити (Guard повернув false) або збагатити і передати далі. Додати крок обробки = вставити ланку, не чіпаючи інші.

**Iterator** — уніфікований обхід колекції без знання її внутрішньої будови. У JS вбудований у мову: протокол ітераторів, `for..of`, генератори `function*` — окремо реалізовувати майже ніколи не треба, досить упізнавати.

Загальне правило групи — те саме, що в Q16: патерн вводиться, коли біль уже є (третій `else if` за алгоритмом, третій статус), а не наперед. Strategy на два варіанти чи State на два статуси — це over-engineering (Q6).

**Підсумок-якір:** Поведінкові патерни — про взаємодію об'єктів: Strategy — взаємозамінні алгоритми, Observer — публікація/підписка, Command — дія як об'єкт у черзі (BullMQ job), State — поведінка за статусом, Chain of Responsibility — ланцюжок обробників (middleware).

**➕ Пов'язане: чим Strategy відрізняється від простого if/else? 🔵**
Це різниця між ❌ і ✅ у прикладі вище, сформульована принципово: `if/else` вшиває всі варіанти в одне місце — додати новий означає **змінити** робочий код (ризик регресії, тести функції розпухають). Strategy робить систему розширюваною **додаванням**: новий варіант = нова стратегія, наявний код не чіпається (це принцип Open-Closed: відкритий для розширення, закритий для модифікації). Плюс кожна стратегія ізольовано тестується і може обиратися в рантаймі (з конфігу, з БД). Але напрямок не односторонній: для двох стабільних варіантів if/else читабельніший — критерій переходу: «варіанти множаться».

**➕ Пов'язане: де Observer у знайомих інструментах? 🟢**
Продовжимо список з основної відповіді — Observer буквально скрізь, де є слова «підписатися» або «слухати»: `element.addEventListener` у DOM, `EventEmitter` у Node і `EventEmitter2` у NestJS, RxJS `Observable.subscribe`, `store.subscribe` у Redux, вебхуки (Observer через HTTP: сервіс «підписує» твій URL на події), навіть реактивність React-стейту концептуально звідси. Тому патерн і треба знати: ти не будеш його писати з нуля — ти щодня користуєшся готовими реалізаціями, і розуміння моделі «один публікує — багато реагують, не знаючи одне про одного» пояснює поведінку всіх цих API.

## Q18. Що таке Structural patterns (структурні патерни)? 🔵

**🎯 Відповідь на іспиті:** Structural (структурні) патерни — про композицію: як зібрати з об'єктів більші конструкції так, щоб частини можна було міняти незалежно. Основна четвірка розводиться за дією: Adapter змінює інтерфейс — перекладає чужий SDK у наш, щоб зовнішні бібліотеки не протікали в бізнес-логіку (сервіси залежать від нашого `FileStorage`, а не від S3-клієнта); Decorator додає поведінку, зберігаючи інтерфейс, — обгортки з кешем чи логуванням, що складаються як матрьошка (NestJS Interceptors); Facade спрощує — один метод ховає оркестрацію цілої підсистеми (`purchase()` замість чотирьох викликів у кожному клієнті); Proxy контролює доступ при тому самому інтерфейсі — ліниве створення, перевірка прав (lazy relations у TypeORM). Межа та сама, що всюди: шар заради шару — біль без вигоди, адаптер над бібліотекою, яку ніколи не зміниш, — over-engineering.

Структурні патерни — про **композицію**: як зібрати з об'єктів і класів більші конструкції, щоб частини можна було міняти незалежно. Якщо породжувальні (Q16) — про народження об'єктів, поведінкові (Q17) — про їхню взаємодію, то структурні — про те, як вони **складені** один з одним. Основна четвірка для L3 — через біль і впізнавані місця в стеку:

**Adapter — «чужий інтерфейс не пасує до нашого коду».** Біль: файли зберігаємо в S3 (хмарне файлове сховище AWS), і виклики `s3Client.putObject(...)` зі специфічними параметрами AWS розповзлися по сервісах. Тепер треба локальне сховище для тестів чи міграція на інший провайдер — правити доведеться скрізь. Adapter — прошарок, що «перекладає» чужий інтерфейс у наш:

```ts
// НАШ інтерфейс — така сигнатура зручна нашому коду
interface FileStorage { save(key: string, data: Buffer): Promise<void>; }

class S3Adapter implements FileStorage {          // перекладач: наш інтерфейс → мова S3 SDK
  constructor(private s3: S3Client) {}
  async save(key: string, data: Buffer) {
    await this.s3.send(new PutObjectCommand({ Bucket: BUCKET, Key: key, Body: data }));
  }
}
// сервіси залежать від FileStorage; заміна S3 на що завгодно = один новий адаптер,
// а в тестах — InMemoryAdapter без жодного мокання AWS
```

Правило, що з нього випливає: **зовнішні SDK не повинні протікати в бізнес-логіку** — на межі з кожною чужою системою став адаптер під власний інтерфейс.

**Decorator — «додати поведінку, не змінюючи клас».** Біль: сервісу потрібне кешування і логування, але вшивати їх у бізнес-логіку — значить змішати відповідальності (і продублювати в кожному сервісі). Decorator «загортає» об'єкт у обгортку з **тим самим інтерфейсом**, яка додає поведінку до/після делегування: `new CachedStorage(new LoggedStorage(new S3Adapter(...)))` — обгортки комбінуються як матрьошка, у будь-якому наборі. У стеку: **NestJS Interceptors** — це декоратори навколо обробника запиту (кеш, логування, трансформація відповіді додаються, не торкаючись контролера). Не плутай із TS-декораторами `@Injectable()` — це синтаксис метаданих, а патерн — про обгортання об'єктів.

**Facade — «клієнту треба проста дія, а всередині складна підсистема».** Біль: щоб оформити купівлю курсу, треба смикнути платіжку, створити enrollment, надіслати листа, оновити статистику — і кожен виклик цієї послідовності ризикує щось забути. Facade — один простий метод (`PurchaseService.purchase(userId, courseId)`), що ховає оркестрацію за собою. Клієнт не знає нутрощів → нутрощі можна перебудовувати вільно. По суті кожен добрий NestJS-сервіс — фасад над своєю ділянкою; **Repository над TypeORM** — теж він: доменні методи (`findActiveStudents()`) замість розсипаних по коду QueryBuilder-ланцюжків.

**Proxy — «між клієнтом і об'єктом потрібен контролер доступу».** Той самий інтерфейс, що в справжнього об'єкта (у цьому схожий на Decorator), але мета інша: не додати поведінку, а **керувати доступом** — ліниво створити дорогий об'єкт при першому зверненні, перевірити права, обмежити частоту викликів. У стеку живе непомітно: **lazy relations у TypeORM** (сутність-заглушка, що довантажує дані при зверненні), JS `new Proxy()` як механізм реактивності у Vue.

Швидке розведення на співбесіді, бо ці четверо схожі: Adapter **змінює інтерфейс** (чужий → наш), Decorator **додає поведінку** (інтерфейс той самий), Facade **спрощує** (багато об'єктів → один вхід), Proxy **контролює доступ** (інтерфейс той самий, об'єкт «за дверима»). І незмінне правило меж: шар заради шару — це біль без вигоди; адаптер над бібліотекою, яку ти ніколи не зміниш і чий інтерфейс тебе влаштовує, — чистий over-engineering (Q6).

**Підсумок-якір:** Структурні патерни — про композицію: Adapter змінює інтерфейс (чужий → наш), Decorator додає поведінку (інтерфейс той самий), Facade спрощує вхід у підсистему, Proxy контролює доступ. Шар без реального болю — over-engineering.

**➕ Пов'язане: чим Decorator відрізняється від успадкування? 🔵**
Обидва «додають поведінку», але по-різному. Успадкування — **статично**, на етапі написання коду: для комбінацій «кеш + лог», «тільки кеш», «лог + ретрай» доведеться плодити підклас на кожну комбінацію (комбінаторний вибух ієрархії). Decorator компонує **в рантаймі**: три обгортки дають будь-яку з восьми комбінацій простим складанням матрьошки, хоч за конфігом. Загальніший принцип за цим — «composition over inheritance»: композиція гнучкіша і не прив'язує розширення до ієрархії класів.

**➕ Пов'язане: що таке Facade і навіщо? 🟢**
Розгорнемо приклад з основної відповіді: без фасаду **кожен** клієнт (контролер, cron-задача, обробник черги) сам оркеструє «платіж → enrollment → лист → статистика» — послідовність продубльована, і одного дня хтось забуде лист. Facade збирає оркестрацію в одному місці: клієнти кличуть `purchase()`, дублювання зникає (DRY, Q6), а зв'язність падає (зв'язність — кількість речей, від яких код залежить і зміна яких може його зламати): клієнти залежать від одного інтерфейсу, а не від чотирьох підсистем, тож заміна платіжного провайдера не торкається жодного клієнта.

## Q19. Які є способи оцінки User Story (time, Fibonacci, T-shirt)? 🔵

**🎯 Відповідь на іспиті:** Є три способи оцінки user story, і це фактично драбина за рівнем знання про задачу. Time-based (години/дні) виглядає найточнішим, але страждає від систематичного оптимізму і читається менеджментом як обіцянка — чесно працює лише для дрібних добре зрозумілих задач. Story points за Fibonacci (1, 2, 3, 5, 8, 13, 21) — відносна оцінка складності через порівняння з уже зробленими задачами; зростаючі проміжки шкали чесні щодо невизначеності — «12 проти 13» ти все одно не розрізниш, — а оцінка 13+ означає «ми не розуміємо задачу, треба різати». T-shirt sizes (XS–XL) — найгрубша шкала для епіків на ранньому етапі, коли поінти вдавали б точність, якої не існує. У терміни поінти конвертуються не курсом «поінт = X годин», а емпірично через velocity команди.

Навіщо взагалі оцінювати: щоб планувати спринт (Q10 — скільки задач влізе), давати прогнози стейкхолдерам і вчасно помічати «цю задачу треба різати». Проблема в тому, що пряме питання «скільки годин займе?» дає систематично оптимістичні відповіді: людина уявляє happy path написання коду і не враховує з'ясування вимог, рев'ю, зламаний CI, хворого колегу. Усі три техніки нижче — різні способи жити з цією непереборною неточністю.

- **Time-based (години/дні)** — пряма оцінка часу. Виглядає найточнішою — і саме тому найоманливіша: цифра «6 годин» створює хибне відчуття зобов'язання, менеджмент читає її як обіцянку, а систематичний оптимізм нікуди не дівається. Чесно працює лише для дрібних, добре зрозумілих задач (до ~дня) на короткому горизонті.
- **Story points за Фібоначчі (1, 2, 3, 5, 8, 13, 21)** — оцінка **відносної складності/зусилля**, а не часу: «ця задача — як та минула п'ятірка чи вдвічі більша?» Порівнювати дві задачі людям дається значно краще, ніж вгадувати абсолютний час. Чому саме Фібоначчі, а не 1–10: зростаючі проміжки **чесні щодо невизначеності** — у великих задачах ти фізично не розрізниш «12 проти 13» (це самообман точності), тож шкала і не пропонує такого вибору: після 8 одразу 13. Оцінка 13+ — сигнал не «довго», а «ми не розуміємо задачу — різати» (назад до refinement, Q10).
- **T-shirt sizes (XS–XL)** — найгрубша шкала для раннього етапу: оцінити епіки (великі шматки роботи, що згодом розпадуться на десятки задач) на квартал уперед, коли деталей ще нема і поінти вдавали б точність, якої не існує. Розмір «L» каже достатньо для пріоритезації: «це велике, або ріжемо, або свідомо інвестуємо».

Це фактично драбина за рівнем відомого: **T-shirt** на далекому горизонті (roadmap, епіки) → **Fibonacci points** для беклогу і спринтів (задачі обговорені на refinement) → **години** хіба що для декомпозованих підзадач усередині спринту. Рухатися шкалою «точніше», ніж дозволяє реальне знання про задачу, — головна помилка оцінювання.

І про механіку прогнозу: поінти конвертуються в терміни не формулою, а **емпірично** через velocity (Q9) — команда фактично закриває ~30 поінтів за спринт, отже беклог на 90 поінтів — це орієнтовно три спринти. Оцінка калібрується самою історією команди, а не чиїмось відчуттям часу.

**Підсумок-якір:** Драбина точності за рівнем знання: T-shirt (XS–XL) для епіків → Fibonacci-поінти для беклогу і спринтів → години хіба для декомпозованих підзадач. 13+ поінтів = різати задачу; у дати — тільки через velocity.

**➕ Пов'язане: що таке Planning Poker? 🔵**
Техніка, якою команда реально ставить ті самі Фібоначчі-оцінки. Задачу обговорили → кожен **мовчки** обирає картку зі своєю оцінкою → всі відкривають **одночасно**. Одночасність — не ритуал, а захист від якоря: якщо сеньйор першим скаже «та тут 2», решта несвідомо підтягнеться, і незалежних оцінок не буде. Найцінніший момент — розбіжність: у когось 2, у когось 13 — отже, люди бачать різні задачі («а ти врахував міграцію даних?»); коротке обговорення крайніх оцінок і повторне коло часто виявляють приховану складність або зайвий страх. Тобто покер — це не так про цифру, як про **синхронізацію розуміння задачі**.

**➕ Пов'язане: чому story points не переводять напряму в години? 🟢**
Спокуса зрозуміла: «скажіть уже, 1 поінт — це скільки годин?» Але фіксований курс «1 поінт = X годин» ламає систему двічі. По-перше, він фактично повертає оцінку в години — з усім систематичним оптимізмом, від якого поінти і втікали; відносна шкала працює саме тому, що не прив'язана до часу. По-друге, у різних людей і команд той самий поінт займає різний час — і це нормально, бо прогноз робиться через **власну** velocity команди (скільки поінтів вона реально закриває за спринт), яка вже враховує і швидкість людей, і мітинги, і рев'ю. Потрібні дати — рахуй через velocity, а не через курс обміну.
