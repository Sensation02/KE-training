## Q1. Що таке interceptors, transient requests і global error handling? 🔵

**Interceptor** — проміжний обробник, через який проходить кожен HTTP-запит і/або кожна відповідь. Дозволяє в одному місці робити те, що інакше довелося б повторювати в кожному виклику: додавати заголовки (токен, тенант) до запитів і обробляти помилки у відповідях.

Розрізняють два типи:
- **request interceptor** — виконується перед відправкою (додати `Authorization`, спільні заголовки);
- **response interceptor** — виконується після отримання відповіді (централізовано зловити `401`, `500`).

На нативному `fetch` обидва типи реалізують тонкою обгорткою:

```ts
async function apiFetch<T>(path: string, options: RequestInit = {}): Promise<T> {
  // ── request interceptor: спільні заголовки в одному місці ──
  const headers = new Headers(options.headers);
  headers.set("Content-Type", "application/json");
  const token = getAccessToken();
  if (token) headers.set("Authorization", `Bearer ${token}`);

  const res = await fetch(`/api${path}`, { ...options, headers });

  // ── response interceptor: реакція на статус у одному місці ──
  if (res.status === 401) { redirectToLogin(); throw new Error("Unauthorized"); }
  if (!res.ok) throw new Error((await res.json().catch(() => null))?.message ?? "Request failed");

  return res.status === 204 ? (undefined as T) : ((await res.json()) as T);
}
```

**Transient request** — разовий, «прохідний» запит, стан і результат якого не зберігаються й не кешуються (напр. подія аналітики). Найлегше зрозуміти через контраст зі звичайним запитом:

```ts
// ЗВИЧАЙНИЙ запит: чекаємо результат (await), використовуємо його, помилка важлива
const user = await apiFetch<User>("/me"); // await → результат потрібен далі
renderProfile(user);                      // дані йдуть у UI / кеш

// TRANSIENT запит: «надіслати й забути» — результату не чекаємо, не зберігаємо
function trackEvent(name: string, payload: unknown) {
  void apiFetch("/analytics", { method: "POST", body: JSON.stringify({ name, payload }) })
    .catch(() => {}); // подія аналітики некритична — помилку свідомо ковтаємо
}
```
Різниця не в самому запиті, а в тому, **що ми робимо з результатом**: звичайний — чекаємо (`await`) і використовуємо; transient — відпускаємо (без `await`, без збереження), бо він некритичний.

**Global error handling** — єдина точка, де будь-яка помилка запиту перетворюється на зрозумілу реакцію (тост, редірект на логін), щоб помилки не оброблялися вроздріб і не губилися. Працює у **два шари**: транспортний — усередині `apiFetch` вище (`401 → редірект`, `!ok → throw`); і застосунковий — глобальний `onError` у React Query, що показує тост для будь-якого запиту, який впав:

```ts
const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError: (err) => toast.error(err.message), // одна реакція на всі помилки запитів
  }),
});
```

**➕ Пов'язане: чим `Promise.reject` в interceptor кращий за проковтування помилки? 🔵**
Якщо response-interceptor «зловив» помилку і нічого не повернув, код, що викликав запит, вважатиме його успішним і працюватиме з `undefined`. Тому в interceptor помилку **перекидають далі** (`throw` / `Promise.reject`), щоб її обробив `try/catch` або `onError` у React Query — а не ковтають (виняток — свідомо transient-запити, як вище):

```ts
// ❌ проковтнули: виклик «успішний», але data === undefined → прихований баг
async function bad() { try { return await apiFetch<User>("/me"); } catch { /* тиша */ } }

// ✅ перекинули далі: помилка спливає туди, де її реально обробляють
async function good() { return apiFetch<User>("/me"); } // throw дійде до try/catch чи onError
```

## Q2. Як з нуля підключити UI-бібліотеку (shadcn / Mantine / Ant Design)? 🟣

«З нуля» = встановити, обгорнути застосунок у провайдер теми, задати дизайн-токени (кольори, радіуси, шрифти) централізовано, переконатися, що стилі підвантажуються. Треба розуміти, що три популярні варіанти влаштовані **принципово по-різному**:

- **Ant Design** — пакет готових компонентів; тема через `ConfigProvider` (токени). Стилі з коробки (CSS-in-JS у v5).
- **Mantine** — пакет компонентів + `MantineProvider`; тема об'єктом; треба імпортувати CSS.
- **shadcn/ui** — **не пакет**, а компоненти, які копіюються у твій код (на Radix + Tailwind); стилями володієш повністю, оновлюєш вручну.

Звідси й два різні флоу «з нуля»: **пакетні** (Ant, Mantine) — `npm install` + обгорнути в провайдер теми; **CLI-копіювання** (shadcn) — не встановлюєш пакет, а генеруєш компоненти у свій код.

**Ant Design** (пакет, стилі з коробки):
```tsx
// npm i antd
import { ConfigProvider, theme } from "antd";

<ConfigProvider theme={{
  token: { colorPrimary: "#0e7c86", borderRadius: 8 }, // дизайн-токени
  algorithm: theme.defaultAlgorithm,                    // або darkAlgorithm
}}>
  <App />
</ConfigProvider>
```

**Mantine** (пакет, але CSS треба імпортувати вручну):
```tsx
// npm i @mantine/core @mantine/hooks
import { MantineProvider, createTheme } from "@mantine/core";
import "@mantine/core/styles.css"; // без цього стилі не застосуються

const t = createTheme({ primaryColor: "teal", defaultRadius: "md" });
<MantineProvider theme={t}><App /></MantineProvider>
```

**shadcn/ui** — не пакет: CLI ініціалізує проєкт і копіює код компонентів до тебе (далі вони твої, оновлюєш вручну):
```bash
npx shadcn@latest init        # одноразово: підключає Tailwind, шляхи, CSS-змінні теми
npx shadcn@latest add button  # копіює вихідний код <Button> у твій репозиторій
```
Тема — через CSS-змінні (не провайдер), бо компоненти вже у твоєму коді:
```css
:root { --primary: 200 80% 30%; --radius: 0.5rem; } /* HSL-токени */
```

**Як налаштовується сам Tailwind** (фундамент shadcn): встановлюють пакет із Vite-плагіном (`npm i tailwindcss @tailwindcss/vite`), реєструють плагін у `vite.config.ts` і одним рядком `@import "tailwindcss"` підключають усі утиліти в головному CSS. Кастомні дизайн-токени (кольори, радіуси, шрифти) у Tailwind v4 задають директивою `@theme` прямо в CSS — без окремого `tailwind.config.js`, як це було у v3.
```ts
// vite.config.ts — реєстрація плагіна
import tailwindcss from "@tailwindcss/vite";
export default { plugins: [tailwindcss()] };
```
```css
/* головний CSS: підключення Tailwind + власні токени */
@import "tailwindcss";
@theme { --color-primary: #0e7c86; --radius-md: 0.5rem; } /* → утиліти bg-primary, rounded-md */
```

**Головний принцип:** тема живе в токенах (об'єкт або CSS-змінні), а не хардкодиться в кожному компоненті — тоді змінити бренд = змінити токени.

**➕ Пов'язане: чим CSS-in-JS відрізняється від utility-first (Tailwind)? 🔵**
CSS-in-JS (styled-components, Emotion, Ant v5) генерує стилі з JS у рантаймі — динамічно за пропсами, але з накладними витратами на виконання. Utility-first (Tailwind) — готові атомарні класи (`px-4 text-sm`), стилі рахуються на білді, рантайму майже нема; звідси менший bundle і швидкість, ціною «довгих» `className`. shadcn спирається на Tailwind саме тому.

**➕ Пов'язане: що таке headless (unstyled) бібліотека? 🟢**
Headless-бібліотека (Radix UI, Headless UI) дає **поведінку й доступність** (фокус, клавіатура, ARIA) без візуальних стилів — стилі ти додаєш сам. Це дозволяє мати надійну логіку компонента і повний контроль над виглядом; на Radix і побудований shadcn/ui.

## Q3. Як обирати npm-пакет для UI-задачі і контролювати вплив на bundle? 🟣

Перед додаванням залежності оцінюють: **розмір**, чи **tree-shakeable** (можна взяти частину, а не всю бібліотеку), кількість під-залежностей, активність підтримки. Один пакет може додати сотні кілобайт до фінального білду й уповільнити завантаження.

**Перевірка розміру** — [bundlephobia.com](https://bundlephobia.com): показує вагу пакета (minified + gzip) і чи він tree-shakeable.

**Tree-shaking — погано vs добре:**
```ts
// ❌ moment ~70kB, НЕ tree-shakeable — уся бібліотека потрапляє в bundle
import moment from "moment";

// ✅ date-fns: у bundle піде лише імпортована функція
import { format } from "date-fns";
```

**Bundle analyzer** — інструмент, що будує карту фінального білду: який пакет скільки важить. Для Vite:
```ts
// vite.config.ts
import { visualizer } from "rollup-plugin-visualizer";
export default { plugins: [visualizer({ open: true, gzipSize: true })] };
```
Після `npm run build` відкривається діаграма, де одразу видно «важкі» залежності — кандидати на заміну чи lazy-load.

**➕ Пов'язане: чим tree-shaking відрізняється від code-splitting? 🔵**
Це різні речі, які часто плутають. **Tree-shaking** — видалення *невикористаного* коду з bundle на етапі білду (взяв одну функцію з lodash — решта не потрапить). **Code-splitting** — розбиття bundle на окремі частини (chunks), що вантажаться на вимогу (код адмінки — окремим файлом). Перше зменшує загальний розмір, друге відкладає завантаження частин.

**➕ Пов'язане: що таке `"sideEffects": false` у package.json? 🟢**
Це прапорець, яким пакет повідомляє bundler'у, що його модулі не мають побічних ефектів при імпорті — тоді tree-shaking може сміливо викидати невикористані. Якщо пакет цього не вказує (або має глобальні імпорти CSS), bundler боїться щось видалити, і tree-shaking працює гірше.

**Чи немає негативних ефектів?** Є — якщо збрехати. Коли модуль насправді *має* побічний ефект (глобальний `import "./styles.css"`, полифіл, реєстрація web-component, патч прототипу), а ти вказав `false`, bundler вважатиме такий імпорт «невикористаним» і викине його — тоді зникнуть стилі або зламається рантайм, причому лише в production-білді (у dev tree-shaking не працює, тож баг не видно). Тому `false` став лише коли впевнений, що ефектів немає; для часткових ефектів вказують не `false`, а список винятків:
```json
{ "sideEffects": ["*.css", "*.scss"] }
```

## Q4. Які є state-менеджери (Redux / React Query / Context / Zustand) і як обрати? 🟣

Вибір залежить від **типу стану** — це головна ідея, а не «який менеджер модніший».

| Інструмент | Тип стану | Коли брати |
|---|---|---|
| **React Query (TanStack)** | Серверний (дані з API) | Майже завжди для роботи з бекендом: кеш, рефетч, інвалідація |
| **Zustand** | Клієнтський глобальний, простий | Глобальний UI-стан без бойлерплейту |
| **React Context** | Рідкозмінний глобальний | Тема, поточний користувач |
| **Redux Toolkit** | Складний клієнтський, суворі правила | Велика команда, складні потоки, потрібні dev-tools |

**З чого починати — порядок рішення** (від найчастішого до рідкісного):
1. **Дані з бекенду?** → **React Query**. Це основна маса стану в типовому застосунку — не тягни його в Redux/Context вручну.
2. **Стан потрібен лише одному компоненту (чи кільком поруч)?** → звичайний `useState`/`useReducer`, без жодного менеджера.
3. **Глобальний, але рідкозмінний** (тема, мова, поточний користувач)? → **Context**.
4. **Глобальний і часто змінюваний** (UI: модалки, сайдбар, тост-черга)? → **Zustand** — мінімум бойлерплейту, селектори не спричиняють зайвих ререндерів.
5. **Складні потоки, сувора структура, велика команда, потрібні time-travel dev-tools?** → **Redux Toolkit**.

**Дефолт:** React Query для серверного стану + `useState`/Zustand для клієнтського закривають майже все. Redux бери свідомо, коли переріс простіше рішення, а не «на всяк випадок» — інакше отримаєш бойлерплейт без вигоди.

Часта помилка — тримати **серверні дані** (відповіді API) вручну в Redux/Context. Це робота React Query: він кешує, стежить за `isLoading`/`error` і оновлює дані.

**Серверний стан — React Query:**
```tsx
const { data, isLoading, error } = useQuery({
  queryKey: ["courses"],
  queryFn: () => api.get<Course[]>("/courses"),
});
```

**Клієнтський глобальний стан — Zustand:**
```ts
import { create } from "zustand";
export const useUi = create<{ sidebarOpen: boolean; toggle: () => void }>((set) => ({
  sidebarOpen: false,
  toggle: () => set((s) => ({ sidebarOpen: !s.sidebarOpen })),
}));
```

**Рідкозмінний глобальний — Context:**
```tsx
const UserCtx = createContext<User | null>(null);
const user = useContext(UserCtx);
```

**➕ Пов'язане: навіщо `queryKey` і як працює інвалідація кешу? 🔵**
`queryKey` — унікальний ключ кешу запиту (`["courses", expertId]`). React Query за ним зберігає й знаходить дані. `invalidateQueries` — метод **глобального** `queryClient`, тому не важливо, *звідки* ти його викликаєш (типово — з мутації після зміни даних): значення має лише ключ, а не місце виклику.
```ts
const { mutate } = useMutation({
  mutationFn: createCourse,
  onSuccess: () => {
    // оновить список курсів, хоча useQuery(["courses"]) живе зовсім в іншому компоненті
    queryClient.invalidateQueries({ queryKey: ["courses"] });
  },
});
```
Два нюанси, які часто спрощують:
- **Матчинг за префіксом:** `["courses"]` збігається з *усіма* ключами, що з нього починаються — одним викликом оновлюєш і загальний список, і всі його варіанти:
```ts
// інвалідує ["courses"], ["courses", expertId], ["courses", { filter: "active" }] — усе одразу
queryClient.invalidateQueries({ queryKey: ["courses"] });
```
- **Негайний refetch — лише для активних запитів.** Інвалідація *завжди* позначає збіги як застарілі (`stale`), але одразу перезавантажує тільки ті, що зараз на екрані (є змонтований `useQuery`, який їх слухає). Неактивні перезавантажаться при наступному монтуванні чи поверненні фокусу — тому «оновиться» не завжди означає «цієї ж секунди».

**➕ Пов'язане: що таке optimistic update? 🔵**
Це коли UI оновлюється **одразу**, ще до відповіді сервера (напр. лайк ставиться миттєво), а якщо запит впав — зміну відкочують. Дає відчуття миттєвості. У React Query робиться через `onMutate` (застосувати наперед) + `onError` (відкотити).

**➕ Пов'язане: коли Context — погана заміна стор-менеджеру? 🟢**
Context не оптимізований для частих оновлень: будь-яка зміна значення провайдера ререндерить **усіх** споживачів. Для стану, що часто змінюється (позиція курсора, поле вводу), Context спричинить лавину ререндерів — тут краще Zustand/Redux із селекторами (компонент підписаний лише на свій шматок стану).

**Чому лавина ререндерів відчутна?** Один ререндер дешевий — проблема в їх кількості × частоті. JS виконується в одному потоці (main thread), і в ньому ж браузер малює кадри (ціль ~60 fps). Поки React перераховує сотні компонентів на кожну дрібну зміну, браузер не встигає намалювати наступний кадр — падає FPS, звідси лаги вводу, «желейний» скрол і смикані анімації. Ререндер сам по собі не «думання логіки», а перерахунок того, *що показати*; страшна саме лавина, а не окремий ререндер.

## Q5. Чим відрізняються localStorage, sessionStorage і cookies (обмеження, безпека)? 🔵

Три способи зберігати дані в браузері з різним часом життя й різними правилами безпеки.

| | localStorage | sessionStorage | Cookies |
|---|---|---|---|
| Час життя | Поки не видалиш | До закриття вкладки | За `Expires`/`Max-Age` |
| Розмір | ~5–10 MB | ~5 MB | ~4 KB |
| Йде на сервер? | Ні | Ні | **Так**, автоматично з кожним запитом |
| Доступ з JS | Так | Так | Ні, якщо `HttpOnly` |

**Безпека:** localStorage і sessionStorage доступні будь-якому JS на сторінці → вразливі до **XSS** (Cross-Site Scripting — коли на твою сторінку впроваджується й виконується чужий JavaScript). Токени доступу тому безпечніше тримати в **`HttpOnly` + `Secure` cookie**: JS до неї не має доступу, отже XSS її не вкраде. Cookie також автоматично шлеться на сервер (звідси її роль у сесіях), тоді як localStorage — суто клієнтський.

**Що обирати на практиці:**
- **localStorage** — несекретні клієнтські налаштування, що мають пережити перезавантаження (тема, мова, чернетка форми).
- **sessionStorage** — тимчасовий стан однієї вкладки, який не шкода втратити (крок візарда, разові фільтри).
- **`HttpOnly` cookie** — усе, що стосується автентифікації (токен сесії): ставить і читає **сервер**, клієнтський JS не втручається.
- **Антипатерн:** access/refresh-токени в localStorage — часта L3-помилка: перший же XSS їх викраде. Токен → лише `HttpOnly` cookie.

**Як це реалізується** (типовий поділ ролей клієнт/сервер):
```ts
// КЛІЄНТ — лише несекретне:
localStorage.setItem("theme", "dark");        // переживе перезавантаження
sessionStorage.setItem("wizardStep", "2");    // зникне із закриттям вкладки

// СЕРВЕР — автентифікація через cookie, якої JS не бачить:
// Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict
// далі браузер САМ додає цю cookie до кожного запиту на цей домен —
// на клієнті токеном керувати не треба (і не можна).
```

**➕ Пов'язане: що таке `SameSite` і як він захищає від CSRF? 🔵**
`SameSite` — атрибут cookie, що керує, чи шлеться вона на крос-сайтові запити. `Strict` — лише зі свого сайту; `Lax` — свій сайт + переходи за посиланням; `None` — завжди (потребує `Secure`). Значення `Strict`/`Lax` блокують автоматичне надсилання cookie при запиті з чужого сайту — а саме на цьому будується атака CSRF.

**➕ Пов'язане: що таке XSS і CSRF та чим вони відрізняються? 🔵**
**XSS (Cross-Site Scripting)** — зловмиснику вдається виконати свій JavaScript на твоїй сторінці (класично — через неекранований користувацький ввід, що потрапляє в DOM). Далі його скрипт краде дані з localStorage, читає токени, діє від імені користувача.
**CSRF (Cross-Site Request Forgery)** — зловмисний сайт змушує браузер жертви надіслати запит на твій API, скориставшись уже наявною cookie-сесією (браузер автоматично додає cookie до запиту). Жертва нічого не робить свідомо — лише відкрила чужу сторінку, поки залогінена в тебе.
**Ключова різниця:** XSS виконує *чужий код у тебе*, CSRF шле *запит від твого імені без коду*. Захист теж різний: від XSS — екранування виводу + CSP + `HttpOnly` (щоб скрипт не дістав токен); від CSRF — `SameSite`-cookie + CSRF-токени.

## Q6. Як ефективно дебажити і профілювати фронтенд у браузері? 🟣

Мета — **вимірювати** повільні місця через DevTools, а не гадати; дебажити breakpoints, а не `console.log`.

**Інструменти Chrome DevTools:**
- **Sources → breakpoints** — зупинити код у рядку, дивитися значення змінних, call stack.
- **Performance** — записати взаємодію, побачити довгі задачі, layout thrashing, зайві ререндери.
- **Network** — які запити повільні (waterfall, розмір, час).
- **React Profiler** (у React DevTools) — які компоненти ререндеряться і чому.

**Умовний breakpoint** (не спиняється на кожній ітерації): правий клік на breakpoint → *Add conditional breakpoint* → напр. `user.id === 42`.

**Пошук зайвих ререндерів** — часта причина лагів: нове посилання на функцію/об'єкт у пропсах щорендер.
```tsx
const handleClick = useCallback(() => doThing(id), [id]); // стабільне посилання
const Row = React.memo(RowComponent);                     // не ререндерить, якщо пропси ті самі
```

**Профілювання (вкладка Performance) — як читати трейс:**
1. **Record** → виконати повільну взаємодію (клік, ввід, скрол) → **Stop**.
2. Дивишся доріжку **Main** (головний потік): суцільні блоки — це задачі JS.
3. **Long Task** (> 50 мс, помічена червоним кутиком) блокує потік — саме вона «морозить» UI: інпут не реагує, кадри пропускаються (падає FPS).
4. Клік на задачу → **flame chart** знизу показує стек викликів: яка саме функція з'їла час (твій код, ре-рендер чи бібліотека).
5. **Bottom-Up / Call Tree** — агрегує, де сумарно втрачено найбільше часу.

Читаєш згори вниз: довга задача → її flame chart → винна функція → оптимізуєш (memo/debounce, розбити роботу на частини або винести з main-потоку у Web Worker).

**➕ Пов'язане: що таке memory leak у SPA і як його знайти? 🔵**
Витік пам'яті — коли об'єкти не звільняються, бо на них лишились посилання (не прибраний `setInterval`, не відписаний event listener, не скасована підписка в `useEffect`). У SPA накопичується з часом і гальмує вкладку. Шукають у DevTools → **Memory**: робиш heap snapshot до і після взаємодії, дивишся, що росте. Профілактика — прибирати ефекти в cleanup-функції `useEffect`.

**➕ Пов'язане: що показує вкладка Coverage в DevTools? 🟢**
Coverage показує, який відсоток завантаженого JS/CSS реально виконався на сторінці. Багато «мертвого» коду — сигнал, що варто застосувати code-splitting і вантажити його на вимогу.

## Q7. Навіщо специфічні devtools під фреймворк (React / Angular DevTools)? 🟣

Це розширення браузера, що показують **внутрішній стан фреймворку**, невидимий у звичайному DOM-інспекторі: у вкладці Elements видно лише готовий HTML, а не компоненти, їхні `props`, `state` і `hooks`.

**React DevTools — дві вкладки:**

**1. Components** — дерево компонентів у реальному часі:
- значення `props`, `state` і всіх **hooks** (`useState`, `useContext`, `useReducer`) вибраного компонента;
- їх можна **редагувати наживо** — змінити state/проп прямо в панелі й одразу побачити реакцію UI (швидка перевірка гіпотез без правок коду);
- **«rendered by»** — ланцюг батьків: звідки прийшов компонент і хто передав пропси (рятує від «не розумію, звідки цей проп»);
- пошук за іменем + бейджі `Memo`/`ForwardRef`, а також який `Context` постачає значення.

**2. Profiler** — для продуктивності:
- записуєш взаємодію → бачиш кожен commit і **скільки мс** рендерився кожен компонент (flame graph);
- **«Why did this render?»** (увімкнути в налаштуваннях профайлера) — показує *причину* кожного ререндеру: `props changed`, `hook changed`, `parent rendered`. Це головний інструмент проти зайвих ререндерів;
- «Highlight updates» — підсвічує рамкою те, що перемальовується (видно, що мигтить дарма).

**Angular DevTools** (аналог для Angular): дерево компонентів, `@Input`/`@Output`, change-detection profiler.

**Практично — коли що брати:**
- «state не оновлюється як треба» → **Components**: дивишся реальне значення state/hooks наживо, без `console.log`;
- «лагає при вводі/кліку» → **Profiler**: записуєш, знаходиш компонент із довгим рендером чи зайвими ререндерами, дивишся `Why did this render?`;
- перевірити, що `React.memo`/`useMemo` реально працюють → «Highlight updates»: компонент не має мигтіти, коли його пропси не змінились.

```tsx
function Counter() {
  const [count, setCount] = useState(0); // count і його зміни видно в Components наживо
  return <button onClick={() => setCount((c) => c + 1)}>{count}</button>;
}
```
**Нюанс:** у production-білді імена компонентів часто мінімізовані — дерево читається гірше, тож дебаж на dev-білді. І пам'ятай межу: DevTools показує, *що* сталося; *чому* стався конкретний ререндер — відповідає саме `Why did this render?` у Profiler.

**➕ Пов'язане: що робить «Highlight updates» і навіщо? 🟢**
Опція React DevTools, що підсвічує рамкою кожен компонент у момент його ререндеру. Одразу видно зайві перемальовування (мигтить те, що не мало б оновлюватись) — швидкий спосіб знайти проблеми продуктивності без профайлера.

**➕ Пов'язане: що дає time-travel у Redux DevTools? 🟢**
Redux DevTools логує кожну dispatch-нуту дію і дозволяє «перемотувати» стан назад/вперед по діях. Так можна відтворити, як застосунок дійшов до багового стану, і побачити, яка саме дія його зіпсувала.

## Q8. Що таке semver і як опублікувати (у т.ч. приватний) npm-пакет? 🔵

**Semver** — формат версії `MAJOR.MINOR.PATCH`, де кожна частина має чіткий сенс:
- **MAJOR** (`2.0.0`) — ламкі зміни (breaking changes);
- **MINOR** (`1.3.0`) — нова функціональність, сумісна назад;
- **PATCH** (`1.2.4`) — виправлення багів.

Діапазони в `package.json`:
```json
{ "react": "^18.2.0",  // ^ дозволяє MINOR+PATCH: 18.x.x, але не 19
  "lodash": "~4.17.0" } // ~ дозволяє лише PATCH: 4.17.x
```

**Публікація:**
```bash
npm version patch      # підняти версію
npm publish            # для scoped-публічного: --access public
```

**Приватний пакет** — конфігурується через `.npmrc`, який каже npm, з якого реєстру брати пакети певного scope і яким токеном авторизуватись:
```
@myorg:registry=https://npm.pkg.github.com
//npm.pkg.github.com/:_authToken=${NPM_TOKEN}
```
`NPM_TOKEN` тримають у змінній середовища (локально або в CI), **не** в git.

**➕ Пов'язане: що таке `peerDependencies` і навіщо? 🔵**
`peerDependencies` кажуть: «цей пакет вимагає, щоб у проєкті вже був встановлений певний пакет певної версії, але сам його не тягне». Класика — плагіни до React: бібліотека компонентів ставить `react` у peer, щоб не було двох різних копій React у застосунку (це ламає хуки).

**➕ Пов'язане: чим `dependencies` відрізняються від `devDependencies`? 🟢**
`dependencies` потрібні застосунку в рантаймі (React, date-fns). `devDependencies` — лише для розробки/білду (типи, тест-раннер, bundler) і не потрапляють у продакшн-залежності споживача пакета.

**➕ Пов'язане: навіщо lock-файл (package-lock.json / yarn.lock)? 🔵**
Lock-файл фіксує **точні** версії всіх залежностей (включно з транзитивними), щоб `npm install` давав ідентичний результат у всіх і в CI. Без нього `^1.2.0` може підтягнути різні патчі на різних машинах → «в мене працює, у CI ні».

## Q9. Як проєктувати структуру компонентів (page / feature / library)? 🟣

Рівень компонента визначає **одне питання: наскільки він знає про домен**. Це і є критерій, куди його класти:

| Рівень | Куди класти | Приклад |
|---|---|---|
| **Page** | Прив'язаний до URL-роуту, тягне дані й збирає сторінку | `CoursePage` на `/courses/:id` |
| **Feature** | Містить доменну логіку однієї фічі | `CourseList`, `EnrollForm` |
| **Library / UI** | Жодних доменних знань, перевикористовується всюди | `Button`, `Modal`, `Input` |

**Правило залежностей — тільки згори вниз:** page → feature → ui, і **ніколи навпаки**. `Button` нічого не знає про курси (тому його можна взяти в будь-який проєкт); `CourseList` не знає, на якій сторінці його показують. Порушив напрямок (UI-примітив імпортує щось доменне) — і його вже не перевикористаєш.

**Feature-based структура** (кожна фіча володіє своїм кодом, спільне — окремо):
```
src/
  features/
    courses/
      pages/CoursePage.tsx       # page:    роут + дані
      components/CourseList.tsx   # feature: доменна логіка «курсів»
      hooks/useCourses.ts
      api/coursesApi.ts
  shared/
    ui/Button.tsx                # library: без домену, для всіх фіч
    ui/Modal.tsx
```

**Один потік даних крізь усі три рівні** (той самий приклад `courses` згори вниз):
```tsx
// PAGE — знає про роут і дані, сам нічого не малює
function CoursePage() {
  const { data } = useCourses();           // тягне дані
  return <CourseList courses={data} />;     // віддає у feature
}

// FEATURE — знає домен «курс», але не тягне дані й не займається стилями
function CourseList({ courses }: { courses: Course[] }) {
  return (
    <ul>
      {courses.map((c) => (
        <li key={c.id}>
          {c.title} <Button>Записатися</Button>  {/* бере готовий UI-примітив */}
        </li>
      ))}
    </ul>
  );
}

// LIBRARY/UI — про курси не чув, працює в будь-якому проєкті
function Button({ children, ...props }: React.ComponentProps<"button">) {
  return <button className="btn" {...props}>{children}</button>;
}
```
Дані спускаються `CoursePage → CourseList → Button`, а доменні знання не «протікають» униз: `Button` лишається чистим примітивом.

**➕ Пов'язане: що таке compound-компоненти? 🔵**
Compound-компоненти — набір компонентів, що працюють як єдине ціле й **неявно ділять стан** через Context, даючи гнучкий декларативний API. Класичний приклад із HTML — `<select>` і `<option>`: окремо безсенсу, разом — контрол. У React це, напр. `<Tabs>` з під-компонентами:
```tsx
const TabsCtx = createContext<{ active: string; setActive: (v: string) => void } | null>(null);

function Tabs({ defaultValue, children }: { defaultValue: string; children: React.ReactNode }) {
  const [active, setActive] = useState(defaultValue);
  return <TabsCtx.Provider value={{ active, setActive }}>{children}</TabsCtx.Provider>;
}
Tabs.Trigger = ({ value, children }: { value: string; children: React.ReactNode }) => {
  const ctx = useContext(TabsCtx)!;
  return <button aria-selected={ctx.active === value} onClick={() => ctx.setActive(value)}>{children}</button>;
};
Tabs.Panel = ({ value, children }: { value: string; children: React.ReactNode }) => {
  const ctx = useContext(TabsCtx)!;
  return ctx.active === value ? <div>{children}</div> : null;
};

// використання — читабельний, гнучкий API без «prop drilling»:
<Tabs defaultValue="a">
  <Tabs.Trigger value="a">Перша</Tabs.Trigger>
  <Tabs.Trigger value="b">Друга</Tabs.Trigger>
  <Tabs.Panel value="a">Вміст А</Tabs.Panel>
  <Tabs.Panel value="b">Вміст Б</Tabs.Panel>
</Tabs>
```
Перевага перед «одним великим компонентом із купою пропсів»: гнучкість композиції й чистий API. Так побудовані Radix і більшість компонентів shadcn/ui.

**➕ Пов'язане: коли застосовувати compound-компоненти (а коли ні)? 🔵**
Тест — брати, коли виконані **всі три** умови: (1) частини утворюють **одне ціле** (окремо безсенсні, як `<select>/<option>`); (2) мусять **ділити стан** (тригер знає активну панель); (3) споживачу потрібна **свобода композиції** (порядок, склад, верстка). Пропущена умова → є простіше рішення на «драбині»: `пропси` (фіксований вигляд) → `children`/слоти (вміст варіюється, стану не ділять — див. нижче) → **compound** (ділять стан + свобода структури) → headless-хук/render props (повний контроль над рендером кожного елемента). Реалізують через **Context** (дефолт — пробиває будь-яку вкладеність), а не `cloneElement`+`Children.map` (legacy — лише прямі діти, «магічні» пропси, кепська типізація). **НЕ бери**, коли структура фіксована (пропси простіші й безпечніші — композицію не зламати), нема спільного стану (досить `children`) або стан «гарячий»/часто змінюється (один Context → лавина ререндерів).

**➕ Пов'язане: чому композиція через `children` часто краща за пропси-конфіг? 🔵**
Замість «бога-компонента» з десятками пропсів (`<Card title icon footer actions .../>`) передають готові шматки через `children`/слоти (`<Card><Card.Header/>...</Card>`). Це гнучкіше (можна вставити будь-що), уникає «вибуху пропсів» і не змушує компонент знати про всі можливі варіанти вмісту.

## Q10. Як створювати перевикористовуваний код (компоненти, хуки, utils) і коли? 🟣

Виносять у спільний код те, що повторюється **і** стабілізувалось. Передчасна абстракція шкідлива — спершу побач повторення двічі-тричі. Сигнали до винесення: той самий шматок з'явився ≥3 рази; логіка не залежить від конкретної фічі; зміну хочеться робити в одному місці.

**Кастомний хук (перевикористання логіки):**
```ts
function useDebouncedValue<T>(value: T, delay = 300): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id); // скасувати попередній таймер
  }, [value, delay]);
  return debounced;
}
```

**Reusable-компонент (без прив'язки до фічі):**
```tsx
function Modal({ open, onClose, children }: ModalProps) {
  if (!open) return null;
  return (
    <div className="overlay" onClick={onClose}>
      <div className="modal" onClick={(e) => e.stopPropagation()}>{children}</div>
    </div>
  );
}
```

**➕ Пов'язане: чим `useMemo` відрізняється від `useCallback`? 🔵**
Обидва мемоізують між рендерами за масивом залежностей. `useMemo(fn, deps)` кешує **результат** обчислення (значення). `useCallback(fn, deps)` кешує **саму функцію** (стабільне посилання). Фактично `useCallback(fn, d)` === `useMemo(() => fn, d)`. Callback потрібен, коли передаєш функцію в мемоізований дочірній компонент або в залежності іншого хука.

**➕ Пов'язане: які головні правила хуків? 🔵**
Хуки викликають лише (1) на верхньому рівні компонента/іншого хука — **не** в умовах, циклах чи вкладених функціях, і (2) лише з React-функцій. Причина: React зіставляє хуки за порядком виклику; умовний виклик збиває цей порядок і ламає стан.

**➕ Пов'язане: коли HOC, а коли хук? 🟢**
HOC (компонент-обгортка) і render-props — старіші патерни перевикористання логіки. Сьогодні для більшості випадків простіший і читабельніший **кастомний хук** (без «пекла обгорток» у дереві). HOC лишають хіба для наскрізних задач на рівні компонента (напр. обгортка помилок).

## Q11. Які бувають типи state і як пропагувати дані між компонентами? 🔵

**Типи стану:**
- **Component (local)** — `useState`, живе в одному компоненті;
- **Global** — доступний багатьом (користувач, тема);
- **Server** — дані з API → React Query;
- **Browser / URL** — параметри URL, localStorage.

**Методи пропагації:**
- **Top-to-bottom** — дані течуть згори вниз через props або **React Context**;
- **Event injection** — компонент диспатчить подію, стор (**Redux/Zustand**) оновлюється, підписані компоненти реагують.

**Top-to-bottom через Context (без прокидання пропсів крізь рівні):**
```tsx
const UserCtx = createContext<User | null>(null);
function App() {
  const [user] = useState<User | null>(current);
  return <UserCtx.Provider value={user}><Header /></UserCtx.Provider>;
}
function Header() {
  const user = useContext(UserCtx); // дістає напряму
  return <span>{user?.name}</span>;
}
```

**➕ Пов'язане: що таке prop drilling і як його уникнути? 🔵**
Prop drilling — коли пропс проштовхують крізь багато проміжних компонентів, яким він сам не потрібен, лише щоб донести його вниз. Уникають через **Context** (для рідкозмінного глобального) або через **композицію** (передати готовий елемент як `children`, а не дані крізь рівні).

**➕ Пов'язане: чому надмірний Context спричиняє зайві ререндери? 🔵**
Будь-яка зміна значення провайдера ререндерить **усіх** споживачів цього Context, навіть тих, кому потрібна лише незмінна частина. Рішення: ділити на кілька вузьких Context, виносити частозмінне у стор із селекторами, і стабілізувати `value` через `useMemo`.

## Q12. Як інкапсулювати і структурувати state (feature-based)? 🟣

Стан тримають близько до фічі, якій він належить, а не в одному гігантському глобальному об'єкті. Кожна фіча володіє своїм шматком стану — це зменшує зв'язність: зміни в одній фічі не зачіпають інші.

**Стор на фічу.** Сам патерн не прив'язаний до конкретного менеджера — фіча володіє своїм стором, байдуже Zustand, Redux Toolkit чи щось інше.

Zustand — окремий стор на фічу, мінімум коду:
```ts
// features/cart/cartStore.ts
import { create } from "zustand";
export const useCartStore = create<{
  items: CartItem[];
  add: (i: CartItem) => void;
  clear: () => void;
}>((set) => ({
  items: [],
  add: (i) => set((s) => ({ items: [...s.items, i] })),
  clear: () => set({ items: [] }),
}));
```
Той самий шматок стану на Redux Toolkit — slice, який глобальний стор потім збирає разом з іншими фічами:
```ts
// features/cart/cartSlice.ts
const cartSlice = createSlice({
  name: "cart",
  initialState: { items: [] as CartItem[] },
  reducers: {
    add: (state, action) => { state.items.push(action.payload); },
    clear: (state) => { state.items = []; },
  },
});
export const { add, clear } = cartSlice.actions;
export default cartSlice.reducer;
```
В обох випадках логіка стану живе **поруч із фічею**, а не в одному гігантському глобальному об'єкті.

**Інкапсуляція = сховати реалізацію за публічним API.** «Структурувати» — це провести межу: що фіча дає назовні і що ховає всередині. Назовні вона виставляє **лише хук** (через barrel-файл `index.ts`), а конкретний стор (Zustand-хук чи Redux slice+селектори) не «світиться» — тому решта застосунку не знає й не має знати, що саме під капотом:
```ts
// features/cart/index.ts — ПУБЛІЧНИЙ API фічі (тільки це імпортують ззовні)
export { useCart } from "./useCart";
// cartStore / cartSlice, селектори назовні НЕ експортуються

// features/cart/useCart.ts — тонка обгортка, що ховає джерело даних
export function useCart() {
  const items = useCartStore((s) => s.items);   // Zustand: беремо потрібне селектором
  const add = useCartStore((s) => s.add);
  return { items, add };
  // (на Redux тут було б useSelector(selectCartItems) + useDispatch — API хука той самий)
}

// споживач знає лише хук, а не стор під ним:
const { items, add } = useCart();  // заміниш Zustand ⇄ Redux — цей виклик не зміниться
```
Саме тому інкапсуляція цінна: внутрішню реалізацію фічі можна переписати (Zustand ⇄ Redux), не зачепивши жоден компонент, що її використовує.

**Що саме покласти в стор фічі — увесь спектр.** Оскільки реалізація схована за `useCart`, її обирають за потребою фічі, від найпростішого до найважчого:

| Всередині `useCart` | Коли брати |
|---|---|
| `useState` / `useReducer` (локально) | стан лише в піддереві однієї сторінки фічі — стор не потрібен |
| `useContext` (+ `useReducer`) | ділити в піддереві, змінюється рідко/помірно, без бібліотеки |
| Zustand-хук | частозмінний стан або потрібні селектори проти зайвих ререндерів |
| Redux Toolkit | велика команда, суворі правила, потрібні dev-tools |

**Context + `useReducer`** — легкий «стор фічі» без бібліотек: стан роздається через Context, а публічний `useCart` лишається той самий, тож для компонентів нічого не змінюється:
```tsx
const CartCtx = createContext<CartApi | null>(null);

export function CartProvider({ children }: { children: React.ReactNode }) {
  const [items, dispatch] = useReducer(cartReducer, []);
  // useMemo обов'язковий: інакше новий об'єкт щорендер ререндерить усіх споживачів
  const value = useMemo(() => ({ items, add: (i: CartItem) => dispatch({ type: "add", i }) }), [items]);
  return <CartCtx.Provider value={value}>{children}</CartCtx.Provider>;
}

export function useCart() {                 // той самий публічний хук — джерело просто інше
  const ctx = useContext(CartCtx);
  if (!ctx) throw new Error("useCart must be used within CartProvider");
  return ctx;
}
```
Застереження: Context ререндерить **усіх** споживачів при будь-якій зміні `value` (селекторів немає) — тому для частозмінного стану беруть Zustand, а не Context.

**➕ Пов'язане: навіщо мемоізовані селектори (`createSelector`)? 🔵**
Селектор — функція, що дістає похідні дані зі стору (напр. «активні курси»). Без мемоізації вона перераховує результат на кожен рендер і повертає новий масив → зайві ререндери. `createSelector` (Reselect, вбудований у RTK) кешує результат, поки вхідні дані не змінились.

**➕ Пов'язане: що таке нормалізація state і навіщо? 🟢**
Нормалізація — зберігати сутності як мапу за id (`{ [id]: course }`), а не вкладеними масивами. Так оновлення однієї сутності не змушує копіювати глибокі структури, зникає дублювання даних і спрощується пошук. Підхід із Redux (як у нормалізованих БД).

## Q13. Що таке generic (dumb) компоненти і як обрати рівень зберігання стану? 🟣

Дві ідеї: (1) розділяти «дурні» презентаційні компоненти й «розумні» контейнерні; (2) тримати стан на **найнижчому спільному рівні** для тих, кому він потрібен.

**Generic — приймає поведінку через пропси, не знає про домен:**
```tsx
// ❌ з доменною логікою — не перевикористаєш
function Button() {
  const dispatch = useDispatch();
  return <button onClick={() => dispatch(enroll())}>Записатись</button>;
}
// ✅ generic — дію передають ззовні
function Button({ onClick, children }: { onClick: () => void; children: React.ReactNode }) {
  return <button className="btn" onClick={onClick}>{children}</button>;
}
```

**Рівень стану (lift up / keep down):**
- потрібен одному компоненту → тримай локально;
- потрібен кільком сусіднім → підніми до найближчого спільного предка;
- не піднімай вище, ніж треба (зайвий глобальний стан = зайві ререндери).
```tsx
function SearchBox() {                 // стан потрібен і полю, і кнопці → у спільного батька
  const [q, setQ] = useState("");
  return (<>
    <input value={q} onChange={(e) => setQ(e.target.value)} />
    <button disabled={!q}>Шукати</button>
  </>);
}
```

**➕ Пов'язане: чим controlled компонент відрізняється від uncontrolled? 🔵**
**Controlled** — значення поля тримає React-стан (`value` + `onChange`); джерело правди — стан. **Uncontrolled** — значення живе в самому DOM, читаєш його через `ref` при потребі. Controlled дає повний контроль (валідація, форматування на льоту); uncontrolled простіший для великих форм без миттєвої реакції.

**➕ Пов'язане: що таке колокація стану (state colocation)? 🟢**
Принцип: тримати стан якомога ближче до місця, де він використовується, і піднімати лише за потреби. Це протилежність звички «усе в глобальний стор» — зменшує зайві ререндери й спрощує розуміння, хто чим володіє.

## Q14. Як оптимізувати великий UI-застосунок? 🟣

Три головні важелі: **lazy-loading** (вантажити код на вимогу), **code-splitting** (розбиття bundle на chunks), правильна **модуляризація**.

**Lazy-loading роуту** — код сторінки вантажиться лише коли її відкрили:
```tsx
import { lazy, Suspense } from "react";
const AdminPage = lazy(() => import("./features/admin/AdminPage"));

<Suspense fallback={<Spinner />}>
  <AdminPage />
</Suspense>
```
Стартовий bundle стає меншим — користувач не тягне код усіх сторінок одразу.

**Code-splitting важкої залежності:**
```tsx
const Chart = lazy(() => import("./HeavyChart")); // важкий чарт — окремим chunk на вимогу
```

**Мемоізація великих списків** (проти зайвих ререндерів):
```tsx
const List = React.memo(function List({ items }: { items: Item[] }) {
  return <>{items.map((i) => <Row key={i.id} {...i} />)}</>;
});
```

**Модуляризація** — код по фічах із чіткими межами дозволяє bundler'у ефективно ділити chunks, а зміни лишаються локальними.

**➕ Пов'язане: що таке віртуалізація списків? 🔵**
Коли в списку тисячі рядків, рендерити всі в DOM — повільно. Віртуалізація (react-window, TanStack Virtual) малює лише ті рядки, що зараз у вікні перегляду, а решту «симулює» висотою. DOM лишається маленьким незалежно від кількості даних.

**➕ Пов'язане: чим `React.memo` відрізняється від `useMemo`? 🔵**
`React.memo` — обгортка над **компонентом**: пропускає ререндер, якщо пропси не змінились. `useMemo` — хук, що кешує **значення** обчислення всередині компонента. Перше про «чи рендерити компонент», друге про «чи перераховувати дані».

**➕ Пов'язане: навіщо debounce/throttle у важких обробниках? 🟢**
`debounce` відкладає виклик до паузи в подіях (пошук — слати запит після того, як користувач допечатав), `throttle` обмежує частоту (обробник scroll/resize — не частіше ніж раз на N мс). Обидва рятують від зайвих дорогих викликів на потоці частих подій.

## Q15. Що таке unit-тестування (mocks, fakes, shallow/mount, pre/post-conditions)? 🟢

Unit-тест перевіряє **один маленький шматок** коду (функцію/компонент) ізольовано від решти системи.

Ключові поняття:
- **Mock** — підробка залежності, на якій перевіряють *виклики* («чи викликали API рівно раз»);
- **Fake** — спрощена робоча реалізація (in-memory сховище замість справжньої БД);
- **Shallow render** — рендер компонента без дочірніх (ізольовано);
- **Mount** — повний рендер з дітьми;
- **Pre-condition** — що істинне *до* виконання; **post-condition** — *після*.

```ts
import { describe, it, expect, vi } from "vitest";

function discount(price: number, percent: number): number {
  return price - (price * percent) / 100;
}
describe("discount", () => {
  it("рахує знижку", () => {
    // pre-condition: price=200, percent=10
    expect(discount(200, 10)).toBe(180); // post-condition: 180
  });
});

const api = { save: vi.fn() };            // mock
api.save({ id: 1 });
expect(api.save).toHaveBeenCalledOnce();  // перевірка виклику
```

**➕ Пов'язане: що таке патерн AAA (Arrange-Act-Assert)? 🟢**
Структура тесту з трьох частин: **Arrange** — підготувати дані й залежності; **Act** — виконати дію, яку тестуємо; **Assert** — перевірити результат. Тримає тести читабельними й однотипними.

**➕ Пов'язане: що таке code coverage і чи 100% — мета? 🟢**
Coverage — відсоток коду, який виконали тести. Корисний як індикатор «білих плям», але 100% не є самоціллю: можна мати повне покриття без жодної осмисленої перевірки. Важливіше вкрити критичну логіку, ніж ганятись за цифрою.

## Q16. Як писати код з високою тестованістю (чисті та імутабельні функції)? 🔵

Код легко тестувати, коли він **передбачуваний**: чисті функції (без побічних ефектів) і незмінність даних. Бізнес-логіку виносять у такі функції — їх можна перевіряти без рендеру, моків і бекенду.

**Чиста функція** — той самий вхід завжди дає той самий вихід, нічого зовні не змінює:
```ts
// ❌ нечиста: залежить від зовнішнього стану і мутує його
let total = 0;
function addToTotal(x: number) { total += x; return total; }

// ✅ чиста: вхід/вихід — і все
function add(a: number, b: number): number { return a + b; }
```

**Імутабельність** — не міняти вхідні дані, а повертати нові (React також покладається на це, щоб помічати зміни):
```ts
// ❌ мутує аргумент
function addItemBad(list: Item[], item: Item) { list.push(item); return list; }
// ✅ повертає новий масив
function addItem(list: Item[], item: Item): Item[] { return [...list, item]; }
```

**➕ Пов'язане: що таке dependency injection для тестованості? 🔵**
DI — коли залежність (клієнт API, годинник, генератор id) передається ззовні, а не створюється всередині функції. Тоді в тесті легко підставити fake/mock замість справжньої. Замість `new Date()` усередині — приймай `now: Date` параметром; замість імпорту клієнта — передавай його аргументом.

**➕ Пов'язане: що таке side effect і чому він ускладнює тест? 🟢**
Побічний ефект — будь-яка взаємодія із зовнішнім світом (запис у БД, мережа, `console.log`, зміна глобальної змінної, DOM). Функції з ефектами непередбачувані й вимагають моків для тесту. Тому ефекти ізолюють на «краях» застосунку, а ядро логіки лишають чистим.

## Q17. Що таке e2e-тестування і чим відрізняється від unit? 🟢

**E2E (end-to-end)** перевіряє весь застосунок так, як його бачить користувач: відкрив сторінку → натиснув → отримав результат. На противагу unit, який перевіряє один шматок ізольовано.

| | Unit | E2E |
|---|---|---|
| Що перевіряє | Одну функцію/компонент | Повний сценарій у браузері |
| Швидкість | Дуже швидко | Повільно |
| Хто пише | Розробники | Розробники / QA-автоматизатори |
| Приклад | «функція рахує знижку» | «користувач логіниться й записується» |

E2e ловить проблеми **інтеграції** — коли кожен шматок працює окремо, а разом сценарій падає (напр. фронт шле не той формат, що чекає бекенд).

```ts
import { test, expect } from "@playwright/test";
test("користувач записується на курс", async ({ page }) => {
  await page.goto("/courses/1");
  await page.getByRole("button", { name: "Записатись" }).click();
  await expect(page.getByText("Ви записані")).toBeVisible();
});
```

**➕ Пов'язане: чим integration-тест відрізняється від e2e? 🟢**
Integration перевіряє **кілька модулів разом** (компонент + хук + мок API), зазвичай без реального браузера й бекенду — швидше за e2e. E2e ганяє **весь застосунок цілком** у справжньому браузері з реальним бекендом. Integration — «чи дружать сусідні шматки», e2e — «чи працює сценарій наскрізь».

**➕ Пов'язане: що таке flaky-тест і чому e2e часто flaky? 🟢**
Flaky — тест, що то проходить, то падає без змін коду. E2e схильні до цього через таймінги (елемент ще не з'явився), мережеві затримки, анімації. Лікують очікуваннями стану (`await expect(...).toBeVisible()`) замість фіксованих `sleep`, і стабільними тест-даними.
